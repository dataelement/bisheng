## 2. C1 · WebSocket 事件协议（A → H）

**不变量**：前端消费的 `MessageEventType` 共 **10 类**（`state_message_manager.py`），**不新增类型**。新内核所有信号映射进这 10 类。WS 桥 `MessageStreamHandle`（pop Redis → 转发）零改。

**payload schema（`{event_type, data, timestamp}` 中的 `data`）**：

| MessageEventType | data 关键字段 | 生产时机 |
|---|---|---|
| `task_generate` | `{tasks: [ExecuteTask]}` | `write_todos` 产清单 |
| `task_start` | `ExecuteTask`(status=IN_PROGRESS) | task 节点进入 |
| `task_status_update` | `ExecuteTask` | 状态切换 |
| `task_execute_step` | **见下 ExecStep** | 工具/思考/子图/检索 |
| `task_end` | `ExecuteTask`(status, result.answer, task_data) | task 节点完成 |
| `user_input` | **见下 NeedUserInput** | `__interrupt__` |
| `user_input_completed` | `ExecuteTask`(含 history) | 回答提交后 |
| `final_result` | `SessionVersion`(output_result.{answer,final_files,all_from_session_files}) | 成功收尾 |
| `error_message` | `{error: str}` | 异常 |
| `task_terminated` | `{message, session_id, terminated_at}` | 终止收尾（**末帧**） |

**`ExecStep`（task_execute_step.data）字段冻结**：
```jsonc
{
  "call_id": "str",            // 幂等键: start/end 两帧按它合并
  "task_id": "str",            // 归属任务（多任务/子图定位）
  "name": "str",               // 工具名
  "step_type": "tool|thinking|knowledge|subagent|ui_card|call_user_input",
  "status": "start|end",       // 适配层推导调用中/已完成（ToolMessage 仅 success/error）
  "call_reason": "str|null",   // 步骤标题；模型未填 → 前端降级「工具展示名」
  "params": {},                // 输入摘要
  "output": "str|null",        // 输出摘要（超长截断 + extra_info.truncated）
  "namespace": "str|null",     // 子图前缀；非空=嵌套子步骤，前端按层级渲染
  "extra_info": {              // 可选
    "file_info": {"file_name":"", "file_md5":"", "file_url":""},  // 文件溯源
    "is_thinking": true,       // 思考流可折叠（仅 thinking 模型）
    "truncated": false
  }
}
```
**`NeedUserInput`（user_input.data）**：`{task_id, call_reason, params, step_type:"call_user_input"}`。⚠️ 追问卡渲染数据**从 `AIMessage.tool_calls.args` 取**（不依赖 `push_ui_message`，langgraph 1.2.1 halt 缺陷）。

**`task_id` 稳定性规则（A 实现、fixtures 体现、H 信赖）**：deepagents todos 无原生 id —— ① 首次 `write_todos` 按 `md5(svid+":"+content)[:8]` 生成；② 更新按「content 精确匹配 → 序号对齐 → 视为新增」三级 diff 对齐沿用；③ `ExecStep.task_id` = 当前 `in_progress` todo 的 id，无 in_progress 时 = `svid`（session 级伪任务，前端渲染到全局流）。详见 design.md §3.3.1。

**交付物（A 在 Wave 0 末）**：`test/linsight/fixtures/ws_events/*.json` —— 覆盖上述 10 类的真实样例。**A 与 H 都对这一份 fixtures 编程**（A 写映射断言、H 写渲染），是「同一份真相」。

## 3. C2 · WorkspaceBackend 接口 + 布局（C → A/B/H）

**接口（注入 `create_deep_agent(backend=...)`，C 提供 `FakeWorkspaceBackend` 内存实现）**：
```python
class WorkspaceBackend(FilesystemBackend):
    def __init__(self, svid: str, minio, file_dir: str): ...
    def write(self, path: str, content: bytes|str) -> None      # cache 写 + MinIO 写穿（同步）
    def read(self, path: str, offset: int=0, limit: int|None=None) -> bytes|str  # cache 命中/懒加载，分页
    def ls(self, prefix: str="") -> list[FileEntry]             # 以 MinIO 为准
    def edit(self, path: str, ...) -> None                      # 部分编辑（cache 上做，写穿）
```
> ⚠️ **POC P1 修正（以 deepagents 实际协议为准）**：真实 `WorkspaceBackend` 须继承
> `deepagents.backends.filesystem.FilesystemBackend` 并实现其 `BackendProtocol`——方法签名/返回类型为：
> `read(file_path, offset=0, limit=2000)->ReadResult`（按行）、`write(file_path, content)->WriteResult`、
> `ls(path)->LsResult`、`edit(file_path, old_string, new_string, replace_all=False)->EditResult`，另含
> `glob/grep/upload_files/download_files`（及 `a*` 异步版）。上面的 `bytes|str`/`prefix` 是**概念签名**；
> `test/linsight/fixtures/fake_workspace_backend.py` 为 H/B 用的概念级内存 stub，Track C 实现按协议对齐。详见 `scripts/035-linsight-deepagents/RESULTS.md` P1。
**MinIO 布局（真相）**：`workspace/{svid}/`
- `uploads/<name>/index.md`（+ `images/`）— 解析后附件
- `output/` — 交付物（产物区）
- `scratch/` — 中间态（持久，不进产物区）
- `manifest.json` — 大/二进制文件指针清单 `[{path,size,md5,url}]`

**指针块格式（注入首条消息，零正文）**：
```
<uploaded_files>
- path: /uploads/{name}/index.md
  name: {original_filename}
  lines: {line_count}
  images: {image_count}
</uploaded_files>
```
**代码产出枚举契约（E2B `run` 结果回传，H 用于产物溯源/「任务下所有文件」）**：`new_files: [{path, size, is_product: bool}]`（`output/` 前缀 → `is_product=true`）。

**代码工具入参契约（C 产出 → A 装配/H 步骤卡消费）**：`bisheng_code_interpreter` 的 `args_schema`（BiSheng 自有 `CodeInterpreterToolArguments`，非 deepagents 内置）新增：
```python
required_files: list[str] = []   # 脚本需读取的工作区路径; >SIZE_AUTOPUSH(5MB) 的文件必须声明
```
语义：**沙箱不可达 MinIO**，一切文件进出沙箱由 worker 中转（E2B API）。≤`SIZE_AUTOPUSH` 自动 delta push（模型零负担）；超阈值文件须在 `required_files` 声明，工具实现流式写入沙箱（hash 命中 no-op）。脚本因未声明报 FileNotFoundError → `run` 结果附「需在 required_files 中声明」提示，模型补声明重试。本地 executor 不受影响（cwd=file_dir）。H 在工具步骤卡 `params` 中展示 `required_files`。

**`<name>` 规整规则**：小写化、非法字符归一、防穿越、同会话重名加后缀（与 Skill 命名同规）。

**FileEntry**：`{path, size, md5, is_dir, mtime}`。

## 4. C3 · Skill API + 激活 + frontmatter（D → A/I）

> 🔄 **契约变更（2026-06-12，发起方 D，消费方 A/I 须同步）**：经用户确认（PRD §4.5/§4.8 同步修订）——① 新增 **`display_name`（展示名称）**：前端列表/选择器/chip 一律显示它，slug `name`（UI 称「技能 ID」）仍是唯一键/激活键/目录名；② 上传支持 **多形态**（单 `.md` / `.zip` / `.skill` / 文件夹），技能为 **bundle 目录形态**（SKILL.md + 附属文件），新增 `GET /skill/{name}/file` 只读端点与 `SkillDetail.files[]`；③ 管理列表不展示 `name` 与 `source`（仅内部溯源）。mock 已同步。

**REST（`/api/v1/linsight/skill`，`PageData[T]={data,total}` 分页）** —— 端点对齐 design §7.5（按 `name` 寻址；冻结时已将原 §4 草案的 `{id}` / `PATCH .../enabled` 收敛到此版本，mock 见 `test/linsight/fixtures/skill_api_mock.json`）：
| 方法 | 路径 | 鉴权 | 入参 | 出参 |
|---|---|---|---|---|
| GET | `/skill` | `get_tenant_admin_user` | `?keyword&enabled&page&page_size`（keyword 匹配 display_name/description） | `PageData[SkillBrief]`（仅本租户自定义技能） |
| GET | `/skill/selectable` | `get_login_user` | — | `[{name,display_name,description}]`（本租户已启用，终端选择器） |

| GET | `/skill/{name}/file` | `get_tenant_admin_user` | `?path=`（bundle 内相对路径） | `{path, content}`（附属文件只读；防穿越；不存在 11053） |
| POST | `/skill` | `get_tenant_admin_user` | multipart(`file`=单 .md 或 .zip/.skill 整包；文件夹由前端打包) 或 form(`display_name`/`name`/`description`/`content`) | `SkillDetail` |

**`SkillBrief`**：`{id, name, display_name, description, enabled, source:"manual|sop_migrated", create_time}`（**UI 仅展示 display_name/description/enabled**；`name` 详情可见、`source` 不展示仅溯源）。
**`SkillDetail`**：`SkillBrief + {preview, source_text, files:[{path,size}]}`（`files` 含 `SKILL.md` 自身；附属文件内容经 `GET /skill/{name}/file?path=` 按需取）。
**`linsight_skill` 表字段（C7）**：`id, tenant_id, name, display_name(新增, ≤255), description, enabled, source, object_path(SKILLS_ROOT 下磁盘相对路径), size(bundle 总字节), created_by, create_time, update_time`；唯一约束 `(tenant_id, name)`；`(tenant_id, display_name)` 应用层唯一校验。

**激活契约（A 消费，不变）**：per-run 白名单经 `config.configurable.active_skills: list[str]`（**skill `name` 列表**，display_name 仅 UI 层）；`[]`=本轮全禁；前端**不下发 `None`**。

**frontmatter spec（校验）**：`name`(必填, ≤64, `^[a-z0-9]+(-[a-z0-9]+)*$`——小写/数字/单连字符、不得首尾或连续连字符，**且须等于所在目录名**，对齐 deepagents `_validate_skill_name`) / `description`(必填, ≤1024) / `allowed-tools`(可选) / `license`(可选) / `metadata`(可选, dict[str,str])：
- `metadata.display-name`：展示名称（中文），缺省回退 `name`；
- `metadata.sop-id`：迁移溯源（仅 Track G 迁移产物写入）。

**bundle 规则**：`.zip`/`.skill`/文件夹根目录须含 `SKILL.md`，目录结构原样落盘 `data/skills/{tenant_id}/<name>/`；**整包 ≤10MB**；form 新建仅产生单 `SKILL.md`。**built-in skill 不入表、不出 API、不暴露前端。**

**错误码（C7，110 段）**：11051 校验失败（frontmatter/技能 ID 非法、压缩包缺 SKILL.md、name≠目录名） / 11052 整包超 10MB / 11053 不存在（含 built-in 名与附属文件 path 不存在） / 11054 无权限 / **11055 重名（`name` 或 `display_name` 任一同租户重复）**。
> ⚠️ 重名码由原定 11050 顺延到 **11055**：11050 已被存量 `LinsightVectorModelError`（SOP 检索链路，design §8.6 计划下线）占用，Track 0 不得删在用码，故新码落该段未占用槽位。见 tasks.md §8 实际偏差记录。落码文件 `common/errcode/linsight.py`：`SkillValidationError`/`SkillFileTooLargeError`/`SkillNotFoundError`/`SkillPermissionError`/`SkillNameDuplicateError`。

### C5 checkpointer / 续跑（B → A/端点）

- `thread_id = session_version_id`；**`PlainRedisCheckpointer`**（自研，Wave 0 已交付），连接复用 `RedisManager`。
- **决策（2026-06-11，D4 已关闭）**：弃用 `langgraph-checkpoint-redis`（经 redisvl 依赖 RediSearch/Redis Stack，现网不支持）；改为自研 `PlainRedisCheckpointer`，仅依赖 plain Redis 标准命令。interrupt/resume 机制 POC P3 已验证 GREEN。
- **实现**：`bisheng/linsight/domain/services/checkpointer.py`，公开 `make_checkpointer()` 工厂函数。
- **Key schema**（plain Redis，无 RediSearch 依赖）：

  | 用途 | Key 格式 | Redis 数据结构 |
  | ---- | -------- | -------------- |
  | 检查点数据 | `linsight:ckpt:data:{thread_id}:{checkpoint_ns}:{checkpoint_id}` | HASH |
  | 时序索引（按写入时间排序） | `linsight:ckpt:idx:{thread_id}:{checkpoint_ns}` | ZSET（score=Unix timestamp） |
  | 待写入条目 | `linsight:ckpt:write:{thread_id}:{checkpoint_ns}:{checkpoint_id}:{task_id_b64}:{idx}` | HASH |

  所有 key TTL = 7 天（`_DEFAULT_TTL`，可在 `make_checkpointer(ttl_seconds=...)` 覆盖）。序列化使用 `JsonPlusSerializer`（langgraph 内置）。

- **`adelete_thread(thread_id)`**：park 期终止时由终止端点调用，SCAN 删除该 thread 所有 `linsight:ckpt:*:{thread_id}:*` key，确保 resume payload 无效化。
- **续跑入队 payload**（`/workbench/user-input` 成功后 **`lpush` 队头**——resume 优先于排队新任务，PRD §4.4.4）：约定与普通提交同结构，附 `resume=true` 标记；worker 拾取时按标记走「同 thread_id 复活 + `Command(resume=<最近 user_input 事件>)`」，否则走新任务；**拾取前校验任务非终态**，终态项丢弃（park 期被终止的任务不复活）。
- **park 期终止**：驻留期无进程在跑——终止端点直接置终态 + push `task_terminated` + 调用 `adelete_thread()` 删 checkpoint，不经 worker。
- A 集成前用 `InMemorySaver` 顶替。

### C8 统一会话模型 / 任务模式收敛（J → A/H · 设计见 [跨模式会话上下文共享设计](./跨模式会话上下文共享设计.md)）

> 任务模式不再是独立会话类型，降级为**日常工作台会话（`flow_type=WORKSTATION=15`，一个 `chat_id`）内的逐轮处理标记**。普通轮与任务轮同流落 `ChatMessage`，问答上下文双向共享、均喂模型（D2，与 §9.3.8 选择态记忆正交）。存量 `flow_type=20` 会话**不迁移**（D3），按旧只读形态保留。

**会话约定**：
- 一条会话 = 一行 `MessageSession`，`flow_type=15`（[`FlowType.WORKSTATION`](src/backend/bisheng/database/models/flow.py#L32)）；开/关任务模式**不改变 flow_type、不新建会话**。
- linsight `session_id` 仍 == 该会话 `chat_id`（既有事实，[`workbench_impl.py:161/193/203`](src/backend/bisheng/linsight/domain/services/workbench_impl.py#L161)）；`linsight_session_version`/`LinsightExecuteTask` 不动键、退位为「某条 bot 消息的执行详情附属」。

**① 统一提交入口请求体（前端 → 后端，J 冻结）**：
```jsonc
{
  "chat_id": "已存在的 15 会话 chat_id",   // 必填；不再由 linsight 侧新建
  "question": "用户输入",
  "task_mode": true,                       // 逐轮标记：true→linsight 内核, false→工作台日常链
  "tools":  [], "model": null,             // 任务模式相关，task_mode=false 时可忽略
  "files":  [], "skills": []
}
```
- 后端按 `task_mode` 分流；任务链复用既有 deepagents 内核（Track A），仅入口装配改为「接收外部 `chat_id`、不自建 `flow_type=20` 会话」。

**② 任务轮 `ChatMessage` 双写字段约定（后端写 → 前端读，双方对此编程）**：

| 字段 | 用户轮 | 任务 bot 轮 | 说明 |
|---|---|---|---|
| `is_bot` | 0 | 1 | |
| `message` | 用户问题 | `output_result.answer`（最终答案文本） | 失败/进行中为占位文本 |
| `category` | `question` | **`task`** | 前端据此切富面板渲染分支 |
| `extra` | — | **`{"linsight_session_version_id": "SV-…"}`** | 执行详情指针，前端懒加载用 |
| `intermediate_steps` | — | 可选：步骤摘要 | 非渲染必需 |

- HITL 等待 / 进行中：bot 行先落占位态，恢复后**回填同一行**（沿用 design §4 park-and-release）；实时事件仍走 C1 WS 流。

**③ 会话历史查询（后端 → 前端，两段式）**：
- **列表（轻量）**：按 `chat_id` 取 `ChatMessage` 线性流（`ChatMessageDao.get_messages_by_chat_id`），时间序、分页、多租户隔离；任务轮**只带 `extra.SV` 指针**，不含 tasks/sop/产物。
- **详情（懒加载）**：按 `linsight_session_version_id` 取 tasks/sop/steps/files（复用既有 linsight 详情读取），仅前端展开任务轮时调用。

**④ 历史注入（喂模型，chat_id 维度）**：
- `_get_history_summary` 由 `session_version_id` 维度改为按 `chat_id` 读 `ChatMessage`（普通轮 + 任务轮答案）→ 注入既有 `history_summary` 通道；日常链注入同理读 `ChatMessage`（含任务轮答案）。token 裁剪复用 design §3.8。

**A 消费要点**：内核 astream/事件流/工具/HITL 全不变；A 只需保证执行结果可被 J 在完成回调里取到 `output_result.answer` 与 `session_version_id` 回写 `ChatMessage`。
**H 消费要点**：统一消息列表按 `is_bot`+`category` 分发；任务轮答案气泡 + 折叠面板（展开按 `extra.SV` 拉详情，复用 `Linsight/Execution/*`）；`linsightMapState` 降级为执行态缓存。
**release-contract**：统一提交入口端点 + `ChatMessage` 的 `category='task'`/`extra` 字段语义须在 release-contract 备案（无新表、无新错误码段；若入口为新端点则登记路由）。
