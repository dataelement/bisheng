# BiSheng 平台 ReBAC 权限体系改造技术方案

## Context

BiSheng 平台当前采用传统 RBAC 权限模型（Role → RoleAccess → Resource），存在以下核心问题：

1. **无部门概念**：`group` 表是扁平结构，既承载组织架构又承载项目组，概念混淆
2. **权限分散**：资源权限散落在 `role_access` + `group_resource` + `space_channel_member` 三套机制中
3. **无文件夹级权限**：知识空间内文件夹无法独立设置权限，粒度只到空间级
4. **无法从资源视角授权**：不支持直接将资源授权给部门或个人
5. **无操作级细粒度**：仅有读/写二元权限，缺少发布、分享、管理等操作级区分

### 技术选型：OpenFGA

本方案采用 **OpenFGA**（Apache 2.0 协议）作为 ReBAC 权限引擎：

| 决策项 | 结论 |
|--------|------|
| 引擎 | OpenFGA Server（Docker 部署）+ openfga-sdk（Python SDK） |
| 协议兼容性 | OpenFGA 及 SDK 均为 Apache 2.0，与 BiSheng 完全兼容 |
| 跨项目能力 | 未来可与 Open Ontology 等项目共享同一 OpenFGA 实例，统一权限体系 |
| 部署拓扑 | OpenFGA 作为基础设施层服务，与 MySQL/Redis/Milvus 同级 |

**为什么选 OpenFGA 而非自研**：
- 经过 Google Zanzibar 论文验证的成熟模型，Check/ListObjects 性能有保障
- 内置关系图遍历、环检测、继承解析，无需自行实现 BFS/DFS
- 支持 authorization model 版本管理，动态更新不停服
- 社区活跃（CNCF 项目），长期维护有保障

### 现有权限体系概览

| 表 | 用途 | 核心字段 |
|---|---|---|
| `user` | 用户账号 | user_id, user_name |
| `role` | 角色定义（按 group 划分） | id, role_name, group_id |
| `user_role` | 用户-角色关联 | user_id, role_id |
| `group` | 用户组（扁平，无父子） | id, group_name |
| `user_group` | 用户-组关联 | user_id, group_id, is_group_admin |
| `role_access` | 角色-资源权限 | role_id, third_id, type(AccessType) |
| `group_resource` | 组-资源关联 | group_id, third_id, type(ResourceTypeEnum) |
| `space_channel_member` | 知识空间/频道成员 | business_id, user_id, user_role(CREATOR/ADMIN/MEMBER) |

**权限检查流程**（`LoginUser.access_check`）：owner → admin bypass → RoleAccessDao.judge_role_access

### 新旧体系职责分工

```
新权限体系分工：

OpenFGA (ReBAC)                     role 表 (策略角色 RBAC)
├── 谁能访问什么资源                  ├── 能看到哪些菜单 (WEB_MENU)
├── 谁能编辑什么资源                  ├── 各资源创建上限 (Quota)
├── 谁能管理什么资源                  └── 角色作用域（部门级）
├── 部门/用户组/文件夹继承
└── super_admin 短路判定
```

## 一、授权模型设计（OpenFGA DSL）

```dsl
model
  schema 1.1

# ============================================
# 基础类型
# ============================================
type user

# ============================================
# 系统 — 超级管理员短路逻辑
# 权限引擎第一步校验，直接放行
# ============================================
type system
  relations
    define super_admin: [user]

# ============================================
# 部门 — 严格树形组织架构
# ============================================
type department
  relations
    define parent: [department]
    define admin: [user] or admin from parent        # 部门管理员权限向下传递
    define member: [user]                             # 成员仅限直接隶属，不自动继承
    # ⚠️ 重要：department:X#member 只展开 X 自身的 member 集合，不会递归遍历子部门。
    # 当授权给"部门(含子部门)"时，由业务层（PermissionService.authorize）查出子部门树，
    # 为每个子部门分别写入元组，详见 Section 三.7 DepartmentChangeHandler。

# ============================================
# 用户组 — 跨部门临时项目组
# ============================================
type user_group
  relations
    define admin: [user]
    define member: [user]

# ============================================
# 知识空间（含文档知识库/QA知识库/知识空间三种类型）
# ============================================
type knowledge_space
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer

# ============================================
# 文件夹 — 支持嵌套，权限从父级继承
# ============================================
type folder
  relations
    define parent: [knowledge_space, folder]          # 父级可以是空间或文件夹
    define owner: [user]
    define manager: [user, department#member, user_group#member] or can_manage from parent
    define editor: [user, department#member, user_group#member] or manager or can_edit from parent
    define viewer: [user, department#member, user_group#member] or editor or can_read from parent
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer

# ============================================
# 知识文件 — 权限继承自父文件夹/空间
# ============================================
type knowledge_file
  relations
    define parent: [folder, knowledge_space]
    define owner: [user]
    define can_read: [user, department#member, user_group#member] or can_read from parent
    define can_edit: [user, department#member, user_group#member] or can_edit from parent
    define can_delete: owner or can_manage from parent

# ============================================
# 频道 — 信息聚合与订阅，与知识空间同级
# ============================================
type channel
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer

# ============================================
# 工作流
# ============================================
type workflow
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 助手
# ============================================
type assistant
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 工具
# ============================================
type tool
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 看板
# ============================================
type dashboard
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner
```

### 核心设计要点

| 设计决策 | 说明 | 原因 |
|----------|------|------|
| **知识库统一为 knowledge_space** | 管理后台文档/QA知识库(type=0,1) 和工作台知识空间(type=3) 在 ReBAC 层共用同一个 object_type | DB 本身共用 `knowledge` 表，权限模型统一更简洁；前端入口和展示形态差异由业务层处理 |
| 部门 admin 向下传递 | `admin from parent` | 父部门管理员自动管理子部门，符合组织架构直觉 |
| 部门 member 不继承 | member 仅限直接隶属 | 避免"加入总公司=加入所有部门"。**注意**：`department:X#member` 只展开 X 自身的 member，不会递归子部门 |
| **部门授权业务层展开** | 授权给"部门(含子部门)"时，业务层查出子部门树，为每个子部门分别写入 `department:Y#member` 元组 | `department#member` userset 在 OpenFGA 中不递归遍历 parent/child 关系图。部门新增/移动/删除时需级联维护元组，详见 Section 三.7 |
| 附属部门同等 member | 主属和附属都写入 `department#member` | 在资源授权层面不区分主属/附属，简化模型 |
| 三种授权主体 | `[user, department#member, user_group#member]` | 支持授权给个人、部门全员、项目组全员 |
| 文件夹权限继承 | `can_read from parent` 链式传递 | 空间 → 文件夹 → 子文件夹，逐级继承 |
| 权限金字塔 | owner > manager > editor > viewer | 高角色自动包含低角色权限，减少元组数量 |
| 命名对齐 | manager↔can_manage, editor↔can_edit, viewer↔can_read | 角色与计算权限一一对应，语义清晰 |
| 关系取并集 | 多路径授权取最高权限 | OpenFGA 天然支持：用户被部门授予 viewer 又被单独授予 editor，最终为 editor |

## 一·附：权限级别 × 资源操作 映射表

权限金字塔：**owner ⊃ can_manage ⊃ can_edit ⊃ can_read**，高级别自动包含所有低级别操作。

> **设计原则**：
> - **can_read** = 查看 + 使用（执行/对话/下载），只消费不修改
> - **can_edit** = 修改资源内容，但不控制资源生命周期和权限分配
> - **can_manage** = 资源生命周期管理（发布/下线/删除）+ 权限分配 + 成员管理
> - **owner** = can_manage + 不可被他人收回的最终控制权（删除资源、转让所有权）

## 二、数据库模型设计

### 2.1 OpenFGA 管理关系存储

关系元组（tuples）不再存储在 BiSheng 的 MySQL 中，而是由 OpenFGA Server 内部管理。BiSheng 通过 openfga-sdk 的 `Write` / `Read` / `Check` / `ListObjects` API 与 OpenFGA 交互。

**OpenFGA 存储后端**：推荐使用 PostgreSQL 或 MySQL，由 OpenFGA Server 自行管理 schema。

**关系元组示例**（通过 SDK 写入 OpenFGA）：

| object | relation | user |
|--------|----------|------|
| `system:global` | `super_admin` | `user:1` |
| `department:5` | `member` | `user:12` |
| `department:5` | `parent` | `department:1` |
| `knowledge_space:42` | `owner` | `user:7` |
| `knowledge_space:42` | `viewer` | `department:5#member` |
| `folder:101` | `parent` | `knowledge_space:42` |
| `folder:101` | `editor` | `user_group:3#member` |
| `workflow:abc-123` | `viewer` | `department:5#member` |
| `workflow:abc-123` | `editor` | `user:50` |

### 2.2 部门表 `department`

```sql
CREATE TABLE department (
    id               INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    dept_id          VARCHAR(64)  NOT NULL COMMENT '部门ID（含前缀，如 BS@89757）',
    name             VARCHAR(128) NOT NULL COMMENT '部门名称',
    parent_id        INT UNSIGNED DEFAULT NULL COMMENT 'NULL 表示根部门',
    path             VARCHAR(512) NOT NULL DEFAULT '' COMMENT '物化路径: /1/3/5/ 用于快速子树查询',
    sort_order       INT          NOT NULL DEFAULT 0 COMMENT '同级排序',
    source           VARCHAR(32)  NOT NULL DEFAULT 'local' COMMENT '来源: local / feishu / wecom / dingtalk',
    external_id      VARCHAR(128) DEFAULT NULL COMMENT '第三方部门ID（同步用）',
    status           VARCHAR(16)  NOT NULL DEFAULT 'active' COMMENT '状态: active / archived',
    default_role_ids JSON         DEFAULT NULL COMMENT '部门默认角色ID列表',
    create_user      INT UNSIGNED DEFAULT NULL,
    create_time      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    UNIQUE KEY uk_dept_id (dept_id),
    UNIQUE KEY uk_external_id (source, external_id),
    INDEX idx_parent (parent_id),
    INDEX idx_path (path),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='部门表（树形组织架构）';
```

**物化路径说明**：
- 根部门(id=1): `path = '/1/'`
- 工程部(id=2, parent=1): `path = '/1/2/'`
- API 小组(id=3, parent=2): `path = '/1/2/3/'`
- 查询工程部及其所有子部门: `WHERE path LIKE '/1/2/%'`

### 2.3 用户-部门关联表 `user_department`

```sql
CREATE TABLE user_department (
    id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id       INT UNSIGNED NOT NULL,
    department_id INT UNSIGNED NOT NULL,
    is_primary    TINYINT(1)   NOT NULL DEFAULT 1 COMMENT '是否主部门',
    source        VARCHAR(32)  NOT NULL DEFAULT 'local' COMMENT '来源: local / feishu / wecom / dingtalk',
    create_time   DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_user_dept (user_id, department_id),
    INDEX idx_dept (department_id),
    INDEX idx_user_primary (user_id, is_primary)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户-部门关联表';
```

> `user_department` 与 OpenFGA 中的 `(department, X, member, user, Y)` 元组保持**双写同步**。
> `user_department` 用于 SQL 快速查询（如"列出某部门所有成员"），OpenFGA 元组用于权限判定。

### 2.4 关系定义表 `relation_definition`（支持自定义关系层级）

超管可以新增 owner/manager/editor/viewer 之外的自定义关系层级。关系定义存储在 MySQL，用于动态生成 OpenFGA authorization model。

```sql
CREATE TABLE relation_definition (
    id              INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name            VARCHAR(32)  NOT NULL COMMENT '关系名称（英文标识，如 reviewer）',
    display_name    VARCHAR(64)  NOT NULL COMMENT '显示名称（如 审阅者）',
    priority        INT          NOT NULL COMMENT '优先级，数字越大权限越高，用于构建继承链',
    resource_types  JSON         NOT NULL COMMENT '适用的资源类型列表，如 ["workflow","assistant"]',
    operations      JSON         NOT NULL COMMENT '该关系可执行的操作列表，如 ["read","use","edit"]',
    is_system       TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '是否系统预设（预设不可删除）',
    create_time     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    UNIQUE KEY uk_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='ReBAC 关系定义表（支持自定义关系层级）';
```

**系统预设数据**：

| name | display_name | priority | resource_types | operations | is_system |
|------|-------------|----------|---------------|------------|-----------|
| viewer | 查看者 | 10 | ["*"] | ["read","use"] | 1 |
| editor | 编辑者 | 20 | ["*"] | ["read","use","edit"] | 1 |
| manager | 管理者 | 30 | ["*"] | ["read","use","edit","manage"] | 1 |
| owner | 所有者 | 40 | ["*"] | ["read","use","edit","manage","delete"] | 1 |

**⚠️ P0 阶段使用静态固定的 authorization model**，不实现动态生成。自定义关系层级为 P2 优先级功能。

**动态 Model 生成流程（P2 阶段实现）**：

1. 从 `relation_definition` 表读取所有关系定义
2. 按 `priority` 升序排列，自动构建继承链（高优先级 `or` 低优先级）
3. 对每种资源类型，根据 `resource_types` 筛选适用的关系
4. 生成 OpenFGA DSL 字符串
5. 调用 `openfga_client.write_authorization_model()` 写入新版本
6. 新的 `Check` / `ListObjects` 调用自动使用最新 model 版本

**P2 阶段安全措施**：
- 删除关系定义前，**必须先清理**所有引用该关系的元组，否则 Check 会报错
- 加**分布式锁**防止并发生成 model 产生竞态条件
- 生成后调用 OpenFGA 的 **model validation API** 校验 DSL 正确性
- 保留最近 N 个 model 版本，支持回滚

### 2.5 角色表改造：从「权限角色」到「策略角色」

新 ReBAC 体系接管了资源访问控制，`role` 表职责收窄为：
- **菜单可见性**：控制用户能看到哪些前端页面/功能入口（WEB_MENU）
- **资源配额**：控制各类资源的创建上限
- **角色作用域**：限定角色的可见及可授权边界（部门级）

#### 角色表扩展

```sql
ALTER TABLE role
    ADD COLUMN department_id INT UNSIGNED DEFAULT NULL COMMENT '角色作用域：所属部门ID，NULL 表示全局角色',
    ADD COLUMN quota_config JSON DEFAULT NULL COMMENT '资源配额配置（覆盖默认值）';
```

**部门上下文中可分配角色（与 `department_id`）**：在部门 **T** 的成员创建/编辑、默认角色等 API 中，可选项为 `department_id IS NULL` 的全局类角色，以及 `department_id` 指向 **T 或其任意上级部门** 的角色（即作用域部门 S 满足：T 属于 S 的子树）。不返回作用域仅落在兄弟子树的角色。

`quota_config` JSON 结构：

```json

    "knowledge_space_file": 500,
    "channel": 10,
    "channel_subscribe": 20,
    "workflow": -1,
    "assistant": -1,
    "tool": -1,
    "dashboard": -1
```

> `-1` 表示不限制，`0` 表示禁止创建，正整数表示上限。
> 未在 JSON 中出现的 key 使用系统默认值。

#### 配额解析逻辑

用户可能绑定多个角色，取**各角色中该资源配额的最大值**：

```python
SYSTEM_DEFAULT_QUOTAS = {
    'knowledge_space': 30,
    'knowledge_space_file': 500,
    'channel': 10,
    'channel_subscribe': 20,
    'workflow': -1,
    'assistant': -1,
    'tool': -1,
    'dashboard': -1,
}

async def get_effective_quota(user_id: int, resource_type: str) -> int:
    """获取用户对某类资源的有效配额。"""
    roles = await UserRoleDao.get_user_roles(user_id)
    if any(r.role_id == AdminRole for r in roles):
        return -1
    role_ids = [r.role_id for r in roles]
    role_list = await RoleDao.get_roles_by_ids(role_ids)
    max_quota = None
    for role in role_list:
        quota = (role.quota_config or {}).get(resource_type)
        if quota is None:
            continue
        if quota == -1:
            return -1
        if max_quota is None or quota > max_quota:
            max_quota = quota
    return max_quota if max_quota is not None else SYSTEM_DEFAULT_QUOTAS.get(resource_type, -1)
```

#### OpenFGA Client 封装

```python
from openfga_sdk import OpenFgaClient, ClientConfiguration
from openfga_sdk.client.models import ClientCheckRequest, ClientWriteRequest, ClientListObjectsRequest

class FGAClient:
    """OpenFGA Client 单例封装。"""

    _instance = None

    def __init__(self, settings):
        configuration = ClientConfiguration(
            api_url=settings.openfga_api_url,    # e.g. "http://localhost:8080"
            store_id=settings.openfga_store_id,
        )
        self._client = OpenFgaClient(configuration)

    @classmethod
    def get_instance(cls) -> 'FGAClient':
        if cls._instance is None:
            from bisheng.core.config import settings
            cls._instance = cls(settings)
        return cls._instance

    async def check(self, user: str, relation: str, object: str) -> bool:
        """
        检查权限。
        user: "user:7"
        relation: "can_edit"
        object: "knowledge_space:42"
        """
        body = ClientCheckRequest(user=user, relation=relation, object=object)
        response = await self._client.check(body)
        return response.allowed

    async def list_objects(self, user: str, relation: str, type: str) -> list[str]:
        """
        列出用户可访问的所有资源 ID。
        返回: ["knowledge_space:42", "knowledge_space:55", ...]
        """
        body = ClientListObjectsRequest(user=user, relation=relation, type=type)
        response = await self._client.list_objects(body)
        return response.objects

    async def write_tuples(self, writes: list[dict], deletes: list[dict] = None):
        """
        批量写入/删除关系元组。
        writes: [{"user": "user:7", "relation": "editor", "object": "workflow:abc"}]
        deletes: [{"user": "user:8", "relation": "viewer", "object": "workflow:abc"}]
        """
        body = ClientWriteRequest(writes=writes, deletes=deletes or [])
        await self._client.write(body)

    async def read_tuples(self, object: str = None, relation: str = None, user: str = None) -> list:
        """读取匹配条件的关系元组。"""
        body = {}
        if object:
            body['object'] = object
        if relation:
            body['relation'] = relation
        if user:
            body['user'] = user
        response = await self._client.read(body)
        return response.tuples

    async def write_authorization_model(self, model_json: dict) -> str:
        """写入新的授权模型版本，返回 model_id。"""
        response = await self._client.write_authorization_model(model_json)
        return response.authorization_model_id
```

#### PermissionService — 业务层统一入口

```python
class PermissionService:
    """统一权限检查入口（单例）。不包含 super_admin 判定——管理员短路由 LoginUser 层负责。"""

    _instance = None

    def __init__(self):
        self.fga = FGAClient.get_instance()

    @classmethod
    def get_instance(cls) -> 'PermissionService':
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance

    async def check(self, user_id: int, relation: str, object_type: str, object_id: str) -> bool:
        """
        核心权限检查。
        注意：super_admin 判定不在此处进行，由 LoginUser.is_admin() 在调用前短路。
        """
        return await self.fga.check(
            user=f"user:{user_id}",
            relation=relation,
            object=f"{object_type}:{object_id}"
        )

    async def list_accessible_ids(self, user_id: int, relation: str, object_type: str) -> list[str]:
        """
        列出用户可访问的资源 ID 列表。
        始终返回 list[str]，不再对管理员返回 None。
        管理员走无过滤查询分支的逻辑由调用方（LoginUser 层）负责。
        """
        objects = await self.fga.list_objects(
            user=f"user:{user_id}",
            relation=relation,
            type=object_type
        )
        # "knowledge_space:42" → "42"
        return [obj.split(":")[-1] for obj in objects]

    async def authorize(self, object_type: str, object_id: str,
                        grants: list[dict], revokes: list[dict] = None):
        """
        资源授权：批量授予/撤销关系。
        grants: [{"subject_type": "user"|"department"|"user_group", "subject_id": 7, "relation": "editor",
                   "include_children": True}]
        当 subject_type == 'department' 且 include_children == True 时，
        自动查出子部门树并为每个子部门分别写入元组。
        """
        writes = []
        for g in grants:
            user_strs = self._expand_subject(g['subject_type'], g['subject_id'],
                                              g.get('include_children', True))
            for user_str in user_strs:
                writes.append({
                    "user": user_str,
                    "relation": g['relation'],
                    "object": f"{object_type}:{object_id}"
                })
        deletes = []
        for r in (revokes or []):
            user_strs = self._expand_subject(r['subject_type'], r['subject_id'],
                                              r.get('include_children', True))
            for user_str in user_strs:
                deletes.append({
                    "user": user_str,
                    "relation": r['relation'],
                    "object": f"{object_type}:{object_id}"
                })
        # 写入 OpenFGA，失败时记入补偿表（见 Section 三.5）
        try:
            await self.fga.write_tuples(writes, deletes)
        except Exception as e:
            await self._save_failed_tuples(writes, deletes, str(e))
            raise
        # 清除相关缓存
        await self._invalidate_cache(object_type, object_id)

    @staticmethod
    def _expand_subject(subject_type: str, subject_id: int, include_children: bool = True) -> list[str]:
        """
        展开授权主体。
        部门授权时默认展开子部门树（include_children=True），
        解决 department#member 不递归遍历子部门的问题。
        """
        if subject_type == 'user':
            return [f"user:{subject_id}"]
        elif subject_type == 'department':
            if include_children:
                # 查出该部门及其所有子部门 ID
                sub_dept_ids = DepartmentDao.get_subtree_ids(subject_id)  # [1, 2, 3]
                return [f"department:{dept_id}#member" for dept_id in sub_dept_ids]
            else:
                return [f"department:{subject_id}#member"]
        elif subject_type == 'user_group':
            return [f"user_group:{subject_id}#member"]
        raise ValueError(f"Unknown subject_type: {subject_type}")
```

### 3.3 与现有 LoginUser 的集成

保持 `LoginUser.access_check` 方法签名不变，内部委托给 PermissionService：

```python
# user/domain/services/auth.py

ACCESS_TYPE_MAPPING = {
    AccessType.KNOWLEDGE:       ('knowledge_space', 'can_read'),
    AccessType.KNOWLEDGE_WRITE: ('knowledge_space', 'can_edit'),
    AccessType.ASSISTANT_READ:  ('assistant', 'can_read'),
    AccessType.ASSISTANT_WRITE: ('assistant', 'can_edit'),
    AccessType.GPTS_TOOL_READ:  ('tool', 'can_read'),
    AccessType.GPTS_TOOL_WRITE: ('tool', 'can_edit'),
    AccessType.WORKFLOW:        ('workflow', 'can_read'),
    AccessType.WORKFLOW_WRITE:  ('workflow', 'can_edit'),
    AccessType.DASHBOARD:       ('dashboard', 'can_read'),
    AccessType.DASHBOARD_WRITE: ('dashboard', 'can_edit'),
}

class LoginUser(BaseModel):
    async def async_access_check(self, owner_user_id, target_id, access_type):
        # 保留旧签名兼容性，内部转发到 OpenFGA
        # ① 管理员在此短路，不走 OpenFGA（避免每次请求双倍 OpenFGA 调用）
        if self.is_admin():
            return True
        # ② 资源所有者检查
        if self.user_id == owner_user_id:
            return True
        # ③ OpenFGA 权限检查
        object_type, relation = ACCESS_TYPE_MAPPING[access_type]
        perm_service = PermissionService.get_instance()
        return await perm_service.check(self.user_id, relation, object_type, target_id)

    async def get_accessible_ids(self, object_type: str, relation: str) -> list[str] | None:
        """
        获取用户可访问的资源 ID 列表。
        管理员返回 None 表示无需过滤（调用方直接走无过滤 SQL 查询）。
        非管理员返回具体 ID 列表。
        """
        if self.is_admin():
            return None
        return await PermissionService.get_instance().list_accessible_ids(
            self.user_id, relation, object_type
        )
```

> **super_admin 判定优化**：管理员判定不走 OpenFGA，而是复用现有 `LoginUser.is_admin()` 方法（JWT 解析阶段从 `user_role` 表加载 `role_id == AdminRole`，结果缓存为 `cached_property`，零网络开销）。OpenFGA 中的 `system:global / super_admin` 元组仅作为数据迁移桥梁和审计用途，不参与热路径 Check。
>
> **list_accessible_ids 返回值设计**：管理员与非管理员的分流在 `LoginUser.get_accessible_ids()` 统一处理。`PermissionService.list_accessible_ids()` 本身始终返回 `list[str]`，语义清晰，调用方无需处理 None 分支。

### 3.4 缓存策略

| 层级 | 缓存内容 | 存储 | TTL | 失效时机 |
|------|----------|------|-----|----------|
| L1 | 管理员判定 | LoginUser 实例内存 | 请求生命周期 | 无需主动失效（每次请求重新加载） |
| L2 | list_objects 结果 | Redis | 10s | 以下任一操作触发清除 |

> OpenFGA Server 内部已有 check 结果缓存，BiSheng 侧仅对高频查询做轻量缓存。
>
> **L2 缓存失效触发点**（所有会改变权限的操作）：
> - `PermissionService.authorize()` 调用后 — 资源的直接授权/撤销
> - 用户加入/移出部门 — 通过部门管理 API 或第三方同步（org_sync）
> - 用户加入/移出用户组 — 通过用户组管理 API
> - 文件夹 parent 关系变更 — 文件夹移动操作
> - 部门树结构变更 — 部门新增/移动/删除/归档（联动 DepartmentChangeHandler）
>
> 统一通过方法调用后发送 Redis 缓存清除信号实现。TTL 设为 10s 作为兜底，接受短暂的最终一致性。
>
> **降级策略**：OpenFGA 不可用时，fail-closed（拒绝），避免越权。

### 3.5 事务一致性保障

MySQL 与 OpenFGA 是异构系统，无法参与同一分布式事务。采用**先写 MySQL，再写 OpenFGA + 补偿队列**策略：

```
资源创建流程：
  ① MySQL 写入资源记录（事务提交）
  ② 调用 OpenFGA 写入 owner 元组
     ├── 成功 → 正常返回
     └── 失败 → 记入 failed_tuples 补偿表 → 返回成功（用户无感知）
  ③ 定时任务每 30s 扫描 failed_tuples，重试写入，最多 3 次
  ④ 超过重试上限 → 状态标为 failed，触发告警
```

**Owner 回退逻辑**：在 OpenFGA 元组尚未写入成功的窗口期内，`PermissionService.check` 增加兜底判断：如果 OpenFGA check 返回 False，额外检查用户是否为资源 DB 记录的 `user_id`（创建者），如果是则返回 True。确保资源创建者在补偿完成前不会丢失权限。

### 3.6 资源生命周期联动

资源创建/删除时需要同步维护 OpenFGA 元组：

| 生命周期事件 | 元组操作 |
|-------------|---------|
| 资源创建 | 写入 `(resource:id, owner, user:creator_id)` |
| 资源删除 | 调用 `fga.read_tuples(object="resource:id")` 获取所有关联元组并批量删除 |
| 文件夹移动 | 删除旧 parent 元组 + 写入新 parent 元组（OpenFGA 自动重新计算继承链） |
| 知识空间删除 | 级联删除所有子文件夹和文件的 parent 元组 + 直接授权元组 |

**防御性措施**：增加定时任务（每日凌晨），扫描 OpenFGA 中存在但 MySQL 中已删除的资源（孤立元组），自动清理。

### 3.7 部门树变更处理（DepartmentChangeHandler）

由于 `department#member` 不递归遍历子部门（见 Section 一 DSL 注释），当部门树结构发生变更时，需要级联维护所有引用该部门的资源元组：

| 变更事件 | 处理逻辑 |
|---------|---------|
| **新增子部门** | 查出所有授权给父部门（含子部门）的资源，为新子部门补写对应的 `department:new_id#member` 元组 |
| **移动部门**（变更 parent） | ① 从旧父部门树中移除：删除旧树中因"含子部门"策略写入的该部门及其子部门的元组；② 加入新父部门树：查出新父部门树中"含子部门"的资源授权，为移动的部门及其子部门补写元组 |
| **删除/归档部门** | 删除该部门的所有 `department:id#member` 元组引用（不影响子部门，子部门归档时各自处理） |

**实现要点**：
- 通过 `department.path`（物化路径）高效查询子部门树：`WHERE path LIKE '/1/2/%'`
- 授权记录中需标记"是否含子部门"（`include_children` 字段），以便部门变更时定位需要级联维护的元组
- 部门变更操作提交后异步执行元组维护（Celery 任务），避免阻塞用户操作
- 授权查询 API 返回时，将多个子部门元组合并展示为一条"工程部(含子部门)"的聚合记录

### Phase 2: 数据迁移脚本

一次性迁移脚本，通过 OpenFGA SDK 写入元组，支持 `--dry-run` 模式：

```python
async def migrate_rbac_to_rebac():
    fga = FGAClient.get_instance()
    writes = []

    # 1. 超管迁移
    admin_users = await UserRoleDao.get_users_by_role(AdminRole)
    for u in admin_users:
        writes.append({"user": f"user:{u.user_id}", "relation": "super_admin", "object": "system:global"})

    # 2. group → user_group (OpenFGA type)
    #    group 表数据保留但语义改为"用户组"，Department 从零开始建
    for ug in all_user_groups:
        writes.append({
            "user": f"user:{ug.user_id}",
            "relation": "admin" if ug.is_group_admin else "member",
            "object": f"user_group:{ug.group_id}"
        })

    # 3. role_access → 展开为每个用户的直接关系元组
    for ra in all_role_access:
        if ra.type == AccessType.WEB_MENU.value:
            continue  # WEB_MENU 保留在 MySQL，不迁移
        object_type, relation = ACCESS_TYPE_MAPPING[ra.type]
        users_with_role = await UserRoleDao.get_users_by_role(ra.role_id)
        for ur in users_with_role:
            writes.append({
                "user": f"user:{ur.user_id}",
                "relation": relation_to_role_name(relation),  # can_read → viewer, can_edit → editor
                "object": f"{object_type}:{ra.third_id}"
            })

    # 4. group_resource — 不迁移到 OpenFGA
    #    group_resource 在旧系统中不参与访问控制（仅用于资源归组展示和管理页过滤），
    #    直接转为 viewer 元组会导致权限扩大（组内用户可能原本没有该资源的访问权限）。
    #    该表在新系统中直接废弃，资源可见性完全由 OpenFGA 元组管理。

    # 5. SpaceChannelMember → knowledge_space / channel 元组
    role_map = {'creator': 'owner', 'admin': 'manager', 'member': 'viewer'}
    for scm in all_active_space_members:
        object_type = 'knowledge_space' if scm.business_type == 'space' else 'channel'
        writes.append({
            "user": f"user:{scm.user_id}",
            "relation": role_map[scm.user_role],
            "object": f"{object_type}:{scm.business_id}"
        })

    # 6. 资源 ownership（从各资源表的 user_id 字段）
    for knowledge in all_knowledge:
        writes.append({"user": f"user:{knowledge.user_id}", "relation": "owner", "object": f"knowledge_space:{knowledge.id}"})
    for flow in all_flows:
        obj_type = 'assistant' if flow.flow_type == FlowType.ASSISTANT.value else 'workflow'
        writes.append({"user": f"user:{flow.user_id}", "relation": "owner", "object": f"{obj_type}:{flow.id}"})
    # ... channel, tool, dashboard 同理

    # 7. 知识空间文件夹层级关系
    for file in all_knowledge_files:
        if file.file_type == 0:  # 文件夹
            parent_type, parent_id = resolve_parent(file)
            writes.append({"user": f"{parent_type}:{parent_id}", "relation": "parent", "object": f"folder:{file.id}"})
        else:
            parent_type, parent_id = resolve_parent(file)
            writes.append({"user": f"{parent_type}:{parent_id}", "relation": "parent", "object": f"knowledge_file:{file.id}"})

    # 去重后批量写入 OpenFGA（每批 100 条）
    unique_writes = deduplicate(writes)
    checkpoint = load_checkpoint()  # 断点续传：加载上次成功位置
    for i, batch in enumerate(chunks(unique_writes, 100)):
        if i < checkpoint:
            continue  # 跳过已完成的批次
        try:
            await fga.write_tuples(batch)
        except Exception as e:
            if 'already exists' in str(e):
                # 幂等性处理：逐条写入，跳过已存在的元组
                for tuple in batch:
                    try:
                        await fga.write_tuples([tuple])
                    except Exception:
                        pass  # 忽略 already exists 错误
            else:
                save_checkpoint(i)  # 记录失败位置，支持断点续传
                raise
        save_checkpoint(i + 1)
```

**迁移脚本健壮性设计**：

| 特性 | 实现方式 |
|------|---------|
| **幂等性** | 元组写入捕获 "already exists" 错误并跳过，重跑不会报错 |
| **断点续传** | 每批写入成功后记录 checkpoint（文件或 Redis），中途失败后重跑从上次成功位置继续 |
| **dry-run 模式** | `--dry-run` 只生成元组列表不实际写入，输出统计信息（总数、去重数、各类型分布） |
| **verify 模式** | `--verify` 迁移完成后，遍历所有资源和用户，用新系统（OpenFGA）和旧系统（RoleAccessDao）分别 check，对比结果差异并生成报告 |
| **部分失败处理** | 批量写入失败时降级为逐条写入，记录每条失败的元组到日志文件，不阻塞后续批次 |

## 七、关键场景验证

### 场景 1：部门层级权限传递（业务层展开）

**设定**：工程部(id=1) → 后端组(id=2) → API 小组(id=3)。Alice 在 API 小组。管理员将工作流 wf-001 授权给「工程部(含子部门)」查看。

> 因此，当授权给"部门(含子部门)"时，由 `PermissionService.authorize()` 在业务层查出子部门树 [1,2,3]，为每个子部门分别写入元组。

```
(department:2, member, user:bob)       # Bob 直属后端组
(department:3, member, user:alice)     # Alice 直属 API 小组
(department:2, parent, department:1)
(department:3, parent, department:2)

# 授权给"工程部(含子部门)"→ 业务层展开为 3 条元组
(workflow:wf-001, viewer, department:1#member)
(workflow:wf-001, viewer, department:2#member)
(workflow:wf-001, viewer, department:3#member)
```

```python
await fga.check(user="user:alice", relation="can_read", object="workflow:wf-001")

```

## 九、验证方案

1. **单元测试**：为 `PermissionService` 编写测试用例，覆盖上述 5 个场景 + 边界情况
   - 空权限（用户无任何关系）
   - 管理员短路（通过 LoginUser.is_admin()，不走 OpenFGA）
   - 继承深度（多层文件夹）
   - 多路径取并集
   - 部门授权业务层展开（验证子部门成员确实获权）
   - 文件夹移动后权限链变更
2. **迁移验证**：
   - 迁移脚本 `--dry-run`，输出统计信息（总元组数、各类型分布）
   - 迁移脚本 `--verify`，遍历所有资源和用户，对比新旧系统 check 结果差异
   - 验证幂等性：连续执行两次，确认第二次无错误
3. **双写期监控**：记录并分析新旧系统判定不一致的请求
4. **集成测试**：通过 API 端到端测试
   - 创建部门树 → 添加成员 → 资源授权（含子部门展开）→ 验证权限
   - 知识空间 → 文件夹层级 → 文件夹级权限 → 验证继承
   - 文件夹移动 → 验证权限链变更 → 验证缓存失效
   - 跨部门用户组 → 资源授权 → 验证访问
   - 第三方同步 → 验证部门和成员写入正确 → 验证元组联动
   - 资源删除 → 验证 OpenFGA 元组清理
   - 事务补偿 → 模拟 OpenFGA 写入失败 → 验证 failed_tuples 重试机制
5. **性能基准测试**：
   - Check 单次延迟基准（目标 < 10ms p99）
   - ListObjects 在不同元组量下的延迟（100 / 1,000 / 10,000 元组）
   - BatchCheck 批量检查延迟（列表页 20 个资源 × 4 级权限）
   - 文件夹继承深度对 Check 延迟的影响（3 层 / 5 层 / 10 层）
   - 部门授权展开的写入延迟（含 10 / 50 / 200 个子部门的部门树）

## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| OpenFGA 服务不可用 | 所有权限检查失败 | fail-closed 降级 + 健康检查告警 + 至少 2 实例 + 负载均衡（见下方 HA 方案） |
| ListObjects 性能 | 大量关系时列表查询慢 | Redis 缓存 10s + 分页策略 + 超管走 SQL 直查 |
| Model 动态更新风险 | 删除关系类型导致已有元组失效 | P0 使用静态 model；P2 实现时加分布式锁 + validation + 删前清理元组 |
| 迁移数据不一致 | 新旧系统判定不同 | 双写期宽松策略 + 差异监控 + `--verify` 模式 + 人工审核 |
| 元组数据量增长 | 写入 OpenFGA 的元组量大 | 资源删除时同步清理 + 每日定时扫描孤立元组 |
| 第三方同步复杂度 | 飞书/企微/钉钉 API 差异大 | 抽象 Provider 接口 + 各平台独立适配 + 同步日志完整记录 |
| 前端改造工作量 | 新增部门管理、授权弹窗等大量 UI | 分阶段交付：P0 权限引擎 + API → P0 前端适配 → P1 高级功能 |
| MySQL-OpenFGA 事务不一致 | 资源已创建但无权限 | 先写 MySQL 再写 OpenFGA + failed_tuples 补偿队列 + owner 回退逻辑 |
| 部门树变更影响面大 | 子部门新增/移动导致权限元组不同步 | DepartmentChangeHandler 级联维护 + 异步 Celery 任务 + 审计日志 |

### OpenFGA 高可用方案

| 维度 | 方案 |
|------|------|
| **实例数** | 生产环境至少 2 个 OpenFGA 实例，Docker Compose 中 `deploy.replicas: 2` 或独立部署 |
| **负载均衡** | Nginx/HAProxy 反向代理，轮询策略 |
| **健康检查** | OpenFGA 内置 `/healthz` 端点，接入监控告警（Prometheus + Grafana） |
| **Circuit Breaker** | BiSheng 侧实现熔断器：OpenFGA 连续 N 次调用失败后进入熔断状态，快速返回 fail-closed 而非等待超时。熔断恢复后自动重试 |
| **超时控制** | Check/ListObjects 调用设置 3s 超时，避免慢查询阻塞请求线程 |

## 附录：迁移逻辑案例

以"角色 A 的用户可以使用应用 X"这一典型场景，完整演示旧有授权数据如何迁移到 OpenFGA。

### A. 旧系统 (v2.4) 的数据链路

**三张表联动**：

```
user_role 表                role_access 表                     实际含义
┌────────┬─────────┐       ┌─────────┬──────────┬──────────┐
│user_id │ role_id │       │ role_id │ third_id │   type   │
├────────┼─────────┤       ├─────────┼──────────┼──────────┤
│  10    │    A    │  ──→  │    A    │    X     │    5     │  ASSISTANT_READ → 可使用
│  20    │    A    │       │    A    │    X     │    6     │  ASSISTANT_WRITE → 可编辑
│  30    │    A    │       └─────────┴──────────┴──────────┘
└────────┴─────────┘
```

**旧权限检查流程**（`LoginUser.access_check`）：

```
用户 10 访问应用 X
  │
  ├─ ① is_admin()?  → role_id 含 AdminRole(1) 则直接放行
  │
  ├─ ② self.user_id == flow.user_id?  → 是资源创建者则放行
  │
  └─ ③ RoleAccessDao.judge_role_access(
           role_ids=[A],         # 用户 10 初始化时从 user_role 加载的角色列表
           third_id='X',         # 目标资源 ID
           access_type=5         # ASSISTANT_READ
       )
       → SQL: SELECT * FROM role_access
              WHERE role_id IN (A) AND third_id = 'X' AND type = 5
       → 找到记录 → 放行 ✅
```

**关键特征**：`role_access` 是"角色级"授权 —— 一条记录覆盖所有拥有该角色的用户。

### B. 迁移步骤

#### 步骤 1：确定映射关系

| 旧 AccessType | 值 | 新 OpenFGA 对象类型 | 新关系 (relation) | 语义 |
|---|---|---|---|---|
| ASSISTANT_READ | 5 | `assistant` | `viewer` | can_read（含"使用"） |
| ASSISTANT_WRITE | 6 | `assistant` | `editor` | can_edit |
| WORKFLOW | 9 | `workflow` | `viewer` | can_read |
| WORKFLOW_WRITE | 10 | `workflow` | `editor` | can_edit |
| KNOWLEDGE | 1 | `knowledge_space` | `viewer` | can_read |
| KNOWLEDGE_WRITE | 3 | `knowledge_space` | `editor` | can_edit |
| GPTS_TOOL_READ | 7 | `tool` | `viewer` | can_read |
| GPTS_TOOL_WRITE | 8 | `tool` | `editor` | can_edit |
| DASHBOARD | 11 | `dashboard` | `viewer` | can_read |
| DASHBOARD_WRITE | 12 | `dashboard` | `editor` | can_edit |
| WEB_MENU | 99 | **不迁移** | — | 保留 MySQL RBAC |

#### 步骤 2：展开 role_access 为用户级元组

```python
# 输入：role_access(role_id=A, third_id=X, type=5)
#        即"角色 A 可读应用 X"

# 1. 找出拥有角色 A 的所有用户
users_with_role_A = UserRoleDao.get_users_by_role(role_id=A)
# → [user:10, user:20, user:30]

# 2. 映射：ASSISTANT_READ(5) → object_type='assistant', relation='viewer'

# 3. 展开写入 OpenFGA（一条 role_access → N 条 tuple）
for user_id in [10, 20, 30]:
    fga.write_tuple(object='assistant:X', relation='viewer', user=f'user:{user_id}')
```

**结果**——OpenFGA 中写入：

| object | relation | user |
|--------|----------|------|
| `assistant:X` | `viewer` | `user:10` |
| `assistant:X` | `viewer` | `user:20` |
| `assistant:X` | `viewer` | `user:30` |

> **去重优化**：若同一用户对同一资源同时有 READ 和 WRITE（如 role_access 同时有 type=5 和 type=6），只需写入更高级别 `editor`（editor 自动包含 viewer 的所有权限）。

#### 步骤 3：迁移资源所有权

资源表中的 `user_id`（创建者）迁移为 `owner` 关系：

```python
# flow 表：id=X, user_id=7, flow_type=ASSISTANT(5)
fga.write_tuple(object='assistant:X', relation='owner', user='user:7')
```

#### 步骤 4：group_resource — 不迁移

`group_resource` 在旧系统中**不参与 access_check**，仅用于：
1. 资源创建时自动关联到创建者所属的组（前端展示 `group_ids` 字段）
2. 组管理员在管理页查看/转移组内资源

如果将 `group_resource` 直接转为 OpenFGA `viewer` 元组，会导致**权限扩大**——组内用户即使原本没有通过 `role_access` 获得资源的访问权限，迁移后也会意外获权。

**处理方式**：`group_resource` 表直接废弃，不写入 OpenFGA。资源可见性完全由其他迁移步骤产生的元组（role_access 展开、space_channel_member 映射、resource ownership）保证。

#### 步骤 5：迁移 space_channel_member

知识空间/频道的成员制关系直接映射：

```python
# space_channel_member(business_id=ks-10, user_id=7, user_role='creator')
fga.write_tuple(object='knowledge_space:ks-10', relation='owner', user='user:7')

# space_channel_member(business_id=ks-10, user_id=20, user_role='admin')
fga.write_tuple(object='knowledge_space:ks-10', relation='manager', user='user:20')

# space_channel_member(business_id=ks-10, user_id=30, user_role='member')
fga.write_tuple(object='knowledge_space:ks-10', relation='viewer', user='user:30')
```

### C. 迁移前后对比

```
旧系统 (v2.4)                                 新系统 (v2.5)

── role_access 授权链 ──                       ── OpenFGA 直接元组 ──
user:10 → role:A → role_access → app:X        (assistant:X, viewer, user:10)
user:20 → role:A → role_access → app:X        (assistant:X, viewer, user:20)
user:30 → role:A → role_access → app:X        (assistant:X, viewer, user:30)

── 资源 ownership ──                           ── owner 关系 ──
flow.user_id = 7                               (assistant:X, owner, user:7)

── group_resource 归组 ──                      ── 不迁移（废弃）──
group:G → group_resource → app:X              不写入 OpenFGA（仅管理展示，非访问控制）

── space_channel_member ──                     ── 空间关系 ──
creator:7 / admin:20 / member:30              (knowledge_space:ks-10, owner, user:7)
                                               (knowledge_space:ks-10, manager, user:20)
                                               (knowledge_space:ks-10, viewer, user:30)
```

### D. 语义变化说明

迁移后存在一个关键的语义变化：

| 维度 | 旧系统 | 新系统 |
|------|--------|--------|
| 授权粒度 | **角色级**：给角色 A 添加资源 X 权限，所有 A 角色用户自动获权 | **主体级**：直接对用户/部门/用户组授权 |
| 新增用户联动 | 把新用户加入角色 A → 自动获得 A 的所有资源权限 | 把新用户加入角色 A → 仅获得菜单和配额，资源权限需单独授予 |
| 角色职责 | 同时管菜单、配额和资源访问 | 仅管菜单和配额，资源访问由 ReBAC 管理 |
| 授权入口 | 在角色管理页配置 | 在资源详情页、部门/用户组页直接授权 |

> 这意味着迁移后，管理员不再通过"给角色加资源"来批量授权，而是通过"给部门/用户组授权"实现同样的批量效果——且粒度更灵活（可以精确到文件夹级别、操作级别）。
