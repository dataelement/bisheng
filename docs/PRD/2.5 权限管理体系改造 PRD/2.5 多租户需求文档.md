# BiSheng 2.5 多租户需求文档

## 速览（通俗版）

> 给所有读者的入门导读：先看这一节建立心智模型，再读后续技术细节。

**一句话**：一个 bisheng = 一个集团。集团里有"总部"和若干"子公司"，总部能看全部，子公司之间互相隔开。就像中粮总部下面有中粮油、中粮粮食一堆子公司，但他们用同一套 OA、同一套人员目录。

围绕三个核心问题来看本次改造涉及的所有功能。

### 问题 1：每个人看到的数据，谁来分？

**答**：给每条数据贴个"娘家"标签，查询时自动过滤——你是哪家子公司的，就只能看自家的 + 总部公开的。

- **Tenant 树**：总部叫 Root（系统初始化时自动建好，永远删不掉），子公司叫 Child，由总部 IT 在 UI 上"右键标记某个部门为子公司"挂上去。
- **数据贴标签**：知识库、工作流、对话记录等 23 张表都加一个 `tenant_id` 字段。
- **自动过滤**：所有查询自动加上"只看自家的 + 总部共享的"。文件、向量库、Redis 也按子公司分目录存。
- **权限检查 5 道闸**：超管放行 → 子公司管理员放行 → 数据归属对吗 → 资源权限够吗 → 菜单/配额够吗，一道一道过。

详见 §2 数据模型、§3 权限体系、§4 数据隔离实现。

### 问题 2：谁来管？管什么？

**答**：两类管理员，加一套后台工具。

- **两类管理员**
  - **总部超管**（全局超管）：集团 IT，能看能管整棵树（含所有子公司内部）。
  - **子公司管理员**（Child Admin）：只管自家这一摊，看不到别家。
- **超管的特殊技能 —— 切视图（admin-scope）**：超管平时看全树；当他想"临时进入子公司 A 的管理视角"配模型/角色/配额时，点一下切换，4 小时内管理界面就只看子公司 A。**注意：只影响管理菜单，不影响他自己用业务（聊天、跑工作流还是用总部身份）**。
- **管子公司的全套动作**：挂载 / 解绑 / 归档 / 禁用、改名字 logo、设管理员、设配额。
- **配额怎么算**：每家子公司有自己的存储容量额度，同时总部还压一个"集团总上限"——全部子公司存储加起来不能超过。资源数 / Token / 用户数等其他维度推到 v2.6（见附录 G.1）。
- **审计日志**：超管和管理员的所有重要操作（挂载、改配额、转交资源等）全部留痕，谁啥时候动了啥都查得到。

详见 §5 系统管理功能、§6 配额管理体系。

### 问题 3：人和资源怎么在树上流动？

**答**：人跟着主部门走，资源不跟人走，需要共享/移交时有专门通道。

- **登录就自动归属**：你 SSO 登录后，系统沿你的主部门往上找，碰到第一个"被挂载的子公司节点"就把你归到那家；都没碰到就归总部。**没有"切租户"按钮，归属唯一**。
- **调岗/离职怎么办**：你换部门时，名下的知识库、工作流不会跟你走（避免你把集团核心知识库带去子公司）。提供"一键转交"功能，调岗前自己交给同事，或离职后管理员代转。
- **总部资源共享给子公司**：总部创建知识库/频道时勾一下"共享给集团"，所有子公司就能读（默认只读）。
- **LLM 模型也走共享**：总部 IT 统一采购的模型（比如 Azure OpenAI Key）默认下发给所有子公司用；某个敏感子公司想用自己的模型？子公司管理员可以自己注册本地模型，互不干扰。Token 消耗算各自子公司账上。
- **模板下发**：总部做好的应用模板可以"复制"给子公司，子公司拿去后能自己改。
- **组织同步**：本期支持企业微信、飞书、通用 REST API 三种来源，每 6 小时定时拉一次部门和人员；客户也能自己用 HMAC 接口推送。

详见 §5.6 登录与归属派生、§7 集团共享资源、§9 部门同步。

### 几个底层兜底

- **关掉就退化成单租户**：社区版或小客户不需要这套，配置里关一下（`multi_tenant.enabled=false`），行为和老版本一样。
- **升级零成本**：老数据全归到 Root（默认子公司=总部），存储路径不变。
- **MVP 不做的**：3 层以上嵌套、配额跨子公司转移、审批流、合规自动留存——这些都推到 v2.6（详见附录 G）。
- **bisheng 不背的锅**：法律层面的数据保留期、合规审批，由客户自己的 OA/SOC 流程接（详见 §1.5 合规责任声明）。

## 1. 背景与目标

### 1.1 背景

BiSheng 面向大型集团企业（中粮、首钢等）做私有化部署，客户诉求为：

- **统一 SSO + 一棵部门树**：整个集团共享一套 SSO 体系与一棵组织部门树（常见 10 万人级规模）
- **分子公司逻辑隔离**：集团内某些部门子树代表子公司或事业部，需相对隔离 + 独立配额管控
- **集团统一管控**：集团总部需要跨子公司的用量统计、配额管理、资源共享
- **模型服务默认统一、可按 Child 扩展**：多数场景下所有 Child 共享集团 IT 统一采购的模型（如 Azure OpenAI）；但存在子公司需自行配置独立模型服务的场景（数据敏感子公司用独立部署、合规要求独立区域端点、成本分账独立 API Key 等）。因此既支持 Root 统一下发，也允许 Child Admin 自主注册本 Child 专属模型（详见 §7.1）

本次改造采用 **Tenant 树形模型**：整个集团是 1 个 Root Tenant，各子公司作为 Child Tenant 挂载到部门树的子节点。

**作用域声明**： 多租户能力**不包含** SaaS 多客户共存能力bisheng。一个 bisheng 实例只服务一个集团组织（社区版单租户也作为此模型的退化形态——仅 Root，无 Child）。如需服务多个独立法人客户，请独立部署 bisheng 实例。

2.5 版本同步进行权限体系改造（RBAC → ReBAC），二者共享大量改造点（部门树、用户管理、权限检查链路）。合并到同一版本交付，可避免二次改造的成本和风险。

### 1.2 核心概念

| 概念 | 定义 |
|------|------|
| **Tenant（租户）** | 独立管理与配额的边界单元。本实例内通过 `parent_tenant_id` 形成树形结构，MVP 锁 2 层 |
| **Root Tenant** | 本实例唯一的顶级节点（tenant_id=1），代表集团总部。系统初始化时自动创建，不可删除/禁用 |
| **Child Tenant** | 挂在 Root Tenant 之下的子节点，代表集团内的子公司/事业部。可按需挂载/解绑 |
| **叶子 Tenant（Leaf Tenant）** | 用户实际归属的最深层 Tenant。无 Child 挂载的用户归属到 Root；挂载路径下的用户归属到 Child |
| **Tenant 挂载点（Tenant Mount Point）** | 部门树中 `is_tenant_root = true` 的节点。部门树上这一节点及其子树在业务逻辑上属于挂载的 Child Tenant |
| **全局超管（Global Super Admin）** | `system:global` 的 `super_admin`；作用域覆盖 Root 及其所有 Child（即集团 IT 管整棵树）。**两层管理员模型下取代原独立的 "Root 管理员" 角色** |
| **Child 管理员（Child Admin）** | `tenant:{child_id}#admin`，管理本 Child Tenant 内部资源，即"子公司管理员"。不能跨 Child、不能管 Root 配额、不能挂载/解绑 Tenant |
| **管理员模型** | 两层：全局超管 + Child Admin（2026-04-18 Review 决策） |
| **集团共享资源（Group-Shared Resource）** | Root Tenant 创建、标记为共享的资源，自动对所有 Child Tenant 成员可读 |
| **逻辑隔离** | 共享 MySQL 实例，通过 `tenant_id` 字段过滤 + OpenFGA 关系检查实现隔离 |
| **MVP 树深度约束** | Tenant 树锁 2 层。3+ 级集团客户将子公司拍平到同层 Child 管理；UI 禁止 Child 再挂 Grandchild |

**用户与 Tenant 的归属关系**

用户归属唯一的**叶子 Tenant**，由 SSO 登录后根据用户**主部门**沿部门树 path 反向查找最近的 Tenant 挂载点派生：取挂载点的 `mounted_tenant_id` 作为 leaf；若无挂载点，归属到 Root Tenant。

**数据归属规则**

| 数据类别 | 归属 | 说明 |
|----------|------|------|
| 用户基本信息（用户名、头像、个人偏好） | **跟人走**，不受 Tenant 约束 | 存储在 `user` 表，不加 `tenant_id` |
| 对话记录、创建的资源 | **跟叶子 Tenant 走**，按 `tenant_id` 归属 | 创建者的叶子 Tenant |
| 集团共享资源 | 挂在 Root Tenant 的 `tenant_id`，通过 OpenFGA `shared_to` 向 Child 分发读权限 |
| 共享资源衍生数据（对话、消息 token、调用日志） | **跟用户叶子 Tenant 走** | Child 用户读 Root 共享知识库产生的会话归属 Child |
| 操作日志、审计记录 | 按叶子 Tenant 归属，全局超管跨 Child 操作时额外记录 `operator_tenant_id` | 见 §2 audit_log 表 |
| 角色、部门归属 | **跟叶子 Tenant 走**（部门树统一但角色按 Tenant 配置） | — |

### 1.3 层级关系

```
全局超管（集团 IT 管理员，管整棵树；两层管理员模型下取代原 Root Admin）
│
└── Root Tenant（集团总部，tenant_id=1，唯一且不可删除）
    │
    ├── 统一部门树（集团组织架构，由 SSO 同步）
    │   ├── 集团总部（部门）
    │   │   └── 职能部门...
    │   ├── 子公司 A 部门（【Tenant 挂载点】→ 挂 Child Tenant A）
    │   │   ├── 研发部
    │   │   └── 市场部
    │   ├── 子公司 B 部门（【Tenant 挂载点】→ 挂 Child Tenant B）
    │   │   └── ...
    │   └── 事业部 X（不挂载，归 Root Tenant）
    │
    ├── Child Tenant A（子公司 A）
    │   ├── Child Admin A
    │   ├── 配额 A（存储容量）
    │   └── 资源 A（知识空间、频道...）
    │
    ├── Child Tenant B（子公司 B）
    │   └── ...
    │
    └── 集团共享资源（tenant_id = Root）
        └── 通过 shared_to 自动对 Child A/B 可读
```

**核心设计变化（相比原 PRD）**：

- ❌ 原"租户 = 部门树根节点" → ✅ **Root Tenant = 整棵部门树，Child Tenant = 挂载在部门树的某个子节点**
- ❌ 原"用户可属于多个租户" → ✅ **用户归属唯一叶子 Tenant**，切换即违反主部门归属
- ❌ 原"跨租户强隔离" → ✅ **全局超管默认见全树**；Root 创建的共享资源对 Child 可读
- ❌ 原"三层管理员（全局/Root/Child）" → ✅ **两层管理员（全局超管 + Child Admin）**；Root Admin 角色废弃
- ❌ 原 PRD 暗含的"每客户一个 Tenant" → ✅ **本实例固定单 Root**；SaaS 场景由独立部署实例覆盖

### 1.4 目标与优先级

在 `[原 PRD §2]` 基础上新增多租户相关目标：

| 优先级 | 目标 | 说明 |
|--------|------|------|
| **P0** | Tenant 树数据模型与基础隔离 | tenant_id 自动过滤（IN 列表注入）、外部存储隔离、Tenant 上下文透传 |
| **P0** | 权限检查链路集成两层管理员 | 全局超管短路 + Child Tenant admin 短路；资源归属 IN 列表校验 |
| **P0** | 用户叶子 Tenant 派生 | SSO 登录后沿部门 path 派生 |
| **P1** | Tenant 树管理后台 | Child Tenant CRUD、挂载/解绑子公司、配额设置（Root 不可删/禁，仅编辑） |
| **P1** | 子公司挂载工作流 | 全局超管在 UI 右键"标记为子租户"；SSO 协议可选辅助 |
| **P1** | Tenant 级存储配额管理 | leaf 存储配额 + Root 存储硬盖；其他维度（资源数 / Token / 用户数）推到 v2.6 |
| **P1** | 部门同步机制（10 万人关键） | Gateway 实时同步 + Celery 定时校对 + Redis 缓存 |
| **P1** | 所有者交接机制 | 调岗/兼职/离职时资源留原 Tenant，提供批量所有者交接 API |
| **P2** | 集团共享资源机制 | Root 资源通过 OpenFGA `shared_to` 自动共享给 Child |
| **P2** | 模板下发 | 从 Root 下发到各 Child，支持复制 |
| **P2** | Tenant 级审计日志 | 全局超管跨 Child 操作强制留痕 |
| **P2** | Tenant 用量统计仪表盘 | 全局超管视角的 Root+Child 用量聚合 |

### 1.5 合规责任声明

> 2026-04-18 PRD Review 新增。bisheng 作为企业级平台，明确合规边界便于售前/法务沟通。

**产品定位声明**：bisheng 多租户能力以"**集团统一管控**"为核心设计假设。若客户有法人隔离、独立董事会、跨境数据合规（GDPR 等）等严格隔离需求，请在采购前评估以下默认行为是否可接受：

| 默认行为 | 含义 | 合规处理路径 |
|---------|------|-------------|
| 全局超管默认可见全 Root 子树 | 含 Child 的对话记录、审计日志、知识库内容、资源元数据 | 客户通过 NDA / 内部组织管理 / 岗位授权流程控制人员访问 |
| 集团共享资源自动流向 Child | Root 标记为共享的资源对所有 Child 成员可读，Child 不可拒绝接受 | 敏感资源由全局超管选择不共享；知识产权归属由客户内部合约约定 |
| 用户归属切换时资源留原处 | 调岗/兼职/离职后，用户名下资源停留在原叶子 Tenant | 客户通过"所有者交接"功能在调岗前转交资源；超期未交接由 Admin 处置 |
| 数据留存期由客户自行承接 | 取消挂载/注销子公司时，bisheng 仅提供一次性资源处理接口（迁移/归档/手工） | 数据保留期限（网安法 6 年、GDPR 30 天等）由客户 OA/SOC/法务流程触发与管理 |

**bisheng 的合规承诺**：

- 数据一致性：多租户隔离、权限检查、归属校验在代码层准确实施
- 审计日志完整：所有管理员操作强制留痕（含 operator / operator_tenant / action / target / reason / ts），供客户自行审计检索
- 删除操作可追踪：物理删除资源留操作日志，便于事后追溯

**bisheng 不承担的责任**：

- 数据留存期的自动触发（客户负责流程）
- 合规审批流程的内置（客户负责外部 OA/工作流对接）
- 特定行业/区域合规认证（客户负责证书申请与配置）
- 法人级隔离（即跨独立法人实体的 SaaS 级隔离——请独立部署 bisheng 实例）

## 2. Tenant 树形模型设计

### 2.1 Tenant 数据模型

#### tenant 表

```sql
CREATE TABLE tenant (
    id                         INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    tenant_code                VARCHAR(64)   NOT NULL COMMENT 'Tenant 编码（唯一标识）',
    tenant_name                VARCHAR(128)  NOT NULL COMMENT 'Tenant 名称',
    logo                       VARCHAR(512)  DEFAULT NULL COMMENT 'Logo URL',
    root_dept_id               INT UNSIGNED  DEFAULT NULL COMMENT '根部门 ID（该 Tenant 对应的部门树根节点）',
    status                     VARCHAR(16)   NOT NULL DEFAULT 'active'
                                             COMMENT '状态: active / disabled / archived / orphaned',
    contact_name               VARCHAR(64)   DEFAULT NULL,
    contact_phone              VARCHAR(32)   DEFAULT NULL,
    contact_email              VARCHAR(128)  DEFAULT NULL,
    quota_config               JSON          DEFAULT NULL COMMENT 'Tenant 级资源配额（详见 §6）',
    storage_config             JSON          DEFAULT NULL COMMENT 'Tenant 级存储配置覆盖',

    -- 【本次重写】Tenant 树字段（2026-04-20 精简：仅保留 parent_tenant_id）
    parent_tenant_id           INT UNSIGNED  DEFAULT NULL
                                             COMMENT '父 Tenant ID，NULL 表示 Root；MVP 2 层嵌套下非空值永远=Root 的 id',
    share_default_to_children  TINYINT(1)    NOT NULL DEFAULT 1
                                             COMMENT 'Root 创建的资源是否默认共享给所有 Child（仅对 Root 有效）',

    create_user                INT UNSIGNED  DEFAULT NULL,
    create_time                DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time                DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    UNIQUE KEY uk_tenant_code (tenant_code),
    INDEX idx_status (status),
    INDEX idx_parent (parent_tenant_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='Tenant 树形模型';
```

**字段说明**：

| 字段 | 说明 |
|------|------|
| `parent_tenant_id` | 父 Tenant ID。NULL = Root Tenant（本实例唯一）；非 NULL = Child Tenant，其值永远指向 Root（MVP 2 层约束） |

| `status` | `orphaned` 状态：Child 挂载点部门被 SSO 同步删除后进入，不对普通用户可见，待全局超管处理 |

**已废弃字段**（相比 2026-04-18 版本）：

- ~~`tenant_path`~~：2 层下父子关系可直接通过 `parent_tenant_id` 字段枚举，物化路径字段冗余
- ~~`level`~~：通过 `parent_tenant_id IS NULL` 判定即可
- ~~`tenant_kind`~~：通过"parent_tenant_id + 是否存在 Child"可派生；不再作为独立字段

**派生视图**（运行时计算，非数据库字段）：

- 是 Root：`parent_tenant_id IS NULL`（本实例唯一）
- 是 Child：`parent_tenant_id IS NOT NULL`
- Root 是否有 Child：`EXISTS (SELECT 1 FROM tenant WHERE parent_tenant_id = root.id)`

### 2.2 Tenant 与部门树的关系

**核心原则**：

- **Root Tenant**：对应一棵完整的部门树，`tenant.root_dept_id` 指向根节点
- **Child Tenant**：挂载在 Root Tenant 部门树的某个子节点（**Tenant 挂载点**），不单独拥有一棵树
- 整个集团共用 1 棵部门树，Child 通过"挂载点"表达其在部门树中的位置

#### 创建 Tenant 与挂载部门的逻辑

**场景 1：初始化默认 Root Tenant（系统首次部署，自动）**

系统启动/迁移时自动执行（见 §10.2.2），管理员无需手工操作：

1. 创建 tenant 记录（`id=1, parent_tenant_id=NULL, tenant_code='default'`）
2. 自动创建根部门（`parent_id=NULL, is_tenant_root=1, mounted_tenant_id=1`）
3. 回写 `tenant.root_dept_id`
4. 首个用户登录后，手工授予其 `system:global#super_admin`（或升级场景从存量 AdminRole 迁移）

**场景 2：挂载 Child Tenant（集团内新增子公司）**

由全局超管在 UI 上选择一个已存在的部门节点执行"标记为子租户"：

1. 检查 `dept.is_tenant_root = 0`（未被占用）且 dept 不在任何已有 Child 子树内（MVP 锁 2 层，防嵌套）
2. 创建 tenant 记录（`parent_tenant_id=1`）
3. 更新部门：`is_tenant_root = 1, mounted_tenant_id = {new_tenant_id}`
4. 更新 `tenant.root_dept_id = {dept_id}`
5. 刷新子树内所有部门的 `tenant_id`（从 Root 改为 Child）
6. 异步触发：该子树下所有用户的 `UserTenant` 重派生
7. 写入 OpenFGA：`tenant:{new_id}#admin → user:{initial_child_admin_ids}`（Root 的 tenant#admin 不写——全局超管在 system 层管理）
8. 写 `audit_log`：`action='tenant.mount'`

| 部门管理员 | 本部门及子部门（继承部门的 admin 关系） |

```yaml
  group_shared_by_default: true     # Root 创建的资源默认是否共享给 Child

  endpoint_whitelist: []            # Child Admin 注册 LLM Server 时的 endpoint URL 前缀白名单（默认空=不限制）
```

**关闭时的行为**（`multi_tenant.enabled = false`）：

- 仍然存在默认 Root Tenant（tenant_id=1，因 §2.1 Tenant 表是基础设施）
- 前端不展示 Tenant 管理、子租户挂载、Child 配额等入口
- 所有用户归入 Root，JWT 中 `tenant_id = 1`
- API 不要求 tenant_id 参数
- 行为与现有单租户系统完全一致

**开启时的行为**（`multi_tenant.enabled = true`）：

- 前端展示 Tenant 管理菜单（全局超管可见）
- 登录后 JWT 携带派生的叶子 tenant_id
- 所有数据查询自动注入 `WHERE tenant_id IN (leaf + Root)`（2 层下至多 2 个值，含 Root 共享资源）
- 部门树由 Gateway SSO 同步维护，Tenant 挂载由全局超管在 UI 配置

## 3. 权限体系集成

> 以下为 `[原 PRD §3.1]` OpenFGA DSL 和权限检查流程的增量修改。

### 3.1 OpenFGA DSL 增加 tenant 类型

在现有 DSL（`[原 PRD §3.1]`）的 `system` 和 `department` 之间插入：

```yaml
# ============================================
# Tenant 树 — 管理与配额边界
# ============================================
type tenant
  relations
    define admin: [user]                           # Child Tenant 级 admin；不沿 parent 继承
    define member: [user]                          # Tenant 成员
    define shared_to: [tenant]                     # 共享关系：Root → Child 分发可读权限
```

**设计说明**：

| 决策 | 说明 |
|------|------|
| admin 不沿 parent 继承 | 两层管理员模型：全局超管在 `system:global#super_admin` 统一授权，等价于"集团 IT 管整棵树"；`tenant#admin` 仅用于 Child Tenant 内部授权 |
| **Root Tenant 不写 tenant#admin 元组** | Root 级管理由全局超管承担；tenant#admin 仅在 Child Tenant 上写 |
| **不保留 tenant#parent 关系** | parent 关系既不影响 admin 继承、也不影响 member 扩散、shared_to 可直接写 Root→Child 元组；FGA 中冗余，完全依赖 MySQL 的 `parent_tenant_id` 字段 |
| member 不继承 | Child 的 member 不自动成为 Root 的 member，避免跨 Child 可见 |
| `shared_to` 关系 | Root 创建共享资源时，资源的 viewer 连到 `tenant:{root}#shared_to#member`，等价于 Root 下所有 Child 成员均可读 |
| tenant 不参与资源级授权（2026-04-21 收窄） | 资源 viewer 仅通过 `tenant#shared_to#member` 分发**显式共享**；manager/editor 不参与 tenant 级授权；资源授权回归 owner + user + department#member + user_group#member 四源。Tenant 在 FGA 里仅用于：① `admin` 短路（Child Admin，② 级）/ ② `member` 归属反查（不进资源授权）/ ③ `shared_to` 显式共享 |
| department 不加 tenant 关系 | 部门的 Tenant 归属通过 `department.tenant_id` 字段维护；FGA 中不建模避免同步复杂度 |

### 3.2 权限检查流程（五级短路）

`[原 PRD §3.1]` 中的权限检查流程修改为：

```
请求进入
  ↓
① 全局超管检查：system:global#super_admin → 全部放行
  ↓ （非全局超管）
② Child Tenant admin 检查（仅对 Child 类型资源生效）：
   - 若 resource.tenant_id 对应 Child Tenant：FGA check(user, admin, tenant:{tid})
     → 命中放行
   - 若 resource.tenant_id 对应 Root Tenant：跳过此步（Root 资源只走 ①④⑤）
  ↓ （非 Tenant admin）
③ Tenant 归属校验（IN 列表）：
    可见集合 S = {leaf_tenant_id} ∪ {Root_tenant_id}（2 层下至多 2 个，可能 leaf=Root）
    资源 tenant_id 必须 ∈ S 或资源对 S 中任一 tenant 开放共享（shared_to）
    → 不属于则拒绝
  ↓ （归属通过）
④ ReBAC 权限检查：OpenFGA Check（与原 PRD 一致）
  ↓
⑤ RBAC 菜单/配额检查（与原 PRD 一致）
```

**与原 PRD 的差异**：

- ② Child Tenant admin 检查仅对 Child 资源生效；Root 资源的管理完全依赖 ① 全局超管短路
- ③ 归属校验从"严格相等"改为"IN 列表"；2 层下至多 2 个值（leaf + Root，可能 leaf=Root）
- 严格场景（如配额精确计数）通过新增的 `strict_tenant_filter()` 上下文实现（见 §4.1）

### 3.3 管理员身份对照表

**两层管理员模型**（2026-04-18 Review）：合并原独立的 "Root 管理员" 角色。

| 角色 | OpenFGA 关系 | 作用域 | 典型操作 |
|------|-------------|--------|---------|
| 全局超管 | `system:global#super_admin` | Root + 所有 Child（集团全树） | Child Tenant CRUD、挂载/解绑 Child、管理部门/用户/角色/资源、设置 Root 与 Child 配额、管理集团共享资源 |
| Child 管理员 | `tenant:{child_id}#admin` | Child 内部 | 管理本 Child 的资源与成员；无法跨 Child、无法管 Root 配额、无法挂载/解绑 Tenant |
| 部门管理员 | `department:{id}#admin` | 部门内 | 管理本部门及子部门（与 `[原 PRD §3.2.1]` 一致） |

**隐含规则**：

- 原独立的"Root 管理员"角色已合并到全局超管；`tenant#admin` 不沿 `parent` 继承
- Root Tenant **不写 `tenant#admin` 元组**；Root 的管理权完全走全局超管路径（① 短路）
- 全局超管对 Child 内部资源的管理权通过 ① 全局短路实现，不依赖 FGA 继承链
- Child A admin 无法管理 Child B（FGA 检查会失败）
- 部署期内通常只授予集团 IT 核心人员全局超管权限；客户通过 NDA / 岗位流程管控可见性（见 §1.5）

**Child Admin 能力清单**（细化）：

| 能力 | 是否可做 |
|------|---------|
| 本 Child 内部资源 CRUD | ✅ |
| 本 Child 成员邀请/移除 | ✅ |
| 本 Child 角色管理 | ✅ |
| 本 Child 配额查看 | ✅ |
| 本 Child 配额修改 | ❌ 仅全局超管 |
| 挂载/解绑 Tenant（创建 Grandchild） | ❌ MVP 锁 2 层 |
| 集团共享资源 | ✅ 只读 |
| 跨 Child 操作 | ❌ |
| 查看本 Child 相关审计日志 | ✅（含全局超管对本 Child 的操作） |
| 本 Child LLM Server / Model CRUD（注册外部模型端点、配置 API Key） | ✅（2026-04-19 决策 3，详见 §7.1） |
| 本 Child 模型 token 配额设置 | ❌ MVP 不做 token 维度配额，推到 v2.6（见附录 G.1） |
| 修改 Root 共享模型 | ❌ 只读；PUT/DELETE 返回 403 + 19801 |
| 进入 Root 管理视图（admin-scope） | ❌ 仅全局超管可切换（详见 §5.1.5） |

**Child Admin vs 部门管理员（易混区分）** `[2026-04-25 补充]`

PRD Review 时这两个角色频繁被混淆。它们是**不同维度的管理员**，FGA 关系独立、字段独立、互不级联：

| 维度 | **Child Admin（子租户管理员）** | **Department Admin（部门管理员）** |
|------|-------------------------------|----------------------------------|
| **管什么** | 整个 Child Tenant（一个子公司/事业部一锅端） | 部门树某节点 + 其所有子部门 |
| **粒度** | 粗（Tenant 级） | 细（部门子树级） |
| **边界依据** | `tenant_id`（FGA `tenant:{child_id}#admin`） | `department.path`（FGA `department:{id}#admin`，沿 parent 继承） |
| **后端字段** | `is_child_admin`（FGA `tenant:{leaf}#admin` check） | `is_department_admin`（`DepartmentDao.aget_user_admin_departments` 非空） |
| **本范围 LLM** | ✅ LLM Server / Model 完全自主 CRUD（§7.1 决策 3） | ❌ 不涉及模型管理 |
| **本范围配额** | ✅ 仅查看，不能改 | ❌ 不涉及配额 |
| **本范围角色** | ✅ Child 角色 CRUD（§5.5） | ⚠️ 仅在自身部门 scope 内可见 |
| **审计日志** | ✅ 本 Child 全量（含全局超管对本 Child 的操作） | 仅本部门子树 |
| **谁授权** | 全局超管在挂载 Child 时指定（§5.2.1 / §2.2 第 7 步） | 全局超管 / 上层部门管理员通过部门 admin grant 指定 |
| **引入版本** | v2.5 多租户新增 | v1.x 已存在，v2.5 沿用 |

**关键事实（与直觉相反）**：

- 两套 FGA 关系**完全独立**。§2.2 / §5.2.1 挂载 Child 流程只写 `tenant:{new_id}#admin → user:{initial_child_admin_ids}`，**不会**自动写挂载点部门的 `department:{dept_id}#admin`
- 因此默认状态下 Child Admin 的 `is_child_admin=true` 但 `is_department_admin=false`
- 能力上 Child Admin 边界 ⊋ 挂载点部门管理员（粗粒度包含细粒度），但**身份标记上互不蕴含**——需要业务流程在挂载时额外授予才能让两者重叠

**具体场景对照**（参考 §1.3 层级图）：

```text
Root Tenant: 集团总部
└── 部门树
    ├── 子公司 A 部门 ←【Tenant 挂载点】→ Child Tenant A
    │   ├── 研发部     ← 部门管理员 P
    │   │   └── 算法组 ← 部门管理员 Q
    │   └── 市场部
    └── ...
```

| 角色 | 边界 | 能管的事 |
|------|------|---------|
| Child A Admin | 整个 Child Tenant A | 本 Child 成员/角色/LLM/配额查看/审计 |
| 部门管理员 P（研发部） | 研发部 + 算法组 | 这两个部门的成员归属，**不动** Child A 的 LLM/配额/角色 |

## 4. 数据隔离实现

### 4.1 查询层自动过滤

**重写说明**：原 PRD 使用 `WHERE tenant_id = {current}` 严格相等过滤。新方案改为 **`IN 列表注入`**，覆盖 leaf + Root（2 层下至多 2 个值），以支持"子公司读集团共享资源"等场景。

所有包含 `tenant_id` 字段的业务表，在 SELECT 查询时自动注入：

```sql
WHERE tenant_id IN (
    {leaf_tenant_id},
    {root_tenant_id}         -- leaf != root 时额外加入，用于读集团共享
)
```

**可见集合计算规则**：

| 用户身份 | 可见 Tenant 集合 |
|---------|----------------|
| 全局超管（未设 admin-scope） | `None`（不注入 WHERE，可见全部） |
| 全局超管（已设 admin-scope=X） | `{X, root_id}`（仅管理类 API 生效，详见 §5.1.5） |
| Child 管理员 | `{child_id, root_id}`（本 Child + Root 共享） |
| 普通用户（叶子在 Child） | `{child_id, root_id}` |
| 普通用户（叶子在 Root） | `{root_id}` |

**admin-scope 的作用域**（2026-04-19 决策 1 新增）：

- **仅管理类 API 受影响**（`/api/v1/llm`、`/api/v1/roles`、`/api/v1/tenants/{id}/quota`、`/api/v1/audit_log` 等系统管理入口）
- **业务 API 不受影响**（对话、工作流执行、知识库检索等仍走 JWT leaf_tenant_id），避免超管在 scope 下误发起用户级业务操作导致数据归属错乱
- 具体切换机制、失效时机、审计记录见 §5.1.5

**严格隔离上下文（`strict_tenant_filter`）**：

配额计数、审计日志等场景需要精确匹配某个 Tenant。通过新增 ContextVar 实现：

```python
with strict_tenant_filter():
    # 此 block 内查询只注入 WHERE tenant_id = {current}
    count = get_tenant_resource_count(tenant_id)
```

**安全保障**：

- 用户请求必须有叶子 Tenant 上下文，否则查询**不返回任何数据**（防止开发疏漏）
- 全局超管跳过注入但需通过 `is_global_super()` 显式判定
- Celery 异步任务必须从任务 headers 恢复叶子 Tenant 上下文
- 13 处 `bypass_tenant_filter()` 调用保留但加审计日志，便于排查

> 具体实现方案（SQLAlchemy event 机制）见技术方案文档。

### 4.2 写入层自动填充

创建记录时，如果表包含 `tenant_id` 字段且未手动赋值，自动从当前请求上下文获取用户叶子 Tenant ID 填充。

**集团共享资源的写入**（新增）：

全局超管在创建资源时若选择"共享给集团"：

1. API 层接收 `share_scope: 'group'` 标志
2. 后端用 `with set_current_tenant_id(root_tenant_id):` 包裹写入
3. 资源的 `tenant_id` 写为 `root_id`
4. 同步写 FGA tuple：资源 viewer → `tenant:{root}#shared_to#member`
5. 前端将"共享"资源显示给所有 Child 用户（通过 §4.1 的 IN 列表自然命中）

> 具体实现方案见技术方案文档。

### 4.3 Tenant 上下文传递

本章节保留 `[原 PRD §4.3]` 的机制，仅语义调整为"叶子 Tenant"：

| 场景 | 传递方式 |
|------|---------|
| **HTTP 请求** | JWT token 中携带 `tenant_id`（= 叶子 Tenant ID）。中间件解析后设置 `current_tenant_id` ContextVar |
| **WebSocket** | 连接建立时从 JWT 解析，绑定到连接上下文 |
| **Celery 异步任务** | 发送任务时将 `tenant_id` 写入 task headers，Worker 执行前从 headers 恢复 ContextVar |
| **定时任务（Beat）** | 遍历所有活跃 Tenant 逐个执行 |

**新增 ContextVar**：

| ContextVar | 说明 |
|-----------|------|
| `current_tenant_id` | 叶子 Tenant ID（与原 PRD 一致） |
| `current_user_tenant_role` | 用户角色，取值 `global_super / child_admin / member`（两层模型下无 `root_admin`），供 §4.1 IN 列表计算使用 |
| `_strict_tenant_filter` | 严格精确匹配上下文标志（配额/审计计数用） |

### 4.4 外部存储隔离

#### 4.4.1 Milvus 向量存储

沿用 `[原 PRD §4.4.1]` Collection 名称前缀方案：

- Root Tenant（tenant_id=1）保持原命名（向后兼容）
- Child Tenant：`t{child_id}_col_{timestamp}_{uuid}`
- **集团共享**：Root 创建的共享资源使用 Root 的 collection，Child 读取时由应用层回退到 Root collection（通过 §4.1 IN 列表命中）

#### 4.4.2 MinIO 对象存储

沿用 `[原 PRD §4.4.2]` 路径前缀方案：

```
bisheng/                          # 共享 bucket
├── default/                      # Root（tenant_id=1，集团总部）
│   ├── knowledge/                # Root 自己的资源
│   ├── tenant_subA/              # Child Tenant A（子公司 A）
│   │   └── knowledge/
│   └── tenant_subB/              # Child Tenant B（子公司 B）
│       └── knowledge/
```

- Root 保持原路径不变（无前缀）
- Child：`tenant_{child_code}/` 前缀（相对于 Root 下）
- **Root 共享资源读取**：Child 用户读共享资源时，后端用 Root 的前缀重新拼接路径（应用层 fallback）

**独立存储模式**（`[合入自多租户管理 PRD]`）保留，可按 Child 级别配置独立 bucket/AK/SK，通过 `tenant.storage_config.mode='independent'` 启用。详见原 PRD §4.4.2。

#### 4.4.3 Elasticsearch

- Root 保持原 index 命名
- Child：`t{child_id}_{original_index_name}`
- 查询自动附加 IN 列表中所有 Tenant 的 index 名称

#### 4.4.4 Redis

- Root 保持原 key 格式
- Child：`t:{child_id}:{original_key}`
- 封装 tenant-aware 工具类，自动解析叶子 Tenant 前缀
- 部门树/用户归属缓存用新 key：`dept:tree:root`、`dept:user:{uid}:primary`、`dept:tenant:{tid}:subtree_ids`

## 5. 系统管理功能

### 5.1 Tenant 树管理

#### 5.1.0 首次登录初始化 `[合入自多租户管理 PRD，本次修订]`

**多租户关闭时**（`multi_tenant.enabled = false`）：
- 沿用现有逻辑：首个用户自动成为全局超管，归入默认 Root

**多租户开启时**（`multi_tenant.enabled = true`）：
- 系统初始化时自动创建默认 Root Tenant（tenant_id=1）
- 首个通过 Gateway SSO 登录的用户自动授予 `system:global#super_admin`；若存在存量管理员数据（升级场景），从 AdminRole 迁移
- 集团根部门通过 SSO 同步自动就绪
- 全局超管通过 UI 对已有部门"标记为子公司"挂载 Child Tenant（按需，不强制）

#### 5.1.1 Tenant 列表（树形视图）

| 项目 | 说明 |
|------|------|
| 入口 | 系统管理 → Tenant 管理（`multi_tenant.enabled=true` 时可见） |
| 可见性 | 全局超管看整棵树；Child admin 仅看自身 Child |
| 视图形态 | **树形展开**：Root 一级固定显示（不可删除），展开后显示其 Child |
| 列表字段 | 名称、编码、状态（active/disabled/archived/orphaned）、用户数、存储用量/配额、创建时间 |
| 存储用量 | 实时统计叶子 Tenant 的 MinIO 文件总容量，进度条形式 `[合入自多租户管理 PRD]` |
| 操作 | 搜索（名称/编码）、筛选（状态）、分页 |
| 按钮 | **无"新建 Root"按钮**（Root 由系统初始化创建，唯一且不可删除）；全局超管可"挂载子公司"（见 §5.2） |

#### 5.1.2 Root Tenant 初始化（自动，非手工操作）

Root Tenant 在系统初始化/升级迁移时由后端脚本自动创建（见 §10.2.2），**不再作为用户可见的 CRUD 操作**。

| 字段 | 初始值来源 |
|------|----------|
| tenant_id | 固定为 1 |
| tenant_code | 配置项 `multi_tenant.default_tenant_code`（默认 `default`） |
| tenant_name | 配置项 `multi_tenant.default_tenant_name`（默认 `默认租户`，首次登录后全局超管可改） |
| parent_tenant_id | NULL |
| quota_config | 初始为空（无限）；全局超管登录后可编辑 |

**可选编辑**：全局超管登录后可通过 `PUT /api/v1/tenants/1` 修改 Root 的名称、Logo、联系人、配额；**不能修改 tenant_code、不能删除、不能禁用**（见 §5.1.3）。

#### 5.1.3 编辑 Tenant

| 可修改 | 不可修改 |
|--------|---------|
| 名称、Logo、联系人、配额 | 编码、parent_tenant_id |
| Child：状态（active/disabled）、tenant admin（增减、重置密码） | 拓扑位置 |

**Root Tenant 特殊保护**（2026-04-20 新增）：

- Root Tenant（tenant_id=1）**不可禁用**（disable）
- Root Tenant **不可归档/物理删除**
- API 层对 Root 的 `PUT /status` / `DELETE` 请求返回 403 "Root Tenant 受系统保护"
- 原因：Root 禁用会导致全集团所有用户无法登录；Root 删除等于系统报废

**禁用 Child Tenant** `[合入自多租户管理 PRD 细化]`：

拨至"关闭"时弹出二次确认："*停用后，该子公司下所有成员将被立即强制下线，且无法再次登录系统。是否确认停用？*"

确认后执行：

1. **立即吊销**该 Child Tenant 下活跃账号的 JWT Token
2. **API 层拦截**所有携带该 tenant_id 的请求，返回 403 "Tenant 已停用"
3. 该 Child 下 Celery 异步任务挂起：**已入队执行的任务允许完成**（避免大文件上传半途废弃），**新任务挂起入队**但不执行；重启用时自动恢复出队
4. 该 Child 的第三方同步定时任务暂停
5. 数据保留，重启用后恢复
6. 写 `audit_log`：`action='tenant.disable'`

> **"立即下线"的适用范围**：上述 1~6 步是数据冻结期所需的强制下线动作，仅在「全局超管手动 disable Child Tenant」（本节）场景触发。
>
> **不适用解绑（unmount）**：解绑子租户的语义是"资源整体迁回 Root + Child 归档"（见 §5.2.2），用户连同资源迁回总部，业务无感、无冻结期需求。下一次请求经 `UserTenant` 派生表自然落到 Root 上下文（参见 §5.6.1 + INV-T4），无需吊销 JWT 或拦截 API。

**Child Tenant 归档/物理删除**：

- 归档：数据保留，全局超管可恢复
- 物理删除：需二次确认
- 删除前检查：运行中的应用、活跃用户、异步任务

#### 5.1.4 Tenant 配额设置

详见 §6。每个 Tenant（Root 或 Child）均可独立设置配额；系统沿 Tenant 树取最严。

#### 5.1.5 管理视图切换（Admin-Scope）

> 2026-04-19 新增。解决全局超管在 Tenant 树下跨 Child 管理资源（LLM 模型、角色、配额、审计日志等）时缺少"当前管理哪个 Child"上下文的问题。

**动机**：

- 集团 IT（全局超管）经常需要临时进入某个 Child 的视角，查看/配置该 Child 专属的资源（例如在 Child X 下注册本 Child 专用模型、调整本 Child 角色）
- 不能走"重签 JWT 切换归属"路径（会破坏用户数据归属一致性，见 §5.6.1 + INV-T4）
- 因此引入独立于 JWT 的 **管理视图 scope** 机制：Redis 驻留、仅管理类 API 生效、不影响用户归属

**与"用户归属切换"的区别**：

| 项 | 用户归属切换（已废弃，410 Gone） | 管理视图切换（本节新增） |
|----|----------------------------------|--------------------------|
| API | `POST /api/v1/user/switch-tenant` | `POST /api/v1/admin/tenant-scope` |
| 状态 | 永久废弃（见 §10.4） | v2.5.1 新增 |
| 是否重签 JWT | 是（旧设计） | 否（不改 JWT） |
| 存储 | JWT Payload | Redis `admin_scope:{user_id}` |
| 适用对象 | 曾面向所有用户 | 仅全局超管 |
| 影响 | 切换后全部 API 按新 tenant_id 运行 | 仅管理类 API 改变查询上下文 |
| 对用户数据归属 | 改变归属（违反主部门派生） | **不改变归属** |

**API**：

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | `/api/v1/admin/tenant-scope` | body `{tenant_id: int \| null}`；null 表示清除 scope，恢复"看全树" | 全局超管 |
| GET | `/api/v1/admin/tenant-scope` | 返回当前 scope（供前端显示、刷新） | 全局超管 |

**后端行为**：

- POST 接收后写 Redis key `admin_scope:{user_id}` = `tenant_id`，TTL = `multi_tenant.admin_scope_ttl_seconds`（默认 14400s = 4h，滑动刷新）
- 返回 `{scope_tenant_id, expires_at}`
- 每次命中管理类 API 时中间件执行 `EXPIRE admin_scope:{user_id} 14400`（滑动续期）
- 非全局超管请求此 API：返回 403 + 错误码 19701 `llm_scope_forbidden`

**生效规则**：

| 请求类型 | 是否受 scope 影响 | 行为 |
|---------|-------------------|------|
| 普通 API（对话、工作流执行、知识库调用等） | 否 | 仍使用 JWT 的 leaf_tenant_id |
| 管理类 API（`/llm`、`/roles`、`/tenants/{id}/quota`、`/audit_log` 等） | 是 | 读取 Redis scope 覆盖 §4.1 IN 列表为 `{scope_tenant_id, root_id}` |
| 全局超管未设 scope | — | 保持"看全树"默认行为（IN 列表 = None） |
| Child Admin（已在自己 Child） | 不提供 | Child Admin 登录即处于本 Child 上下文，无切换需求；调用 POST 返 403 |

**失效时机**（6 条，逐个说明）：

1. **用户主动切换**：`POST /admin/tenant-scope` 携带新 `tenant_id` → Redis SET 覆盖旧值
2. **用户主动清除**：`POST /admin/tenant-scope` body `{tenant_id: null}` → Redis DEL
3. **登出**：AuthService logout 钩子调 Redis DEL `admin_scope:{user_id}`
4. **主部门变更导致 token_version +1**：旧 JWT 失效，同步 DEL admin-scope
5. **超管角色被撤销**：role 变更钩子触发 DEL（用户不再具有使用该 API 的资格）
6. **scope 指向的 Child 被禁用/归档/删除**：Celery 每 10 分钟巡检 `admin_scope:*`，value 命中非 active Tenant 的 key 主动 DEL

**审计记录**：

- 每次 POST（含清除）写 `audit_log`，`action = admin.scope_switch`
- payload 字段：`from_scope`（切换前 tenant_id 或 null）、`to_scope`（切换后 tenant_id 或 null）、`ip`、`user_agent`、`operator_id`
- 便于安全团队事后检测异常模式（高频切换、跨地理位置同时出现）

**安全边界**（与 JWT 盗用场景的关系）：

- admin-scope 不重签 JWT → 攻击者盗取超管 JWT 后可任意设置 scope
- 但超管 JWT 本就能访问全树，scope 机制**不扩大**攻击面
- 防御依赖 JWT 本身的安全（HTTPS、HttpOnly Cookie、TTL）+ audit_log 事后追溯

### 5.2 子租户挂载工作流

**背景**：集团场景下，全局超管需要把部门树的某些子节点"标记为子租户"，在业务层形成隔离边界。此操作在 bisheng UI 完成。

#### 5.2.1 主入口：UI 手工标记（推荐流程）

```
全局超管登录 → Tenant 管理 → 部门树面板 → 右键选择一个部门节点 → "标记为子租户"
  → 弹窗填写：
      - 子租户名称（tenant_name）
      - 初始配额（存储容量 storage_gb，可复用模板）
      - 初始 Child Admin（用户搜索多选）
  → 后端 POST /api/v1/departments/{dept_id}/mount-tenant
  → 事务执行 §2.2 场景 2 的 8 步挂载流程
  → UI 节点加子租户角标 + 悬浮显示配额用量摘要
```

**约束**：

- 仅全局超管可操作
- **强制操作日志**：所有挂载/解绑操作记录在 `audit_log`（operator、操作时间、子租户 ID、配额值、初始管理员、原因），供客户自行审计；不提供撤销，留日志即可追责
- 不允许挂载到已经是挂载点的部门或其子树内（**MVP 锁 2 层，UI 强制校验**：Child Tenant 下不可再挂载）
- 不允许挂载到 Root 的根部门（根部门默认即 Root Tenant 的挂载点）

**挂载确认 review 步骤**（2026-04-19 新增，对应 §7.1.6 + D.1 风险缓解）：

挂载弹窗提交前显示 review 面板，预览"本 Child 将自动可见的 Root 共享资源列表"：

- 所有 Root 已共享的 LLM Server / Model（含数量、名称、model_type）
- 其他 §7.2 共享资源（知识库、频道、应用模板等）

弹窗提供选项：

- **自动分发**（默认勾选）：挂载后上述资源对新 Child 立即可见；事务内写 `audit_log`，action=`tenant.mount`，payload 含自动分发的资源 id 列表
- **不自动分发**：挂载完成后 Child 不能看到任何 Root 共享资源；后续由全局超管 / Root owner 逐资源手工共享（对应 F017 的精细共享能力）

挂载日志中记录所选分发策略，便于事后撤销或审计。

#### 5.2.2 取消挂载（解绑子租户）

**定位**：bisheng 只提供一次性解绑接口，不内置审批流、不定义数据留存期；留存合规/告警由客户 OA/SOC 外部流程承接（见 §1.5《合规责任声明》）。

**唯一路径：资源迁移到 Root + Child 归档**

解绑无策略选择，行为固定：

```
全局超管在部门页/Tenant 详情页点击"解绑子租户" → 二次确认 → 单步事务：

  1. 资源迁移：该 Child 下所有资源 tenant_id 批量改为 root_id
  2. 更新部门：is_tenant_root=0, mounted_tenant_id=NULL（清挂载点）
  3. UserTenant 重派生：子树下用户的 leaf_tenant_id 自动回到 Root
  4. Child Tenant 状态置为 archived（壳保留供审计追溯，下不再有任何资源）
  5. 撤销 FGA tenant:{child}#shared_to → tenant:{root} 元组
  6. 写 audit_log：action='tenant.unmount'

  ※ 不吊销 JWT、不拦截 API：用户当前会话继续可用，下一次请求经中间件
    按新派生关系自然落到 Root 上下文（JWT 内只有 user_id，租户归属
    每次请求查派生表，参见 §5.6.1 + INV-T4）
```

**为什么解绑不需要"立即下线"**：

- "立即下线"的本意是给数据冻结期上锁（防止用户在资源处理过程中继续写数据，导致归属/所有权状态打架）
- 解绑动作是"用户连同资源整体迁回 Root"，没有冻结期需求；员工视角只是发现"我现在归集团总部了"，业务连续性不被打断
- bisheng 的 JWT 仅承载 `user_id`，租户归属由每次请求查 `UserTenant` 派生表得出（§5.6.1 + INV-T4），所以重派生后下一次请求自动生效，无需重签 JWT、也无需踢人
- 若客户希望"冻结子公司"而不迁移资源，应使用 §5.1.3 的"手动 disable Child Tenant"，与解绑分开管理

**不在本 MVP 范围内**（客户外部流程承接）：

- 数据留存期的自动计算与到期清理
- 解绑动作的二次审批
- 合规报告模板生成
- 子租户注销后的数据交付包
- "归档保留资源不迁移"或"逐项手工处理"等替代解绑路径——本期不提供，留客户用 disable + 手动 SQL 处理

#### 5.2.3 辅助入口：SSO 协议可选字段 （本期可不做）

SSO 登录同步端点 `POST /api/v1/internal/sso/login-sync` 的 payload 支持可选扩展字段：

```json
{
  "external_user_id": "...",
  "primary_dept_external_id": "...",
  "secondary_dept_external_ids": [...],
  "user_attrs": {...},
  "tenant_mapping": [                             // ← 可选辅助字段
    {
      "dept_external_id": "SUB_001",
      "tenant_code": "sub_finance",

      "initial_quota": {...},
      "initial_admin_external_ids": [...]
    }

```

**处理规则（幂等）**：

- 若 dept 曾被删除后重新同步出现：视为新部门，重新 mount 但触发告警（全局超管确认是否重建）

#### 5.2.4 批量初始化策略（MVP）

**MVP 不提供 Excel 批量导入**。首次上线若有 20+ 子公司，由全局超管在 UI **逐个手工挂载**（一次性运营投入，预计 1~2 小时，可接受）。后续版本可加批量能力。

#### 5.2.5 同步鲁棒性（标记独立）

**原则**：子租户挂载标记存储在 bisheng（`Department.is_tenant_root + mounted_tenant_id`、`Tenant.parent_tenant_id`），**独立于 SSO 同步**。

- SSO 同步仅维护 `Department.external_id ↔ Department.id` 映射和部门的父子/命名变更
- 同步不触碰 `is_tenant_root` 和 `mounted_tenant_id` 字段
- **孤儿处理（MVP 简化）**：若 SSO 同步发现某 tenant 挂载点对应的部门被删除：
  - 该 Child Tenant 的 `status` 置为 `orphaned`（不再对普通用户可见）
  - 写 `audit_log` + 发站内消息给全局超管
  - **不定义 SLA**：bisheng 不承诺处理时限；是否重新挂载/归档/迁移资源由客户自行决定
  - 长期未处理的 orphan 由客户 OA/SOC 流程监控（见 §1.5）

### 5.6 登录与归属派生

**原 PRD §5.5 "登录与租户切换"在本次重写中被重新定义**：**去除"切换租户"概念**，用户归属唯一叶子 Tenant，登录时自动派生。

#### 5.6.1 登录流程

```
用户访问集团统一入口
  ↓
Gateway SSO（OAuth/LDAP）认证通过
  ↓
Gateway 同步用户信息到 bisheng（POST /internal/sso/login-sync）
  ↓
bisheng 派生叶子 Tenant：
    - 查用户主部门
    - 沿 Department.path 反向找最近的 is_tenant_root 节点
    - 取其 mounted_tenant_id 作为 leaf_tenant_id
    - 若无挂载点 → leaf_tenant_id = 1（Root）
  ↓
更新 user_tenant 记录（自动维护，见 §5.6.3）
  ↓
签发 JWT，tenant_id = leaf_tenant_id
  ↓
进入工作台
```

#### 5.6.2 JWT Token 结构

在 `[原 PRD]` JWT payload 基础上：

```json
{
  "user_id": 7,
  "user_name": "zhangsan",
  "tenant_id": 15,          // 叶子 Tenant ID（Root=1 或 Child=N）
  "token_version": 3,       // 主部门变更时 +1，强制旧 token 失效
  "exp": 1700000000
}
```

**token_version 失效策略**：用户主部门变更导致叶子 Tenant 变化时，`user.token_version` 自增，旧 JWT 在下次请求时被拒绝，要求重新登录获取新的 JWT。

**与 2026-04-18 版本的差异**：去掉 `tenant_path` 字段（2 层下可见集合直接由 `tenant_id` 和 Root 固定 id=1 推导，无需 payload 承载）。

#### 5.6.3 用户归属自动维护

由 `UserTenantSyncService.sync_user(user_id)` 服务维护，触发场景：

- SSO 登录时（`POST /internal/sso/login-sync`）
- 主部门变更时（`UserDepartment.is_primary` 变更的钩子）
- Celery 定时校对任务（每 6h，详见 §9 同步机制）
- Tenant 挂载/解绑时：挂载时该部门子树下所有用户重派生到新 Child；解绑时立即重派生回 Root（用户无感、资源整体迁回，详见 §5.2.2）

执行逻辑：

1. 调用 `resolve_user_leaf_tenant(user_id)` 计算新叶子 Tenant
2. 比对 `UserTenant` 中 active 记录，若不一致则 upsert
3. 使 Redis 缓存 `user:{uid}:leaf_tenant` 失效
4. 重建 OpenFGA 元组：撤销旧 `tenant:{old}#member`，写入新 `tenant:{new}#member`
5. **数据归属不迁移**（2026-04-18 PRD Review 决策）：用户历史创建的资源 tenant_id 保持不变，留在原叶子 Tenant。避免调岗带走集团核心知识库
6. **若用户名下有资源**，触发 Admin 告警（见 §5.6.3.1 所有者交接），可选配置 `user_tenant_sync.enforce_transfer_before_relocate=true` 强制阻断主部门同步
7. `user.token_version` +1 强制旧 JWT 失效

#### 5.6.3.1 所有者交接（P0-C 决策）

> 2026-04-18 PRD Review 新增。解决"归属跟主部门切，数据留原处不动"带来的资源孤儿问题。调岗/兼职/离职统一走此机制。

**设计目标**：用户主部门变更即归属切换（§5.6.3），但其名下资源（知识空间、工作流、助手、工具、频道等）**不随人迁移**。为避免资源成为无主孤儿，提供批量"所有者交接"能力。

**核心 API**：

```
POST /api/v1/tenants/{tenant_id}/resources/transfer-owner

Body:
{
  "from_user_id": 123,
  "to_user_id": 456,
  "resource_types": ["knowledge_space", "workflow", "assistant", "tool", "channel", "folder", "knowledge_file"],
  "resource_ids": null,
  "reason": "张三调岗到子公司 X，移交知识库给李四"
}

Returns:
{
  "transferred_count": 18,
  "transfer_log_id": "txn_20260420_abc123"
}
```

**事务语义**（2026-04-20 修订 / 2026-04-21 补充）：

1. 校验 from_user 在 tenant_id 下确实拥有这些资源（OpenFGA owner 元组）
2. **校验 to_user 的叶子 Tenant 在 tenant_id 的可见集合内**（即 to_user 叶子 ∈ {tenant_id, tenant_id 的 Root}）——支持从 Child 交接给 Root 成员的典型"交回总部"场景
3. 批量撤销旧 owner 元组，写入新 owner 元组
4. 资源的其他授权（viewer/editor/manager）保持不变
5. 写 audit_log（from_user, to_user, 资源列表、操作时间、原因、操作人）
6. 失败回滚，保证原子性

**不在本 API 范围的交接路径**：

- **Root → Child（资源下沉）**：tenant_id=Root 时可见集合 = `{Root}`，to_user 在 Child ∉ 集合，本 API 拒绝。此场景需先通过 F011 "资源迁移" API（`POST /api/v1/tenants/{child_id}/resources/migrate-from-root`）把资源的 `tenant_id` 改为 Child，再于 Child 内部交接 owner
- **Child A → Child B（跨 Child 交接）**：同理不支持；典型走法是 Child A → Root（先交回总部），再由总部走下沉路径到 Child B

**触发场景**：

| 场景 | 触发时机 | 交接发起方 |
|------|---------|-----------|
| 调岗（主部门变更） | SSO 同步检测到用户主部门跨 Tenant 变更 | 调岗前用户自行交接 / 超期由 Admin 批量处理 |
| 兼职部门增减 | `UserDepartment.is_primary=false` 的变更无需交接（不改归属） | — |
| 离职 | HR 系统触发用户禁用 | Admin 在离职前代为交接；离职后资源进入待处理队列 |

**告警与阻断**：

- SSO 同步检测到用户主部门跨 Tenant 变更时，查询用户名下资源数
- 若 > 0，发送站内消息 + 邮件给原 Tenant 和新 Tenant 的 Child Admin
- 配置项 `user_tenant_sync.enforce_transfer_before_relocate`（默认 `false`）：
  - `false`：允许调岗同步完成，资源成为"待交接"状态
  - `true`：阻断主部门变更，要求先完成交接才能调岗

**用户自服务入口**：

- 用户个人中心 → "我的资源" → 批量选择 → "转交给..."
- 全局超管页面 → "待交接资源"清单（离职用户/超期未交接）

## 6. 配额管理体系

> **2026-04-25 收窄**：MVP Tenant 级配额仅保留**存储容量（storage_gb）**一个维度。原 §6.2 规划的资源数量（knowledge_space / knowledge_file / workflow / assistant / tool / channel / channel_subscribe / dashboard）、Token 用量（model_tokens_monthly）、用户数（user_count）等多维度配额全部推到 v2.6（见附录 G.1）。`tenant.quota_config` JSON 字段保留，未来扩展不破坏 schema。
>
> 角色级配额（`role.quota_config`，原 PRD §2.5）作用域不变，与本章 Tenant 级配额是独立机制。

### 6.1 两级配额层次（仅存储维度）

```
① 叶子 Tenant 存储配额（tenant[leaf].quota_config.storage_gb）
   用户所属叶子 Tenant 的本地存储上限
     ↓ 受约束于
② Root 存储硬盖（仅当 leaf 是 Child 时生效）
   Root 存储配额作为整个集团的总上限
     ↓ 与原 PRD §2.5 角色级配额并列
③ 角色级配额（role.quota_config）[原 PRD §2.5]
   按资源类型限制单用户创建数（独立机制，不受本章收窄影响）
     ↓ 最终
用户有效存储配额 = min(
    leaf_tenant_storage_remaining,
    root_tenant_storage_remaining (当 leaf != root)
)
```

**示例**：

- Root Tenant 存储配额 1TB
- 子公司 A Child Tenant 存储配额 100GB
- 某 Child 用户可用存储 = min(Root 剩余, 子公司 A 剩余)

### 6.2 Tenant 配额字段

`tenant.quota_config` JSON 结构（v2.5 MVP）：

```json
{
  "storage_gb": 100
}
```

**语义**：

- 配置在每个 Tenant（Root 或 Child）上，独立生效
- `storage_gb`：该 Tenant 累计 MinIO 存储上限（GB），`-1` 或字段缺失表示不限
- 其他字段（资源数 / Token / 用户数等）**v2.5 不读不写**；前端只暴露 `storage_gb` 一个 input

**v2.6+ 扩展计划**（见附录 G.1）：`knowledge_space / knowledge_file / workflow / assistant / tool / channel / channel_subscribe / model_tokens_monthly / user_count / dashboard` 等字段按需引入；JSON 字段无需 schema 迁移。

### 6.3 配额检查逻辑

用户上传文件 / 新建知识库等占用存储的操作，按以下顺序检查：

1. **两级 Tenant 存储检查**：
   - 叶子 Tenant 存储用量 < 叶子 Tenant `storage_gb`
   - 若叶子是 Child，再检查 Root 存储硬盖：(Root 自身存储用量 + Σ 所有 Child 存储用量) < Root `storage_gb`
   - 任一不满足则拒绝（HTTP 403 + 错误码"配额耗尽"，详见 §6.4）
2. **角色级检查**（沿用原 PRD §2.5，与本章并列）：用户已创建资源数 < 角色配额；本章不重复定义

**Root 存储用量语义**：

- **Root 存储用量 = Root 自身资源占用 + Σ 所有 Child 存储用量**
- 即 Root 存储配额作为**集团硬盖**，所有 Child 存储总和加上 Root 自身不能超 Root 上限
- 集团总量有上限，Child 之间在配额内竞争

**集团共享资源的存储归属**：

- Root 创建的共享资源（`tenant_id = root_id`）**计入 Root 自身存储**（不重复计入 Child，否则 Child 会因为 Root 共享而被动扣配额）
- 共享资源的**存储空间**仅算 Root 一次，即使多个 Child 可读
- 实现：`get_tenant_storage_usage` 使用 `strict_tenant_filter()` 精确匹配，避免 IN 列表叠加

**Child 用户消费 Root 共享资源的 Token / 对话记录**：本期不计入任何配额（model_tokens_monthly 推到 v2.6）。审计层仍按 INV-T13"衍生数据归用户叶子 Tenant"打 tenant_id 标签，便于未来按字段开启计量。

**并发说明**：配额检查不使用分布式锁，允许轻微超额，超额部分在下次写入时阻断。

> 具体实现见技术方案文档。

### 6.4 存储配额超限行为 `[合入自多租户管理 PRD]`

当叶子 Tenant 的存储用量达到 **100%** 时：

- 该叶子 Tenant 内所有成员调用"上传文件""新建知识库"等占用存储接口时，后端**阻断拦截**
- 前端提示："当前 Tenant 存储配额已耗尽（XX/XX GB），请联系全局超管进行配额扩容。"
- 已有资源的查看、编辑、删除不受影响
- 删除文件可释放配额，释放后恢复上传

**Root 硬盖触发**：若 Root 用量（含所有 Child 累计）达到 Root 配额，所有 Child 的创建类操作也被阻断（Root 配额已满，整个集团无新增空间）。

## 7. 集团共享资源与模板下发（P2）

### 7.1 LLM 模型服务（Root 共享 + Child 扩展）

> 2026-04-19 重写。原版"tenant_id=0 系统级 / Root 模型 / Child 模型"三套并列语义中权限归属不清、Root 与 system 层重复；现统一到"两类来源 + 共享分发"模型，并与 §7.2 集团共享资源机制同构。

#### 7.1.1 两类模型来源

| 来源 | tenant_id | 注册者 | 默认可见性 |
|------|-----------|--------|------------|
| **Root 模型** | `1`（Root） | 全局超管 | 默认随 `Tenant.share_default_to_children=true` 自动共享给所有 Child（通过 OpenFGA `{llm_server}#viewer → tenant:{root}#shared_to#member` 元组） |
| **Child 模型** | `{child_id}` | Child Admin（决策 3：完全自主） | 仅本 Child 可见；不向 Root 或其他 Child 传播 |

**原 `tenant_id=0` 系统级语义废除**：系统级模型通过 Root Tenant 的默认共享模式承载（tenant_id=1 + shared=true），避免两套机制并存。

#### 7.1.2 可见性矩阵

| 用户身份 | 可见 LLM Server / Model 集合 |
|----------|-------------------------------|
| Root 用户 / 全局超管（未设 admin-scope） | Root 所有模型（含未共享的 Root 专用） |
| Child 用户 / Child Admin | 本 Child 模型 + Root 已共享模型 |
| 全局超管（已设 admin-scope = Child X） | Child X 模型 + Root 已共享模型（管理视角内切换，见 §5.1.5） |

调用时的 tenant 校验在 `LLMDao.aget_model_by_id(model_id)` 统一入口实现；若可见集合不含目标 model，抛 `ModelNotAccessibleError`（错误码 19802），前端提示"模型不可用，请联系管理员"。

#### 7.1.3 Root 创建模型的共享开关

- UI 勾选框"**仅 Root 使用，不共享给子公司**"（默认**未勾**=共享）
- 实现方式：复用 §7.2 `tenant#shared_to#member` FGA 机制；**不新增 `is_shared_to_children` 字段到 llm_server/llm_model 表**
- 勾选"仅 Root 使用" → 不写 `{llm_server}#viewer → tenant:{root}#shared_to#member` 元组 → Child 不可见

#### 7.1.4 Root 共享模型对 Child 的只读约束

- Child Admin 在本 Child 的 LLM 管理页看到 Root 共享模型时，UI 显示"**Root 共享（只读）**" Badge，编辑/删除按钮 disabled
- 后端 `PUT /api/v1/llm/{id}` / `DELETE /api/v1/llm/{id}`：若资源 `tenant_id = root_id` 且操作者为 Child Admin → **403 + 错误码 19801 `llm_model_shared_readonly`**
- Child 无法"禁用"已共享的 Root 模型（决策 2 明确保持 Root 统一管控）

#### 7.1.5 调用量归属（与 INV-T13 对齐）

> 2026-04-25 收窄：v2.5 MVP **不做 token 配额阻断**（model_tokens_monthly 推到 v2.6，见附录 G.1）。本节仅约定 `chat_message.tenant_id` 打标规则，便于未来按字段开启计量。

| 场景 | `chat_message.tenant_id`（即归属打标） | v2.5 是否阻断 |
|------|--------------------------|----------|
| Root 用户调用 Root 模型 | Root | ❌ 不阻断（仅打标） |
| Child 用户调用 Root 共享模型 | **Child 叶子**（不计入 Root） | ❌ 不阻断 |
| Child 用户调用本 Child 模型 | Child 叶子 | ❌ 不阻断 |

衍生数据归属规则遵循 INV-T13：**共享资源衍生数据跟用户叶子 Tenant 走，资源本身归 Root**。打标语义保持，未来 v2.6 引入 `model_tokens_monthly` 时无需改造数据流；Root 存储硬盖（INV-T9）按 §6.3 存储维度生效，与 token 维度独立。

#### 7.1.6 单→多租户升级的存量处理（决策 4）

- 升级前（v2.4 或 `multi_tenant.enabled=false`）的所有存量 `llm_server` / `llm_model` 行：`tenant_id` 全部回填为 `1`（Root）
- 随 `Tenant.share_default_to_children=true` 默认值，所有存量模型**自动对未来挂载的 Child 可见**
- 知识库 / 工作流 / 助手对模型 id 的外键引用（`knowledge.model_id`、`workflow.model_id` 等）**不改动**——零成本升级
- 如果客户希望某存量模型"仅 Root 使用"，由全局超管在升级后的 UI 手工关闭该模型的共享开关（§7.1.3）
- 挂载 Child 弹窗提供 review 步骤预览将要自动分发的模型列表 + 可选"不自动分发"选项，降低敏感模型意外泄露风险（见 §5.2.1）

#### 7.1.7 新增错误码

| 错误码 | 含义 | HTTP 状态 | 模块 |
|--------|------|-----------|------|
| `19701` | `admin_scope_forbidden` — 非全局超管调用 admin-scope API（或调用者无切换视图资格） | 403 | 197 admin_scope |
| `19801` | `llm_model_shared_readonly` — Child Admin 尝试修改 Root 共享模型 | 403 | 198 llm_tenant |
| `19802` | `llm_model_not_accessible` — 目标 model 不在当前可见集合 | 404 | 198 llm_tenant |

模块编码 `197 = admin_scope`、`198 = llm_tenant` 由 `features/v2.5.1/release-contract.md` §已分配模块编码表管理。

### 7.2 集团共享资源（新增机制）

**场景**：集团总部创建统一的知识库、频道、应用模板，希望各子公司可读。

**实现机制**：OpenFGA `shared_to` 关系 + 业务层开关。

1. 全局超管创建资源时，UI 提供"共享给本集团子公司租户"开关（默认跟随 `share_default_to_children`）
2. 后端将资源 tenant_id 写为 Root ID
3. 写 FGA：`{resource}#viewer → tenant:{root}#shared_to#member`
4. 每新增 Child Tenant 时，自动写 `tenant:{root}#shared_to → tenant:{new_child}`，Child 成员自然命中 viewer
5. 解绑 Child Tenant 时，自动撤销 `tenant:{root}#shared_to → tenant:{child}` 元组

**Child 视角**：
- 知识空间列表自动包含 Root 共享的知识空间（§4.1 IN 列表 + FGA viewer 双重命中）
- Child 用户仅能读，不能编辑（共享默认只读）

**取消共享**：
- 全局超管在资源详情页切换"共享"开关为关闭 → 撤销 `{resource}#viewer → tenant:{root}#shared_to#member` 元组
- 资源继续存在于 Root Tenant，但 Child 不再可读

**可编辑共享**（待评估）：暂不支持"Root 资源对 Child 可编辑"，保留未来扩展空间。

### 7.3 模板下发

| 项目 | 说明 |
|------|------|
| 系统模板 | 全局超管创建的系统级模板（`tenant_id=0`） |
| Root 模板 | 全局超管在 Root 创建的模板，自动共享给所有 Child |
| 下发操作 | 全局超管选择模板 → 选择目标 Tenant（多选）→ 确认 |

| Tenant 视角 | 用户在模板列表中可看到"系统下发"或"集团下发"标签 |

### 9.1 同步架构

v2.5.0 同步源分两条主干：

```
┌─────────── 管理台配置源（OrgSyncConfig 驱动） ──────────┐
│  企业微信 corpid/secret + agent_id   ← 本期 P0           │
│  Feishu app_id/app_secret                                  │
│  Generic REST API endpoint + field_mapping                 │
│  钉钉 app_key/app_secret（stub，v2.5.2）                   │
└──────────────────────────┬─────────────────────────────────┘
                           │ Celery Beat 按 cron / 管理台手动触发
                           ↓
┌────── Provider 抽象层（OrgSyncProvider ABC） ──────────────┐
│  authenticate() / fetch_departments() / fetch_members()    │
│  / test_connection()                                        │
└──────────────────────────┬─────────────────────────────────┘
                           ↓
┌────── Reconciler 引擎（F009 已实现，无需改造） ───────────┐
│  部门：create / rename / move / archive                     │
│  成员：create / update / transfer / disable / reactivate    │
│  本地冲突：source='local' 强制覆盖改 source                │
└──────────────────────────┬─────────────────────────────────┘
                           ↓
              bisheng user / department / UserDepartment
                           ↑
                           │
┌────── Gateway 主动推送（保留，HR/LDAP/SSO 通用） ─────────┐
│  POST /api/v1/internal/sso/login-sync   ← SSO 登录实时派生 │
│  POST /api/v1/departments/sync          ← 批量 HMAC 推送   │
└────────────────────────────────────────────────────────────┘
```

**路径选择**：

- 企微、Feishu、DingTalk、Generic REST 走**上半部分**（管理台配置 → Celery 定时 → Reconciler）
- 客户自有 HR 系统 / LDAP 同步脚本 / 商业版 Gateway SSO 走**下半部分**（直接 HMAC 推送）
- 两条路径最终都通过 F011 `DepartmentChangeHandler` 通知权限/Tenant 派生

### 9.2 企业微信 Provider 对接详述（本期核心）

#### 9.2.1 管理台配置与凭证

- **入口**：系统管理 → 组织同步 → 新增配置 → Provider 选择「企业微信」
- **配置字段**（对应 `org_sync_config.auth_config` JSON）：

  | 字段 | 必填 | 说明 |
  |------|------|------|
  | `corpid` | ✅ | 企业 ID（企业微信后台「我的企业」） |
  | `corpsecret` | ✅ | 应用 Secret（建议使用「通讯录同步助手」Secret 以获得完整可见范围） |
  | `agent_id` | ✅ | 应用 AgentId（整数字符串） |
  | `allow_dept_ids` | 可选 | 限定同步的顶层部门 ID 列表，默认 `[1]`（企微根部门固定为 1） |

- **存储**：整个 `auth_config` 经 Fernet 加密入 `org_sync_config.auth_config`（F009 已有机制，使用 `settings.secret_key`）
- **脱敏**：GET/LIST 接口返回时 `corpsecret` 替换为 `****`（F009 `mask_sensitive_fields()` 新增规则：provider=wecom 时脱敏 corpsecret 字段）
- **唯一性**：F009 已有约束 `unique(tenant_id, provider, config_name)`，本期无需新增

#### 9.2.2 Access Token 管理

| 项 | 规则 |
|----|------|
| 获取接口 | `GET https://qyapi.weixin.qq.com/cgi-bin/gettoken?corpid=X&corpsecret=Y` |
| Redis Key | `org_sync:wecom:token:{config_id}` |
| TTL | 响应 `expires_in` − 300s 缓冲（企微默认 7200s → 缓存 6900s） |
| 并发控制 | Redis 分布式锁 `org_sync:wecom:token_lock:{config_id}`，TTL 30s，等待锁时 sleep 200ms 再读缓存 |
| 失效重试 | 业务 API 响应 `errcode=42001 / 40014` → 删除缓存 token → 重新 gettoken → 重试原调用**一次** |
| 禁止项 | access_token **严禁**返回前端 / 出现在 `test_connection` 响应 / 写入 `error_details` |

#### 9.2.3 企业微信接口选型与字段映射

**部门拉取**：

- 接口：`GET /cgi-bin/department/list?access_token=X&id={root_id}`
- 一次调用返回子树全部部门（含 root 本身），无需递归翻页
- 若配置了多个 `allow_dept_ids`，对每个 root 分别调用后按 `id` 去重
- 映射 → `RemoteDepartmentDTO`（字段以 `src/backend/bisheng/org_sync/domain/schemas/remote_dto.py` 为准）：

  | 企微字段 | `RemoteDepartmentDTO` 字段 |
  |----------|---------------------------|
  | `id` (int) | `external_id` (str，强制转字符串) |
  | `name` | `name` |
  | `parentid` | `parent_external_id`（若部门为配置的 root 之一，置 `None`） |
  | `order` | `sort_order` |

**成员拉取**：

- 接口：`GET /cgi-bin/user/list?access_token=X&department_id={dept_id}&fetch_child=1`
- 对每个顶层 root（或 `allow_dept_ids` 中每个）调用一次，即可拉到整棵子树成员（含多部门归属），无需按部门递归
- 映射 → `RemoteMemberDTO`：

  | 企微字段 | `RemoteMemberDTO` 字段 |
  |----------|------------------------|
  | `userid` | `external_id` |
  | `name` | `name` |
  | `email` / `biz_mail`（任一非空优先 `email`） | `email` |
  | `mobile` | `phone` |
  | `main_department`（v2020+ 返回） | `primary_dept_external_id`（转字符串）；fallback `department[0]` |
  | `department`（除 main_department 外的部门） | `secondary_dept_external_ids`（转字符串列表） |
  | `status`：1 → `active`；2 / 4 / 5 → `disabled`（其中 4=未激活、5=退出） | `status`（枚举仅 `active` / `disabled`） |

> **关键注解**：`main_department` 字段从 2020-11 起返回。若字段缺失，Provider 必须 fallback `department[0]` 以保持兼容；`status` 未在枚举内的值按 `disabled` 处理，不触发同步失败。

#### 9.2.4 同步策略（对接 F009 Reconciler）

| 策略 | 本期范围 | 说明 |
|------|---------|------|
| 全量同步 | **必做** | 每次执行拉完整部门树 + 成员，交给 Reconciler 与本地 hash diff |
| 定时同步 | **必做** | `org_sync_config.cron_expression` 控制；推荐客户默认 `0 */6 * * *`（每 6h） |
| 管理台手动触发 | **必做** | 复用 `POST /org-sync/configs/{id}/execute` |
| Cursor 增量 | **不做** | 企微 Contact API 无原生 cursor 语义，保留"全量 + Reconciler diff"策略保持简单 |
| `change_contact` 事件回调 | **不做** | 详见 §9.2.8，v2.5.2 独立 Feature 落地 |

**为何全量够用**：F009 Reconciler 已实现"无变更则 noop、增量 upsert/archive"。10 万人量级下瓶颈在企微 API QPS（见 §9.2.5），不在 DB 写入；6h 周期可以接受变更最多 6h 延迟，客户现有诉求无秒级场景。

#### 9.2.5 限流与重试

- 企微通讯录读 API QPS 上限约 **60 / min**（单 agent_id 维度，包含 department/list、user/list、gettoken）
- Provider 内部控制：`asyncio.Semaphore(5)` 限并发，每次调用间隔 ≥ 100ms
- 限流错误码处理：`errcode=45009 / 45033 / 45011` → 指数退避 `[1s, 2s, 4s, 8s]`，最多重试 4 次；仍失败则该同步 Log 标记 `partial`，错误写入 `org_sync_log.error_details`
- 连续 3 次同步全失败 → `org_sync_config.status` 自动改为 `disabled`，写入 `audit_log`（F009 AC-15 已覆盖该降级逻辑）

#### 9.2.6 字段回写与冲突

- `user.source` = `'wecom'`；`user.external_id` = 企微 `userid`（字符串，保留原大小写）
- 唯一约束：`(source, external_id)` 全局唯一（F009 migration 已建）
- **本地冲突**：
  - 同 email 的 `source='local'` 用户 → 强制覆盖合并：将 `source` 改为 `'wecom'`，写入 `external_id`（F009 设计原则，见 spec.md §7 第 7 条）
  - 同 email 的 `source='feishu'` 用户（跨第三方冲突）→ 跳过该成员，`error_details` 记录冲突详情，不自动合并
- **部门归档级联**：企微删除的部门 → Reconciler 走 archive → 若现存用户主部门指向该部门 → 触发 F011 `DepartmentDeletionHandler`（孤儿 Tenant 告警链路）

#### 9.2.7 安全与合规

| 项 | 规则 |
|----|------|
| corpsecret 存储 | Fernet 加密（F009 已有） |
| corpsecret 展示 | 所有 GET/LIST 接口脱敏 `****` |
| access_token | 仅后端 Redis 缓存，永不出接口、永不写入日志/审计 |
| 手机号/邮箱 | `org_sync_log.error_details` 与站点日志走 F009 `mask_sensitive_fields()` 脱敏 |
| 审计日志 action | `org_sync.config.created` / `.updated` / `.deleted` / `.executed` / `.failed` / `.disabled_auto`（F009 已落实，企微 Provider 复用，不新增 action key） |
| 租户隔离 | `org_sync_config.tenant_id` + SQLAlchemy event 注入（与其他业务表一致） |

#### 9.2.8 本期明确不做（Out of Scope）

- **`change_contact` 事件回调**（URL echostr 校验、消息体 AES-CBC 解密、事件类型分派）→ v2.5.2 独立 Feature，届时设计回调端点、AES key 配置、事件 → Reconciler 的最小粒度更新
- **企微 OAuth SSO 登录**（扫码登录、企微内免登）→ 由商业版 Gateway 独立对接；bisheng 侧只需企微同步先行把 `source='wecom'` 用户落库，SSO 回来即可命中
- **外部联系人**（客户通讯录）→ 不在 bisheng 同步范围
- **多 agent_id 同时同步同一 tenant** → v2.5.2+；本期一个 tenant 下允许多 `org_sync_config`，但不做跨配置的数据合并策略

### 9.3 Gateway 主动推送接口（通用，保留）

v2.5.0 保留两个 Gateway/HR 系统驱动的推送接口，**不专属任何一个 Provider**，作为 `OrgSyncConfig` 之外的备选：

#### 9.3.1 SSO 登录实时同步

**端点**：`POST /api/v1/internal/sso/login-sync`（内部接口，HMAC 签名保护）

**Payload**：见 §5.2.3。

**后端处理**：

1. Upsert User（按 `source + external_id` 匹配）
2. Upsert UserDepartment（主/兼职部门）
3. 调用 `UserTenantSyncService.sync_user(user_id)` 派生叶子 Tenant
4. 可选处理 `tenant_mapping` 字段（见 §5.2.3）
5. 返回含叶子 tenant_id 的 JWT

#### 9.3.2 批量部门推送

**端点**：`POST /api/v1/departments/sync`（HMAC 签名保护）

**适用方**：客户自有 HR 系统定时推送、LDAP 同步脚本、商业版 Gateway 主动同步。**企微同步不走此路径**，由 Celery + OrgSyncConfig 触发。

**Payload**（增量协议）：

```json
{
  "upsert": [
    {"external_id": "D001", "name": "研发部", "parent_external_id": "D000", "sort": 1, "ts": 1713400000}
  ],
  "remove": ["D099"],
  "source_ts": 1713400000
}
```

**后端处理**：

- 批量 upsert/archive 部门
- 不触碰 `is_tenant_root` 字段（同步鲁棒性，见 §5.2.5）
- 批处理时延迟 FGA 写入，一次性 batch 提交
- 写入 `org_sync_log` 表记录同步历史（trigger_type=`gateway_push`）

### 9.4 Celery 定时调度

**任务**：`check_org_sync_schedules`（F009 已实现；替换 v2.5.0 之前规划的 `sync_organization_from_ldap`）

**流程**：

1. Celery Beat 每 60s 触发一次，扫描 `org_sync_config` 中 `status='active'` 且 `schedule_type='cron'` 的配置
2. 对每条到期配置 → 派发 `execute_org_sync(config_id, trigger_type='scheduled')` 到 `knowledge_celery` 队列
3. Worker 侧：解密 `auth_config` → 实例化 Provider（企微/飞书/GenericAPI/钉钉） → 调 `fetch_departments` + `fetch_members` → Reconciler diff → 写 `org_sync_log`
4. 主部门变更用户自动触发 `UserTenantSyncService.sync_user` 与 F011 `DepartmentChangeHandler`

**企微推荐 cron**：`0 */6 * * *`（每 6h 触发一次）。客户可按规模调整。

### 9.5 实时 vs 定时校对的冲突处理（2026-04-20 新增）

Gateway 实时同步（§9.3.1）和 Celery 定时校对（§9.4）可能并发同一部门的不同操作（例如实时收到"D001 改名"同时 Celery 发现"D001 已删除"）。**冲突处理规则**：

1. 每条同步消息携带 `ts` 字段（源系统时间戳；企微/飞书 fetch 时取本地 `time.time()`，因 API 不返回变更 ts）
2. bisheng 为每个 `external_id` 记录 `last_sync_ts`
3. **以 ts 最大为准**：如果 `incoming_ts < last_sync_ts`，跳过该消息并写入 `org_sync_log`（level=warn）
4. 同一 ts 下 upsert 与 remove 冲突（极罕见）**以 remove 为准**（从严处理，避免保留幽灵部门）
5. 冲突记录每周汇总发送全局超管站内消息

### 9.6 Redis 缓存策略

针对 10 万人场景的性能优化：

| Key | 内容 | TTL | 失效时机 |
|-----|------|-----|---------|
| `dept:tree:root` | 整棵部门树 JSON | 1h | 部门变更 |
| `dept:user:{uid}:primary` | 用户主部门 ID | 24h | 主部门变更 |
| `dept:tenant:{tid}:subtree_ids` | Tenant 下所有部门 ID | 1h | 部门变更 / 挂载变更 |
| `user:{uid}:leaf_tenant` | 用户叶子 Tenant | 1h | 主部门变更 / UserTenant 变更 |
| `org_sync:wecom:token:{config_id}` | 企微 access_token | ~6900s | 调用收到 42001/40014 主动清除 |
| `org_sync:wecom:token_lock:{config_id}` | 企微 token 获取分布式锁 | 30s | 获取 token 完成后立即释放 |

**失效机制**：部门/用户相关 key 由 `DepartmentChangeHandler` / `UserTenantSyncService` 扩充；企微 token key 由 `WeComProvider._invalidate_token()` 维护。

### 9.7 性能基线要求

> 2026-04-18 PRD Review 决策：**仅审资"已有客户"级别**，10 万人量级压测留客户落地前冲刺；Celery 定时校对压力评估标为 TBD，纳入专项压测。

**MVP 目标量级**：当前客户规模（约 10 万人 / 部门若干层级）

| 操作 | 目标耗时 |
|------|---------|
| 部门树查询 | < 200ms（走 Redis 缓存 < 50ms） |
| 子树 LIKE path 查询 | < 100ms |
| 用户叶子 Tenant 派生 | < 50ms（走 JWT 或 Redis） |
| 部门移动（子树 < 1 万节点） | < 2s |
| 部门移动（子树 > 1 万节点） | 异步后台任务 |
| 企微全量同步（10 万人，1 次 `department/list` + N 次 `user/list`，N = 顶层部门数） | < 10 min（受企微 60/min QPS 限制）；超过 15 min 触发告警 |
| Celery 定时校对（多 Provider 合计全量对照） | TBD — 需在 v2.5.1 发布前 2 周专项压测给出 DB/Redis 消耗基线 |

**更大规模需客户落地前执行专项压测**：当客户实际规模超过 MVP 目标量级的 2 倍（如 10 万+ 人/部门）时，要求执行：

- 10 万人用户登录并发压测（Gateway 实时同步 + 叶子派生）
- 10 万部门全量校对对 MySQL/Redis/OpenFGA 的压力评估
- Redis 缓存击穿场景的兜底验证
- 企微 QPS 受限下的全量同步峰值耗时评估

**SSO 端换 HR 系统的迁移**（文档化指引，不做自动化）：

- SSO 端 `department.external_id` 不保证稳定；换 HR 系统后挂载关系可能断裂
- 提供 `POST /api/v1/internal/departments/relink` 接口：按部门 path 回落匹配，重建 `external_id ↔ internal_id` 映射
- 不做自动迁移；客户需在换型窗口期内手工触发

### 9.8 安全与合规汇总

集中陈述本章的安全准线（与 §9.2.7 的企微专项不重复，此处聚焦跨 Provider 通则）：

- **凭证生命周期**：所有 Provider 的凭证（corpsecret / app_secret / api_key / HMAC secret）仅在管理台录入/更新时明文进入后端（HTTPS 传输），入库即 Fernet 加密，读取接口响应永远脱敏
- **Token 隔离**：各 Provider 的 access_token 按 `config_id` 独立缓存，Redis key 不跨 tenant、不跨 Provider
- **审计必打点**：同步执行结束（无论成败）必写 `org_sync_log`（保留 180 天）+ 关键动作写 `audit_log`（tenant.*、org_sync.*）
- **日志脱敏**：`error_details` / site log 中手机号、邮箱、userid、external_id 按 F009 `mask_sensitive_fields()` 处理，但**保留** `external_id` 的首尾字符以便排查
- **合规声明**：bisheng 仅拉取客户在第三方平台「应用可见范围 / API 授权范围」中授予的员工数据；客户合规边界由客户在其平台后台自主配置，bisheng 不做二次脱敏以外的合规判断
