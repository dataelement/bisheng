# 用户组「特殊业务规则」排查报告

> 扫描分支：`feat/2.6.0-beta3`　日期：2026-06-01
> 新模型前提：①用户不强绑用户组　②角色不属于用户组　③资源不与用户组关联　④历史关联数据不清理。
> 排查目标：找出用户组上挂着的**特殊业务规则**（不能删、不能编辑、不能删成员、按固定 ID 特判等）。

---

## 结论速览

**几乎没有"硬保护"规则残留 —— 不能删除 / 不能编辑 / 不能删成员的限制都已被移除（只剩死代码错误码）。** 当前用户组的删除/编辑/成员操作只受**权限**约束（超管 / 创建者 / 部门管理员），没有任何"默认组特殊对待"。

真正还活着、且与新模型冲突的，是三处**耦合副作用**（删组时迁移资源、删组时删角色、SSO/同步建号兜底进默认组）和**按组名隐藏**的展示过滤。详见下表。

---

## A. 特殊保护规则 —— 排查结果：均已失效 ✅

| 规则 | 错误码 | 状态 | 证据 |
|------|--------|------|------|
| 不能删除/重命名默认组 | `UserGroupDefaultProtectedError` (23002) | **死代码，全仓 0 引用** | 定义在 [errcode/user_group.py:15](src/backend/bisheng/common/errcode/user_group.py#L15)，无任何 `raise` |
| 不能删除有成员的组 | `UserGroupHasMembersError` (23003) | **死代码，全仓 0 引用** | 新版 delete 注释明确"可有成员，删除后成员关系一并清理"（[user_group_service.py:155](src/backend/bisheng/user_group/domain/services/user_group_service.py#L155)） |

**两条 delete 路径都不对默认组特判**：
- 新版 [user_group_service.adelete_group:330](src/backend/bisheng/user_group/domain/services/user_group_service.py#L330)：仅 `_ensure_delete_group`（超管/创建者），无 `if group_id == DefaultGroup`。
- 旧版 [role_group_service.delete_group:144](src/backend/bisheng/api/services/role_group_service.py#L144)：`get → delete → hook`，同样无默认组保护。
- 编辑：[aupdate_group:301](src/backend/bisheng/user_group/domain/services/user_group_service.py#L301) 仅查重名 + 权限，无默认组保护。

> 也未发现任何其他硬编码特殊组 ID（如 group 1 / admin group）的业务特判。

---

## B. 唯一"功能性禁用"规则（与默认组无关，注意别误读）

| 位置 | 行为 |
|------|------|
| [aset_admins:477-485](src/backend/bisheng/user_group/domain/services/user_group_service.py#L477-L485) | 设置"组管理员"接口**恒抛** `UserGroupNoSeparateAdminsError` |

即新模型下"用户组没有独立的组管理员"这一产品决策已落地（不是针对某个组，而是整个特性关闭）。排查时不要把它当成默认组规则。

---

## C. 与新模型冲突的耦合副作用 —— 需重点评估去留 ⚠️

新模型说"资源不关联组、角色不属于组、用户不强绑组"，但下面三处旧耦合逻辑仍在运行：

### C1. 删组时把资源迁移到"兜底组" —— 资源↔组解耦后应废弃
两处重复实现，删组时把"仅属于该组"的 `group_resource` 行迁移到**第一个其它组**：
- 新版 [_sync_user_group_delete_side_effects:31-69](src/backend/bisheng/user_group/domain/services/user_group_service.py#L31-L69)
- 旧版 [delete_group_hook:159-180](src/backend/bisheng/api/services/role_group_service.py#L159-L180)

资源不再与组关联后，这段迁移逻辑无意义；保留会在删组时产生无效的 `group_resource` 搬移 + 网关 Redis `delete_group` 通知。

### C2. 删组时连带删角色 —— 角色↔组解耦后应废弃
两条 delete 路径都调用 `RoleDao.delete_role_by_group_id(group_id)`（[新版:345-346](src/backend/bisheng/user_group/domain/services/user_group_service.py#L345)、[旧版:182-187](src/backend/bisheng/api/services/role_group_service.py#L182)）。
`role.group_id` 字段本身已标注 **`# deprecated: use department_id`**（[role.py:33](src/backend/bisheng/database/models/role.py#L33)）。角色脱离用户组后，"删组级联删角色"会误删本不该归属该组的角色。

### C3. SSO / 组织同步建号兜底进默认组 —— 用户不强绑组后应废弃
- 旧版 SSO：[user/api/user.py:95](src/backend/bisheng/user/api/user.py#L95) `member_group_ids=[user.default_groupid or DefaultGroup]`
- SSO 默认授权 DAO：[user/domain/models/user.py:339](src/backend/bisheng/user/domain/models/user.py#L339) `group_id = default_groupid or DefaultGroup`
- 组织同步：[org_sync_service._ensure_default_user_group_membership:522](src/backend/bisheng/org_sync/domain/services/org_sync_service.py#L522)（在 [:480](src/backend/bisheng/org_sync/domain/services/org_sync_service.py#L480) 调用）

注意：普通注册（[add_user_and_default_role](src/backend/bisheng/user/domain/models/user.py#L316)）**只加角色、不加组**，已符合新模型。只有 SSO/同步建号还在兜底塞默认组。

## D. 按组名隐藏默认组 —— 纯展示过滤，低风险

`LEGACY_HIDDEN_USER_GROUP_NAMES = ('Default user group', '默认用户组')`（[group.py:14](src/backend/bisheng/database/models/group.py#L14)）在 6 处把历史默认组从列表中过滤掉：

| 位置 | 场景 |
|------|------|
| [group.py:168](src/backend/bisheng/database/models/group.py#L168) / [:210](src/backend/bisheng/database/models/group.py#L210) | 全量 / 可见组列表 |
| [user_group.py:510](src/backend/bisheng/database/models/user_group.py#L510) | 成员组信息批量 |
| [department_service.py:1141](src/backend/bisheng/department/domain/services/department_service.py#L1141) | 组织成员列表富集 |
| [resource_permission.py:789](src/backend/bisheng/permission/api/endpoints/resource_permission.py#L789) | 资源授权可选组列表 |
| [UserGroup.tsx:59,70](src/frontend/platform/src/pages/SystemPage/components/UserGroup.tsx#L59-L70) | 前端二次过滤 |

历史关联数据不清理的前提下，**这层过滤要保留**（否则历史默认组会重新暴露在所有列表里）。属于"维持现状"项，不是要移除的特殊规则。

---

## 排查结论与建议

1. **"不能删/不能编辑/不能删成员"——确认无活跃规则**，可直接删除死代码错误码 `UserGroupDefaultProtectedError`(23002) 与 `UserGroupHasMembersError`(23003)，零风险。
2. **C1/C2/C3 三处耦合副作用**是新模型下真正需要拍板的点：建议随"资源/角色/用户与组解耦"一并移除或改造，否则会在删组、SSO 建号时产生无意义的副作用与潜在误删。
3. **D 组名隐藏**在不清历史数据的前提下需保留。
4. 新旧两套用户组服务（`user_group/` F003 与 `api/services/role_group_service.py`）逻辑高度重复，建议确认旧版 `role_group_service` 是否仍有端点在用（`api/v1/usergroup.py` 仍在调用），统一收口可避免在两处各改一遍。
