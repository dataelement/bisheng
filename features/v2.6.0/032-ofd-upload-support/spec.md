# Feature: 全平台 OFD 文件上传支持

> **本文档定位 — 纯 What（需求口径，不随代码漂移）**
> spec 只回答 **做什么 / 验收标准 / 不做什么**；所有 How（决策、数据流、字段、API、Service、前端、文件清单）一律在 [design.md](./design.md) 与 [tasks.md](./tasks.md)。

**关联 PRD**: 无（内部能力补齐，无独立 PRD）
**优先级**: P1
**所属版本**: v2.6.0
**依赖**: 无（复用现有 PDF 解析与预览链路，不新增领域对象）

> **范围边界**
> - **本次纳入**：
>   - 四个上传入口接受 `.ofd`：知识库文件上传、知识空间文件上传、工作流文件上传、日常对话模式文件上传
>   - OFD 作为源文件存储；解析时转换为 PDF 并复用既有 PDF 解析链路（分块 / 入库 / 检索 / 引用溯源）
>   - OFD 的预览为转换后的 PDF
>   - 默认支持，无需配置开关（与 pdf / docx 同级）
> - **本次明确排除**：
>   - Linsight 灵思入口、Report 模板上传、QA 导入等其它零散入口（本期不动，避免 scope 膨胀）
>   - 不为 OFD 另存一份独立的 PDF 源文件（原始文件仍是 `.ofd`，PDF 仅作预览）
>   - 不支持反向生成 OFD（PDF/其它格式 → OFD）
>   - 不做 OFD 电子签章 / 印章的法律有效性校验（仅做版式渲染转换）
>   - 不保证复杂 OFD（矢量图、印章、签章）的像素级保真（尽力渲染；失真不视为功能缺陷，但损坏文件须明确失败）

---

## 1. 用户故事

作为 **使用国产版式文档（政务 / 金融 / 央国企场景常见）的平台用户**，
我希望 **能像上传 PDF 一样上传 `.ofd` 文件，并被正常解析、预览、检索**，
以便 **把 OFD 文档纳入知识库与对话问答，无需先手动转换成 PDF**。

---

## 2. 验收标准

> AC-ID 在本特性内唯一。错误码号在此仅作为「可观测的对外行为」引用。

- **AC-01** — WHEN 用户在「知识库 / 知识空间 / 工作流 / 日常对话」任一入口上传 `.ofd` 文件, THE SYSTEM SHALL 接受该文件并以 `.ofd` 作为源文件存储。
- **AC-02** — WHEN 一个 OFD 文件进入解析流程, THE SYSTEM SHALL 将其转换为 PDF，并复用既有 PDF 解析链路完成分块与入库。
- **AC-03** — WHEN 用户预览一个 OFD 文件, THE SYSTEM SHALL 展示其转换后的 PDF。
- **AC-04** — IF OFD 文件损坏或格式非法导致转换失败, THEN THE SYSTEM SHALL 将该文件标记为解析失败，并返回错误码 10917（前端提示「OFD 文件损坏或格式非法」），不得伪装成功。
- **AC-05** — THE SYSTEM SHALL 在 OFD 文件的检索结果与引用溯源中，正确呈现其来源文件信息，与其它已支持格式表现一致。
- **AC-06** — THE SYSTEM SHALL 默认支持 OFD 上传，不依赖任何配置开关（行为与 pdf / docx 等静态支持格式一致）。

---

## 3. 边界情况

- 当上传的文件后缀为 `.ofd` 但内容并非合法 OFD（后缀伪造）时，系统应转换失败并按 AC-04 返回错误码 10917。
- 当 OFD 内含图片 / 印章 / 矢量图时，系统应尽力完成渲染转换；渲染失真不视为失败，但无法解析的损坏文件应明确失败（AC-04）。
- 当 OFD 文件超大 / 多页时，系统应沿用既有大文件解析的超时与失败语义，不得卡死整条解析队列（具体策略见 design.md §5 / §7）。
- 当同名 / 重复上传 OFD 时，沿用各入口现有的重复文件处理逻辑，不做特殊化。
