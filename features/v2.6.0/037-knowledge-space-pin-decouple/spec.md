# Spec: 知识空间置顶与成员关系解耦

**版本**: v2.6.0
**轨道**: Small feature（SDD 轻级）
**最后更新**: 2026-06-24

---

## 1. 背景与问题

知识空间置顶接口 `POST /api/v1/knowledge/space/{space_id}/set-pin` 返回成功，但实际未置顶。

**根因**：置顶状态 `is_pinned` 寄生在成员表 `space_channel_member` 的成员行上，置顶操作仅做 UPDATE 且无条件 `return True`。对于**没有 ACTIVE 成员行**的用户（通过 ReBAC 授权 / 部门授权继承访问、或老数据），UPDATE 命中 0 行却仍返回成功 → "成功但没置顶"。

**不能简单"补插成员行"**：成员行同时承载成员计数、成员管理列表、订阅/审批等语义（被 `async_get_members_by_space` 等 7+ 处消费），且普通部门成员（viewer）本就没有成员行、换部门后无任何 membership 同步钩子回收 → 补插会产生无人清理的孤儿行，污染计数与成员管理。

---

## 2. 目标

把"置顶"建模为**纯用户个人偏好**，与"成员关系"彻底解耦。

## 3. 非目标

- 不改频道（channel）的置顶逻辑（独立模块，独立 endpoint，本次不动）。
- 不改 ReBAC / 部门授权的访问控制逻辑。
- 不新增/修改对外 HTTP 契约（`set-pin` 路径与请求体不变，仅修复行为 + 补权限校验）。

---

## 4. 验收标准（AC）

| # | 场景 | 期望 |
|---|---|---|
| AC1 | 有 ACTIVE 成员行的用户置顶 | 置顶成功，列表中该空间排到置顶区，`is_pinned=true` |
| AC2 | 仅通过 ReBAC / 部门授权访问（无成员行）的用户置顶 | 置顶成功并真实生效，**不**产生 `space_channel_member` 行，成员数不变 |
| AC3 | 取消置顶 | 置顶记录删除，列表回到普通区 |
| AC4 | 对无 `view_space` 权限的空间调用 set-pin | 返回业务错误（权限拒绝 / 空间不存在），不写任何数据 |
| AC5 | 用户置顶后失去访问（如换部门） | 该空间不再出现在其任何列表（读取按 `view_space` 重新校验）；遗留置顶记录惰性失效、不影响成员数与成员管理 |
| AC6 | 历史已置顶数据 | 升级后仍保持置顶（迁移脚本回填） |
| AC7 | 重复置顶同一空间 | 幂等，不报错、不产生重复记录 |

---

## 5. 边界与兼容

- **双 DB**：新表 + 迁移须 MySQL / 达梦(DM) 双兼容（遵循 `docs/constitution.md` C2）。
- **幂等迁移**：建表迁移须带存在性守卫（参考 `v2_5_1_f021` 写法）。
