## 一、功能 1：E+ 站内信同步

### 1.1 背景与目标

BiSheng 工作台已有「消息提醒」站内信模块（`src/backend/bisheng/message/domain/services/message_service.py`），承载频道订阅、知识空间加入、部门知识空间上传审核三类业务的 12 种 `action_code`。

当前痛点：**站内信只在 BiSheng 工作台内可见，用户不主动登录则查收不及时。**

目标：把「频道订阅」「知识空间加入」两类业务的 6 个目标 `action_code` 站内信 **异步转发** 一份 textcard 类型消息到中粮 E+ 消息中心，让接收人在 E+ 端看到提醒并跳转回 BiSheng 处理。

**重要约束**：本期是冗余通道，不替代站内信；E+ 投递失败必须不阻塞站内信主流程。

### 1.2 同步功能开关（独立小节，对应需求"E+ 站内信同步功能开关"）

通过 `config.yaml` 配置块的**存在 + enabled**作为开关；不在管理后台暴露 UI 开关（与需求文档"中粮集团专属配置"的 YAML 示例一致）。

```yaml
# 中粮集团专属配置 —— 不配置此块即代表关闭
in_app_message_forwarding:
  cofco:
    enabled: true                                  # 开关，显式 true 才生效
    api_base: "http://10.28.64.30:8070/qwmsg-ui"   # 注意 http 内网
    app_id: "<由中粮消息中心提供>"
    secret: "<由中粮消息中心提供>"
    agentid: 1
    timeout_seconds: 5
    bisheng_inbox_url: "https://bisheng.cofco.com"  # 跳回 BiSheng 客户端的 base URL
    enable_duplicate_check: 0
    duplicate_check_interval: 1800
```

**判定逻辑**：
- 配置块缺失 → `enabled=false`，整链路 short-circuit
- 配置块存在但 `enabled=false` → 同上
- 配置块存在且 `enabled=true` → 进入转发流程

> 命名说明：飞书文档原 YAML 字段名 `In_app_message_forwarding`（驼峰大写）不符合 Python YAML 习惯，本设计统一为小写下划线 `in_app_message_forwarding`，向运维交付时同步更新示例。

### 1.3 转发范围（6 个 action_code）

| action_code | 类型 | 接收人 | 触发场景 |
|---|---|---|---|
| `request_channel` | approve | 频道 Creator + Admin | 用户申请订阅频道 |
| `approved_channel` | notify | 申请人 | 频道管理员同意 |
| `rejected_channel` | notify | 申请人 | 频道管理员拒绝 |
| `request_knowledge_space` | approve | 知识空间 Creator + Admin | 用户申请加入知识空间 |
| `approved_knowledge_space` | notify | 申请人 | 知识空间管理员同意 |
| `rejected_knowledge_space` | notify | 申请人 | 知识空间管理员拒绝 |

### 1.4 总体架构（A1）

```
[业务调用方]  channel_service / knowledge_space_service / *_handler
        │
        ▼  send_generic_approval / send_generic_notify
   MessageService.send_message()  ───── 持久化 InboxMessage（不变）
        │
        ▼  _maybe_forward_external(message)        ← 新增钩子（同步、轻量）
        │  · 判断开关
        │  · 判断 action_code ∈ FORWARDABLE_ACTION_CODES
        │  · 解析接收人 → user.external_id where user.source == 'cofco_eplus'
        │  · 渲染 textcard payload（含回调 URL，带 message_id）
        │  · BackgroundTasks.add_task(client.send_textcard, payload)
        ▼
   返回业务调用方（主流程已结束，转发与否不影响成败）
```

**关键约束**：
- `_maybe_forward_external` 是同步函数，只做判断/构造/排程，**不做 HTTP**
- HTTP 在 FastAPI `BackgroundTasks` 中执行，包 `try/except`，失败只 `logger.warning`，**绝不抛回主流程**
- 接收人解析失败 → 静默跳过，INFO 日志归因

### 1.6 E+ 接口契约（基于实测接口文档）

**Endpoint**：
| 环境 | 地址 |
|---|---|
| 测试 | `http://10.28.65.32:8080/qwmsg-ui/v2/message/send` |
| 正式 | `http://10.28.64.30:8070/qwmsg-ui/v2/message/send` |

**鉴权**：HTTP Header 直传 `appId` + `secret`，无 access_token 流程。

**请求体（textcard）**：
```json
{
  "touser": "UserID1|UserID2",
  "msgtype": "textcard",
  "agentid": 1,
  "textcard": {
    "title": "≤128 字节",
    "description": "<div class=\"gray\">..</div><div class=\"normal\">..</div><div class=\"highlight\">..</div>",
    "url": "必须含 http(s):// 协议头，≤2048 字节",
    "btntxt": "去查看"
  },
  "enable_duplicate_check": 0,
  "duplicate_check_interval": 1800
}
```

着色：`gray` / `normal` / `highlight` 三种 `<div class>`。

**响应**：`{ "code": "0", "data": "[\"jobid1\"]", "msg": "操作成功" }`，`code == "0"` 视为成功。

**已知错误码**：
| code | 含义 | 应对 |
|---|---|---|
| 82001 | touser 全部无效 | WARN 日志（说明 external_id 不在 E+ 名单） |
| 500011 | 应用 IP 限制 | ERROR 日志（运维白名单） |
| 其它 ≠ "0" | 通用失败 | WARN |

**安全说明**：实际是 HTTP 明文，secret 在 Header 明文传输是接口侧限制；部署时确保中粮内网链路。

### 1.7 回调 URL 与 Client 自动展开站内信弹窗

这是需求"用户点击 textcard 后跳回 BiSheng，client 自动打开站内信"的细化设计。

#### 1.7.1 URL 设计

textcard 的 `url` 字段拼接为：

```
{bisheng_inbox_url}/?open-notifications=1&message-id={message_id}
```

示例：`https://bisheng.cofco.com/?open-notifications=1&message-id=12345`

- 选择 query 参数而非新路由：现有 client 路由 (`createBrowserRouter`，`src/frontend/client/src/routes/index.tsx:88`) 顶层结构稳定，query 参数与任何根路径并存，且对已登录跳转、SSO 回跳更友好。
- `message-id` 可选：缺省时只打开站内信弹窗、不指定消息（最常见场景：用户点了通知，但 client 默认显示全部）。

#### 1.7.2 Client 侧实现要点

**触发点**：`src/frontend/client/src/layouts/UserPopMenu.tsx`（铃铛 + `NotificationsDialog` 所在层）

> **状态管理**：client app 全局态使用 **Recoil**（不是 Zustand，platform app 才是 Zustand）；本场景的 `notificationsDialogOpen` / `focusedMessageId` 仅在 UserPopMenu → NotificationsDialog 单向 1 层 prop drilling，**用 `useState` 即可**，不引入新的 Recoil atom。

```tsx
import { useSearchParams } from "react-router-dom";

// In UserPopMenuDrawer / UserPopMenuRail
const [searchParams, setSearchParams] = useSearchParams();
const [notificationsDialogOpen, setNotificationsDialogOpen] = useState(false);
const [focusedMessageId, setFocusedMessageId] = useState<number | null>(null);

useEffect(() => {
  if (searchParams.get("open-notifications") === "1") {
    setNotificationsDialogOpen(true);
    const mid = searchParams.get("message-id");
    setFocusedMessageId(mid ? Number(mid) : null);
    // 清理 URL，避免刷新重复触发
    const next = new URLSearchParams(searchParams);
    next.delete("open-notifications");
    next.delete("message-id");
    setSearchParams(next, { replace: true });
  }
}, [searchParams, setSearchParams]);
```

**`NotificationsDialog` 协议扩展**（`src/frontend/client/src/components/NotificationsDialog.tsx`）：
- 新增 prop `focusedMessageId?: number`
- 打开后自动 scrollIntoView 到该消息行，并标记 highlight 1.5s
- 若该消息属于"approve"页 → 切到 request tab；否则保留 all tab

**未登录场景**：先走标准登录跳转，登录回跳保留原 query 参数（现有路由策略已支持，无需新增逻辑——验证一下）。

#### 1.7.3 后端构造 URL

`_payload.py` 内：

```python
def build_textcard_url(message_id: int) -> str:
    base = settings.in_app_message_forwarding.cofco.bisheng_inbox_url.rstrip("/")
    return f"{base}/?open-notifications=1&message-id={message_id}"
```

### 1.9 接收人解析

```python
def resolve_eplus_recipient(target_user_id: int) -> tuple[str | None, str]:
    """Return (e_plus_userid, skip_reason)."""
    user = UserDao.get_user(target_user_id)
    if not user:
        return None, "user_not_found"
    if user.source != "cofco_eplus":
        return None, f"source={user.source}_not_eplus"
    if not user.external_id:
        return None, "external_id_empty"
    return user.external_id, ""
```

**前置假设**（运维 / 数据侧确认，**不属于本功能开发范围**）：
- 中粮租户用户：`User.source = 'cofco_eplus'`、`User.external_id = E+ 员工工号`
- 灌入方式：现有 `org_sync` 体系（参考 wecom provider 模式）扩展或 SSO 时落库

如未对齐，运行期表现：每条消息都走静默跳过分支（INFO 日志，无错误）；上线后补数据立即生效。

### 1.10 失败语义（fire-and-forget）

| 场景 | 处理 | 日志 |
|---|---|---|
| 开关关 | 不进入 forwarder | — |
| `action_code` 不在白名单 | 跳过 | DEBUG |
| 接收人无 external_id | 跳过 | INFO（带原因） |
| HTTP timeout / 网络错误 | catch，不重试 | WARNING |
| 响应 `code != "0"` | catch，不重试 | WARNING（含 code、msg、touser） |
| Python 异常 | catch，不重试 | WARNING（含 traceback） |

## 二、功能 6：知识空间目录树展示

### 2.1 背景与目标

知识空间左侧支持多级目录树结构（**仅文件夹**，不含文件），默认折叠；类似 Windows 文件管理器效果。当前页面是 Tab 布局（文件列表 / 段落预览），无目录导航能力。

**重要事实**：后端基础设施已就绪：
- `KnowledgeFile.file_type`（0=DIR / 1=FILE）
- `KnowledgeFile.file_level_path`（路径如 `/123/456`）
- `SpaceFileDao.async_list_children(knowledge_id, parent_id)`
- **HTTP endpoint `GET /api/v1/knowledge_space/{space_id}/children` 已存在**（`knowledge_space.py:217`），且已支持 `parent_id` 参数

本期只缺：endpoint 增加一个可选 `file_type` 过滤、新增系统级开关、知识空间详情页左侧栏、Zustand 树状态、前端 API client 改造。

### 2.2 系统级开关（对应需求新增的"树形结构目录展示"开关）

中粮场内 UI 要求高，**不需要此功能**；其它客户**默认开启**。

**存储**：直接挂在 `config.yaml`（即 BiSheng 现有的全局 YAML 配置），新增字段：

```yaml
# config.yaml — 全局 YAML 配置
knowledge_space:
  tree_structured_directory_display: true   # 默认 true；中粮部署改 false
```

**读写接口**：完全复用现有 endpoint，**无需新增任何后端代码**：

| 操作 | endpoint |
|---|---|
| 前端读 | `GET /api/v1/config` （`src/backend/bisheng/api/v1/endpoints.py:82`）|
| 运维 / 超管改 | `POST /api/v1/config/save` （`endpoints.py:89`）|

**后端**：完全不需要新增 Python 配置模型；后端业务逻辑也不依赖此开关（API 不变，文件夹和文件都正常返回）。

**前端**：详情页加载时调 `getConfigApi`（已存在）→ YAML parse 出 `knowledge_space.tree_structured_directory_display` → 决定渲染**树版布局**还是**回退到原 Tab 布局**。

### 2.3 后端微调（仅一处可选过滤）

复用现有 endpoint：

```
GET /api/v1/knowledge_space/{space_id}/children
    ?parent_id=<int|空>    # 已支持
    &file_type=<int|空>    # 新增可选：0=只看文件夹，1=只看文件，空=两者都要
    &page=1
    &page_size=200
    &keyword=<可选>
```

**改动点**（约 5 行代码）：
1. `knowledge_space.py:217` endpoint 增加 `file_type: Optional[int] = Query(None)` 参数
2. `KnowledgeSpaceService.list_space_children()` 透传 `file_type`
3. `SpaceFileDao.async_list_children()` 在 filters 列表加 `if file_type is not None: filters.append(KnowledgeFile.file_type == file_type)`

**响应 schema 不变**，已含 `id / file_name / file_type / file_size / status / created_at / updated_at` 等。

**关于 `has_children` 字段**：树节点展开箭头需要知道文件夹是否有子文件夹。两种处理：
- **方案 A（推荐，零后端改动）**：前端**乐观显示** ▶ 箭头，点击展开后若 children 为空则隐藏并 toast "空文件夹"
- 方案 B：后端再加一列 `has_subdirs`（聚合 SQL 或 N+1），增加复杂度

本期采用 **方案 A**，YAGNI。

### 2.7 KnowledgeTree 组件关键行为

- 内部状态：`expandedIds: Set<number>`、组件级 `childrenMap: Map<number, KnowledgeNode[]>` 缓存
- 首屏：fetch 根（`parent_id` 空、`file_type=0`），渲染折叠列表，**所有节点 collapsed**

  id: number;
  file_name: string;
  file_type: 0 | 1;
  file_size: number | null;
  status?: number;
  created_at: string;
  updated_at: string;
}

export async function listKnowledgeChildren(params: {
  knowledge_id: number;
  parent_id: number | null;
  file_type?: 0 | 1;
  page?: number;
  page_size?: number;
  keyword?: string;
}): Promise<{ items: KnowledgeNode[]; total: number }> {
  return await axios.get(`/api/v1/knowledge_space/${params.knowledge_id}/children`, {
    params: {
      parent_id: params.parent_id ?? "",
      file_type: params.file_type,
      page: params.page ?? 1,
      page_size: params.page_size ?? 200,
      keyword: params.keyword,
    },
  });
}

### 2.9 交互细节

| 场景 | 行为 |
|---|---|
| flag=false（中粮） | 详情页保持原 Tab 布局，目录树完全不渲染 |
| 进入空间（flag=true） | 左侧树仅显示根级**文件夹**（折叠状态）；右侧显示根目录全部子节点（含子文件夹和文件） |
| 点击文件夹（树节点或列表行） | `currentParentId` 切到该节点，右侧列表更新，面包屑追加 |
| 点击文件 | 进入段落预览（沿用现有交互） |
| 点击树节点左侧 ▶ | 仅 expand / collapse，**不切换** `currentParentId`；首次 expand 懒加载 |
| 面包屑点击中间项 | `currentParentId` 跳到该层，`selectedFileId` 清空 |
| 顶部搜索框 | 当前层级 keyword 过滤 |
| 浏览器返回 / 刷新 | URL 带 `?parent_id=...&file_id=...`，恢复状态 |

## 五、风险与未覆盖项

| 风险 | 缓解 |
|---|---|
| 中粮 `User.external_id` / `source` 数据未灌入 | 上线后补数据即可生效；不阻塞 0515 代码交付 |
| E+ 接口实际是 HTTP 明文，secret 暴露 | 接口侧限制，BiSheng 端无法改进；运维确保内网链路 |
| 回调 URL 在中粮 E+ 端点击 → BiSheng 首登跳转时 query 丢失 | 验证现有 SSO 回跳行为；如丢失需在登录回调里补持久化 query |
| 树空文件夹乐观箭头点击后才发现为空 | 用户感知"无响应"风险较低（toast 提示）；监控访问量后视情况升级方案 B |
| flag=false 与树版代码并存导致 detail.tsx 复杂度上升 | 用早返回 + 子组件隔离两套布局，避免分支嵌套 |
| 功能 6 与既有"文件搜索"路径耦合 | 面包屑路径源于 click 链 + `file_level_path` 解析；e2e 验证 |
