# 2.4 信息源订阅商业化 PRD

## 1. 背景

### 1.1 业务背景

BiSheng 2.4 版本的 Channel（情报中心）模块已支持信息源订阅功能，用户可以添加网页、微信公众号等外部信息源，实现自动化的情报采集和文章聚合。

当前信息源功能完全免费且无数量限制，随着用户规模增长和 `bisheng-information` 服务的运营成本上升（爬虫带宽、存储、公众号接口费用），需要引入分层商业化模式，在保持开源社区活力的同时实现可持续运营。

### 1.2 核心思路

采用 **SaaS Key + 在线验证** 的方式，将信息源配额与 Key 绑定，通过三个层级覆盖从社区用户到大型企业的不同需求：

| 层级 | 获取方式 | 信息源上限 | 定位 |
|------|---------|-----------|------|
| 社区版 (Community) | 官网免费注册 Key | 3 | 个人开发者、体验评估 |
| 商业版 (Pro) | 购买 Gateway 商业拓展套件 | 30 | 中小企业生产使用 |
| 企业版 (Enterprise) | 联系官方按量付费 | 自定义 | 大型企业、重度使用 |

### 1.3 设计原则

1. **必须注册**：未填入有效 Key 的实例，信息源功能不可用（添加信息源时直接报错）
2. **实时在线验证**：每次添加信息源时调用官方 License 服务校验配额，不考虑离线场景
3. **Key 绑定信息源**：配额以租户为单位，Key 与租户绑定，整个租户共享信息源上限
4. **升级即换 Key**：从社区版升到商业版，用户替换为新的 Key 即可，无复杂迁移逻辑

## 2. 目标与优先级

3. 前端 Key 管理界面（系统设置页）
4. 前端超限报错提示与升级引导

1. 配额用量展示（已用 N / 上限 M）
2. 前端频道管理页信息源配额提示条

## 3. 详细设计

### 3.1 License 服务接口契约

BiSheng 后端作为客户端，调用官方 License 服务进行配额校验。以下为接口契约定义（License 服务的具体实现不在本 PRD 范围）。

#### 3.1.1 校验配额

```
POST {license_service_url}/api/v1/license/check
Content-Type: application/json

Request:
{
  "key": "bsg_xxxxxxxxxxxxxxxx",
  "resource_type": "information_source",
  "action": "create"
}

Response (200 OK):
{
  "allowed": true,
  "tier": "community",          // community | pro | enterprise
  "quota": {
    "information_source": {
      "limit": 3,               // 上限, -1 表示无限
      "used": 2                 // 已使用数量（由 License 服务维护计数）
    }
  }
}

Response (200 OK, 超限):
{
  "allowed": false,
  "tier": "community",
  "quota": {
    "information_source": {
      "limit": 3,
      "used": 3
    }
  },
  "upgrade_url": "https://bisheng.dataelem.com/pricing"
}

Response (401 Unauthorized):
{
  "error": "invalid_key",
  "message": "Key 无效或已过期"
}
```

#### 3.1.2 查询配额信息

```
GET {license_service_url}/api/v1/license/quota?key=bsg_xxx&resource_type=information_source

Response (200 OK):
{
  "tier": "pro",
  "quota": {
    "information_source": {
      "limit": 30,
      "used": 12
    }
  },
  "upgrade_url": "https://bisheng.dataelem.com/pricing"
}
```

#### 3.1.3 释放配额（删除信息源时）

```
POST {license_service_url}/api/v1/license/release
Content-Type: application/json

Request:
{
  "key": "bsg_xxxxxxxxxxxxxxxx",
  "resource_type": "information_source",
  "resource_id": "source_uuid_xxx"
}

Response (200 OK):
{
  "success": true,
  "quota": {
    "information_source": {
      "limit": 30,
      "used": 11
    }
  }
}
```

### 3.2 后端改造

#### 3.2.1 配置项新增

在 `config.yaml` 中新增 License 配置段：

```yaml
license:
  key: ""                                          # 社区 Key 或商业 Key
  service_url: "https://license.bisheng.dataelem.com"  # 官方 License 服务地址
  timeout: 5                                        # 请求超时（秒）
```

支持环境变量注入：`key: !env ${BISHENG_LICENSE_KEY}`

#### 3.2.2 LicenseClient 封装

新增 `core/license/client.py`，封装与 License 服务的交互：

```python
class LicenseClient:
    """官方 License 服务客户端，单例模式，ApplicationContextManager 初始化"""

    async def check_quota(self, resource_type: str, action: str) -> QuotaCheckResult
    async def get_quota(self, resource_type: str) -> QuotaInfo
    async def release_quota(self, resource_type: str, resource_id: str) -> QuotaInfo
```

核心行为：
- 初始化时校验 Key 有效性，无效则日志告警但不阻止启动（其他功能不受影响）
- `check_quota` 在添加信息源前调用，返回 `allowed=false` 时抛出业务异常
- License 服务不可达时，直接报错，不做降级（按需求不考虑离线）

#### 3.2.3 信息源添加流程改造

改造 `channel/domain/services/channel_service.py` 中的信息源添加逻辑：

```
原流程: 用户添加信息源 → 调 bisheng-information 服务 → 写入本地缓存
新流程: 用户添加信息源 → LicenseClient.check_quota() → 通过则继续 → 调 bisheng-information 服务 → 写入本地缓存
                                                      → 不通过则返回错误码 19001
```

改造点：
1. `channel_manager.py` 的 `add_website_source` 和 `add_wechat_source` 端点，在调用 `bisheng-information` 之前插入配额校验
2. 新增错误码 `190xx` 系列（模块编码 190 = license，`130xx` 已被 channel/chat 占用）：
   - `19001`：信息源配额已用尽（`InformationSourceQuotaExceeded`）
   - `19002`：License Key 未配置（`LicenseKeyNotConfigured`）
   - `19003`：License Key 无效（`LicenseKeyInvalid`）
   - `19004`：License 服务不可达（`LicenseServiceUnavailable`）
3. 删除信息源时调用 `LicenseClient.release_quota()` 释放配额

#### 3.2.4 配额查询 API

新增端点供前端展示配额用量：

```
GET /api/v1/channel/manager/quota
Authorization: Bearer {jwt}

Response:
{
  "status_code": 200,
  "data": {
    "tier": "community",
    "information_source": {
      "limit": 3,
      "used": 2
    },
    "upgrade_url": "https://bisheng.dataelem.com/pricing"
  }
}
```

#### 3.2.5 Key 管理 API

新增端点供系统管理员配置和验证 Key：

```
# 保存 Key（仅系统管理员）
PUT /api/v1/system/license
{
  "key": "bsg_xxxxxxxxxxxxxxxx"
}

# 查询当前 License 状态
GET /api/v1/system/license
Response:
{
  "status_code": 200,
  "data": {
    "key_masked": "bsg_****xxxx",     # 脱敏显示
    "tier": "community",
    "valid": true,
    "quotas": {
      "information_source": { "limit": 3, "used": 2 }
    }
  }
}
```

Key 保存后写入 `config.yaml`（加密存储，复用现有 Fernet 机制），同时刷新 LicenseClient 内部状态。

### 3.3 前端改造（Platform 管理端）

#### 3.3.1 系统设置 — License 管理页

**入口**：系统设置 → 新增「License 管理」菜单项

**页面内容**：
- Key 输入框（密码类型，支持显示/隐藏切换），保存按钮
- 保存后展示验证结果：
  - 成功：显示当前层级标签（社区版/商业版/企业版）、配额用量进度条
  - 失败：红色提示"Key 无效，请检查后重试"
- 底部引导文案：
  - 未配置 Key："前往 bisheng.dataelem.com 免费注册获取 Community Key"
  - 社区版："需要更多信息源？了解商业版 →"
  - 商业版："需要更大配额？联系我们 →"

**权限**：仅系统管理员（super_admin）可见此页面

#### 3.3.2 信息源添加 — 超限报错

当用户在频道管理页添加信息源（网页或公众号），后端返回 `19001` 错误码时：

**Toast 提示**：`"信息源配额已用尽（已使用 {used}/{limit}），请升级 License 或删除不需要的信息源"`

**无 Key 时**（错误码 `19002`）：
Toast 提示：`"请先在系统设置中配置 License Key"`

#### 3.3.3 频道管理页 — 配额提示（P1）

在频道管理页的信息源列表区域上方，添加配额提示条：

- 正常（<80%）：灰色文字 `"信息源配额：2 / 3"`
- 接近上限（≥80%）：黄色警告 `"信息源配额即将用尽：2 / 3"`
- 已满（100%）：红色提示 `"信息源配额已满：3 / 3，升级 License 获取更多配额"`

### 3.4 多租户适配

- Key 存储在租户级配置中（`config.yaml` 的 `license.key` 或未来的 `tenant_config` 表）
- 单租户模式：全局一个 Key
- 多租户模式（v2.5）：每个租户独立配置 Key，配额互不影响。需在 `LicenseClient` 中按 `tenant_id` 区分请求

### 3.5 信息源类型覆盖

当前计入配额的信息源类型：

| 类型 | source_type | 计入配额 |
|------|------------|---------|
| 网页 | website | 是 |
| 微信公众号 | wechat | 是 |

后续若新增信息源类型（如 RSS、API 数据源等），默认计入配额。可通过 License 服务端配置特定类型是否计费。

## 4. 用户旅程

### 4.1 社区用户首次使用

```
1. 用户部署 BiSheng，进入频道管理，尝试添加信息源
2. 后端返回 19002 "License Key 未配置"
3. 前端 Toast 提示，引导前往系统设置
4. 用户访问 bisheng.dataelem.com 注册获取免费 Community Key
5. 在系统设置 → License 管理页填入 Key，保存
6. 验证通过，显示"社区版 — 信息源 0/3"
7. 返回频道管理，成功添加信息源
```

### 4.2 社区用户触达上限

```
1. 用户已有 3 个信息源，尝试添加第 4 个
2. 后端调 License 服务，返回 allowed=false
3. 前端 Toast："信息源配额已用尽（已使用 3/3），请升级 License 或删除不需要的信息源"
4. 用户选择：
   a. 删除已有信息源 → 配额释放 → 重新添加
   b. 联系销售购买商业版 Gateway 套件 → 获取 Pro Key → 替换 Key → 配额提升至 30
```

### 4.3 商业版升级

```
1. 用户购买 Gateway 商业拓展套件，获取 Pro Key
2. 在系统设置 → License 管理页替换 Key
3. 验证通过，显示"商业版 — 信息源 3/30"
4. 已有的 3 个信息源保持不变，可继续添加
```

## 5. 技术实现要点

### 5.1 计数由 License 服务维护

信息源的使用计数（used）由官方 License 服务维护，而非 BiSheng 本地统计。原因：

1. **防篡改**：本地计数可被绕过（直接改数据库）
2. **一致性**：Key 与信息源绑定的权威数据在 License 服务端
3. **简化本地逻辑**：BiSheng 后端只需调接口，不需维护计数器

每次添加/删除信息源时，BiSheng 后端调 License 服务的 `check` / `release` 接口，License 服务端完成计数增减。

### 5.2 Key 格式

建议 Key 格式：`bsg_{tier}_{random}`，例如：
- 社区版：`bsg_com_a1b2c3d4e5f6g7h8`
- 商业版：`bsg_pro_x9y8z7w6v5u4t3s2`
- 企业版：`bsg_ent_m1n2o3p4q5r6s7t8`

前缀可用于本地快速判断层级（但最终以 License 服务返回为准）。

### 5.3 错误码汇总

| 错误码 | 常量名 | 含义 | HTTP Status |
|--------|--------|------|-------------|
| 19001 | InformationSourceQuotaExceeded | 信息源配额已用尽 | 200 (业务错误) |
| 19002 | LicenseKeyNotConfigured | License Key 未配置 | 200 (业务错误) |
| 19003 | LicenseKeyInvalid | License Key 无效或已过期 | 200 (业务错误) |
| 19004 | LicenseServiceUnavailable | License 服务不可达 | 200 (业务错误) |

注：遵循项目统一响应规范，业务错误通过 `resp_500(code, msg)` 返回，HTTP 状态码始终 200。
