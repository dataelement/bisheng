# BiSheng 2.5 多租户需求文档

> 本文档是《2.5 权限管理体系改造 PRD》的增量补充，定义多租户能力的需求与设计。文档中以 `[原 PRD §X.X]` 标注与原 PRD 的关联章节，以 `[合入自多租户管理 PRD]` 标注从《2.5 多租户管理 PRD》合入的功能。

---

## 1. 背景与目标

### 1.1 背景

BiSheng 的目标客户为大型集团企业（如中粮、首钢），采用私有化部署模式。集团内部存在多个子公司或事业部，各自拥有独立的组织架构、知识库、应用和数据资产，核心诉求为：

- **数据隔离**：子公司 A 的知识库、应用、对话记录等不可被子公司 B 访问
- **独立管理**：各子公司拥有自己的管理员，独立管理本单位的用户、部门和资源
- **统一管控**：集团总部需要跨子公司视角的用量统计、配额管控和统一模型服务

2.5 版本同步进行权限体系改造（RBAC → ReBAC），二者共享大量改造点（部门树、用户管理、权限检查链路）。合并到同一版本交付，可避免二次改造的成本和风险。

### 1.2 核心概念

| 概念 | 定义 |
|------|------|
| **租户（Tenant）** | 数据隔离的最小单元，对应集团下属的子公司或事业部。每个租户拥有独立的部门树、用户、资源和配额 |
| **系统管理员** | 跨租户的超级管理员（对应 `[原 PRD §3.1]` 中 `system:global` 的 `super_admin`），可管理所有租户 |
| **租户管理员** | 租户内最高权限管理员，管理本租户的部门、用户、角色和资源，不可跨租户操作 |
| **逻辑隔离** | 所有租户共享同一数据库实例，通过 `tenant_id` 字段过滤实现数据隔离 |
| **默认租户** | 系统升级时自动创建的租户（tenant_id=1），所有存量数据归入此租户，确保向后兼容 |

**用户数据归属规则**

用户可属于多个租户，切换租户时数据可见性按以下规则区分：

| 数据类别 | 归属 | 说明 |
|----------|------|------|
| 用户基本信息（用户名、头像、个人偏好） | **跟人走**，跨租户共享 | 存储在 `user` 表，不加 `tenant_id` |
| 对话记录、创建的资源 | **跟租户走**，按 `tenant_id` 隔离 | 切换到另一个租户后不可见 |
| 操作日志、审计记录 | **跟租户走**，按 `tenant_id` 隔离 | 审计日志按租户独立 |
| 角色、部门归属 | **跟租户走** | 同一用户在不同租户中可有不同角色和部门 |

### 1.3 层级关系

```
系统管理员（跨租户）
│
├── 租户 A（子公司 A）
│   ├── 租户管理员 A
│   ├── 部门树 A（根部门 = 租户 A）
│   │   ├── 技术部
│   │   │   ├── 前端组
│   │   │   └── 后端组
│   │   └── 业务部
│   ├── 用户组 A
│   ├── 角色 A
│   └── 资源 A（知识库、应用、工具...）
│
├── 租户 B（子公司 B）
│   ├── 租户管理员 B
│   ├── 部门树 B
│   └── ...
│
└── 系统级共享
    ├── 模型服务（所有租户共用）
    ├── 系统级全局角色
    └── 系统模板（可下发到租户）
```

**核心设计：租户 = 部门树的根节点。** 创建租户时自动生成一个根部门，该租户内所有部门都挂在此根部门下。这使得多租户与 `[原 PRD §3.2.1]` 中的部门树天然融合，无需额外的层级抽象。

### 1.4 目标与优先级

在 `[原 PRD §2]` 基础上新增多租户相关目标：

| 优先级 | 目标 | 说明 |
|--------|------|------|
| **P0** | 租户数据隔离基础能力 | tenant_id 自动过滤、外部存储隔离、租户上下文透传 |
| **P0** | 权限检查链路集成租户 | 租户管理员短路、跨租户访问拦截 |
| **P1** | 租户管理后台 | 租户 CRUD、配额设置、管理员指定 |
| **P1** | 登录与租户切换 | 多租户用户登录时选择/切换租户 |
| **P1** | 租户级配额管理 | 存储、资源数量、模型调用量的租户级限额 |
| **P2** | 共享资源与模板下发 | 集团统一模板下发到各租户 |
| **P2** | 租户级审计日志 | 按租户隔离的操作审计 |
| **P2** | 租户用量统计仪表盘 | 系统管理员视角的跨租户用量分析 |

## 2. 多租户模型设计

### 2.1 租户数据模型

#### tenant 表

```sql
CREATE TABLE tenant (
    id               INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    tenant_code      VARCHAR(64)   NOT NULL COMMENT '租户编码（唯一标识，如 COFCO/SHOUGANG），英文+数字+下划线',
    tenant_name      VARCHAR(128)  NOT NULL COMMENT '租户名称（如"中粮集团"）',
    logo             VARCHAR(512)  DEFAULT NULL COMMENT '租户 Logo URL（MinIO 路径）',
    root_dept_id     INT UNSIGNED  DEFAULT NULL COMMENT '根部门 ID，创建租户时自动生成',
    status           VARCHAR(16)   NOT NULL DEFAULT 'active' COMMENT '状态: active / disabled / archived',
    contact_name     VARCHAR(64)   DEFAULT NULL COMMENT '联系人姓名',
    contact_phone    VARCHAR(32)   DEFAULT NULL COMMENT '联系人电话',
    contact_email    VARCHAR(128)  DEFAULT NULL COMMENT '联系人邮箱',
    quota_config     JSON          DEFAULT NULL COMMENT '租户级资源配额（详见 §6）',
    storage_config   JSON          DEFAULT NULL COMMENT '租户级存储配置覆盖（可选）',
    create_user      INT UNSIGNED  DEFAULT NULL COMMENT '创建人',
    create_time      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    UNIQUE KEY uk_tenant_code (tenant_code),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='租户表';
```

#### user_tenant 表（用户-租户多对多关联）

```sql
CREATE TABLE user_tenant (
    id               BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id          INT UNSIGNED  NOT NULL,
    tenant_id        INT UNSIGNED  NOT NULL,
    is_default       TINYINT(1)    NOT NULL DEFAULT 0 COMMENT '是否为用户的默认租户',
    status           VARCHAR(16)   NOT NULL DEFAULT 'active' COMMENT 'active / disabled',
    last_access_time DATETIME      DEFAULT NULL COMMENT '最后一次访问该租户的时间',
    join_time        DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_user_tenant (user_id, tenant_id),
    INDEX idx_tenant (tenant_id),
    INDEX idx_user_default (user_id, is_default)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户-租户关联表';
```

**设计说明**：

- 用户可属于多个租户（如集团 IT 人员），但同一时刻只在一个租户上下文中操作
- `is_default` 标记用户登录时的默认租户（仅一个租户时自动设为默认）
- `last_access_time` 记录用户最后访问该租户的时间，用于租户选择列表排序 `[合入自多租户管理 PRD]`
- 禁用租户（status=disabled）后，该租户下所有用户无法进入该租户上下文

### 2.2 租户与部门的关系

**核心原则：租户 = 部门树的根节点。**

在 `[原 PRD §3.2.1]` 定义的 department 表基础上，新增 `tenant_id` 字段：

```sql
ALTER TABLE department ADD COLUMN tenant_id INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '所属租户 ID';
ALTER TABLE department ADD INDEX idx_tenant (tenant_id);
```

**创建租户时的自动操作**：

1. 创建 tenant 记录
2. 自动创建根部门：`INSERT INTO department (name, tenant_id, parent_id, path, source) VALUES ('{租户名称}', {tenant_id}, NULL, '/{dept_id}/', 'system')`
3. 回写 `tenant.root_dept_id = {dept_id}`
4. 创建 OpenFGA 元组：`(tenant:{tenant_id}, admin, user:{creator_id})`

**部门树的作用域**：

- 系统管理员：可看到所有租户的部门树，按租户分组展示
- 租户管理员：只能看到本租户的部门树（根节点 = 本租户的根部门）
- 部门管理员：只能看到本部门及子部门（与 `[原 PRD §3.2.1]` 保持一致）

**第三方同步** `[原 PRD §3.2.1 第三方同步]` **增量修改**：

- 同步配置新增"目标租户"选项，同步的部门和人员自动归属到目标租户
- 不同租户可以对接不同的第三方数据源

### 2.4 功能开关

多租户功能通过配置开关控制，社区版或小型部署场景可关闭：

```yaml
# config.yaml
multi_tenant:
  enabled: false                    # 是否启用多租户，默认关闭
  default_tenant_code: "default"   # 默认租户编码
```

**关闭时的行为**（`multi_tenant.enabled = false`）：

- 系统自动创建一个默认租户（tenant_id=1），所有用户和数据归入此租户
- 前端不展示租户切换入口、不展示租户管理菜单
- 登录流程无租户选择步骤
- API 不要求 tenant_id 参数，自动使用默认租户
- 行为与现有单租户系统完全一致

**开启时的行为**（`multi_tenant.enabled = true`）：

- 前端展示租户切换入口和租户管理菜单（系统管理员可见）
- 登录后根据用户所属租户数量决定是否展示租户选择
- 所有数据查询自动注入 `WHERE tenant_id = ?`

## 3. 权限体系集成

> 以下为 `[原 PRD §3.1]` OpenFGA DSL 和权限检查流程的增量修改。

### 3.1 OpenFGA DSL 增加 tenant 类型

在现有 DSL（`[原 PRD §3.1]`）的 `system` 和 `department` 之间插入：

```yaml
# ============================================
# 租户 — 数据隔离单元
# ============================================
type tenant
  relations
    define admin: [user]                              # 租户管理员
    define member: [user]                             # 租户成员（所有属于该租户的用户）
```

**设计说明**：

| 决策 | 说明 |
|------|------|
| tenant 不参与资源级权限继承 | 资源的 viewer/editor/manager/owner 仍由 department/user/user_group 管理，tenant 只做身份校验和管理员短路 |
| tenant.admin 用于管理员短路 | 类似 `system.super_admin`，但作用域限制在本租户内 |
| tenant.member 用于归属校验 | 验证"用户是否属于当前租户"，防止跨租户访问 |
| department 不加 tenant 关系 | 部门的租户归属通过 MySQL 的 `department.tenant_id` 字段管理，不在 OpenFGA 中重复建模 |

### 3.2 管理员层级扩展

`[原 PRD §3.1]` 中的权限检查流程修改为三级短路：

```
请求进入
  ↓
① 系统管理员检查：system:global.super_admin → 全部放行
  ↓ （非系统管理员）
② 租户归属检查：当前用户是否属于当前租户（tenant:{id}.member）→ 不属于则拒绝
  ↓ （属于当前租户）
③ 租户管理员检查：tenant:{id}.admin → 本租户内全部放行
  ↓ （非租户管理员）
④ ReBAC 权限检查：OpenFGA Check（与原 PRD 一致）
  ↓
⑤ RBAC 菜单/配额检查（与原 PRD 一致）
```

**与原 PRD 的差异**：

- 原流程只有 ①④⑤ 三步，现在在 ① 和 ④ 之间插入 ②③ 两步
- ② 租户归属检查是**安全底线**，确保用户不可能通过任何手段访问其他租户的数据
- ③ 租户管理员是 `[原 PRD §3.2.1]` 中"超级管理员"概念的租户内等价物

### 3.3 管理员身份对照表

| 角色 | OpenFGA 关系 | 作用域 | 典型操作 |
|------|-------------|--------|---------|
| 系统管理员 | `system:global` / `super_admin` | 跨租户 | 创建/管理租户、管理系统级配置、跨租户用量统计 |
| 租户管理员 | `tenant:{id}` / `admin` | 租户内 | 管理本租户的部门/用户/角色/资源，设置租户内配额 |
| 部门管理员 | `department:{id}` / `admin` | 部门内 | 管理本部门及子部门的人员和资源（与 `[原 PRD §3.2.1]` 一致）|

## 4. 数据隔离实现

### 4.1 查询层自动过滤

所有包含 `tenant_id` 字段的业务表，在 SELECT 查询时自动注入 `WHERE tenant_id = 当前租户ID`，业务代码无需逐个修改。

**安全保障**：

- 系统管理员的请求不设置租户上下文（跳过租户过滤，可看全部数据）
- 未设置租户上下文时，查询**不返回任何数据**（而非返回全部），防止开发疏漏
- Celery 异步任务必须从任务上下文中恢复当前租户ID

> 具体实现方案（SQLAlchemy event 机制等）见技术方案文档。

### 4.2 写入层自动填充

创建记录时，如果表包含 `tenant_id` 字段且未手动赋值，自动从当前请求上下文获取 `tenant_id` 并填充。

> 具体实现方案见技术方案文档。

### 4.3 租户上下文传递

| 场景 | 传递方式 |
|------|---------|
| **HTTP 请求** | JWT token 中携带 `tenant_id` 字段。中间件解析 JWT 后设置 `current_tenant_id` ContextVar |
| **WebSocket** | 连接建立时（握手阶段）从 JWT 解析 `tenant_id` 并绑定到连接上下文 |
| **Celery 异步任务** | 发送任务时将 `tenant_id` 写入 task headers，Worker 执行前从 headers 恢复到 ContextVar |
| **定时任务（Beat）** | 遍历所有活跃租户逐个执行，每次迭代设置对应的 `current_tenant_id` |

### 4.4 外部存储隔离

#### 4.4.1 Milvus 向量存储

| 方案 | 说明 |
|------|------|
| **Collection 名称前缀（推荐）** | 新租户的 collection 命名为 `t{tenant_id}_col_{timestamp}_{uuid}`。默认租户保持原命名不变（向后兼容） |
| Partition 机制 | 在同一 collection 中按 tenant_id 分 partition。优点是 collection 数量少，缺点是删除租户时清理困难 |

**推荐方案一**（Collection 前缀），理由：

- 租户删除/归档时可直接 drop collection，干净利落
- 不同租户的 collection 参数可独立调整（如 index type）
- 5~50 个租户不会导致 collection 数量爆炸

#### 4.4.2 MinIO 对象存储

支持两种隔离模式，通过 `tenant.storage_config` 配置：

**模式一：路径前缀隔离（默认）**

```
bisheng/                          # 共享 bucket
├── default/                      # 默认租户（升级兼容，等同于无前缀的旧路径）
│   ├── knowledge/
│   ├── flow/
│   └── ...
├── tenant_cofco/                 # 中粮租户
│   ├── knowledge/
│   ├── flow/
│   └── ...
└── tenant_shougang/              # 首钢租户
    └── ...
```

- 默认租户保持原路径不变（无 tenant 前缀），确保升级后旧文件无需迁移
- 新租户使用 `tenant_{tenant_code}/` 路径前缀
- 上传/下载时，根据 `current_tenant_id` 自动拼接路径前缀

**模式二：独立存储（高级，按租户配置）** `[合入自多租户管理 PRD]`

部分客户（如中粮）的 Bucket 为私有访问，要求租户自带独立存储。此模式下：

- `tenant.storage_config` 中配置该租户独立的 `bucket`、`access_key`、`secret_key`、`endpoint`
- 后端开发**分发临时上传凭证**接口（Presigned URL），前端通过 Presigned URL 直传到租户的独立 Bucket
- 下载/预览文件同样通过**动态生成的临时下载链接**访问
- 前端上传流程变更：`POST /api/v1/upload` → `GET /api/v1/upload_url`（获取 Presigned URL）→ `PUT {presigned_url}`（直传文件）

```json
// tenant.storage_config 示例（独立存储模式）
{
  "mode": "independent",
  "minio": {
    "endpoint": "https://oss.cofco.com",
    "bucket": "cofco-bisheng",
    "access_key": "****",
    "secret_key": "****"
  }
}
```

> **注意**：两种模式不互斥。平台默认使用路径前缀隔离，特定租户可覆盖为独立存储模式。未配置 `storage_config` 的租户使用平台默认存储。

#### 4.4.3 Elasticsearch

**Index 名称前缀**：

- 默认租户保持原 index 名称不变
- 新租户 index 命名为 `t{tenant_id}_{original_index_name}`
- 所有 ES 查询自动附加租户对应的 index 名称

#### 4.4.4 Redis

**Key 命名空间**：

- 默认租户保持原 key 格式不变（向后兼容）
- 新租户 key 格式：`t:{tenant_id}:{original_key}`
- 封装 tenant-aware 的 Redis 工具类，自动添加/解析租户前缀

## 5. 系统管理功能

### 5.1 租户管理（新增模块，仅系统管理员可见）

#### 5.1.0 首次登录初始化 `[合入自多租户管理 PRD]`

当多租户功能开启（`multi_tenant.enabled = true`）时，系统首次部署的初始化流程：

1. 首个注册/登录的用户自动成为**平台超级管理员**（`system:global` 的 `super_admin`）
2. 登录后系统检测到尚无租户，**强制弹出新建租户页面**，平台超级管理员必须先创建至少一个租户才能进行后续操作
3. 创建首个租户后，该管理员自动成为该租户的管理员，进入租户工作台

> 多租户关闭时（`multi_tenant.enabled = false`），沿用现有逻辑：首个用户自动获得管理员权限，自动归入默认租户。

#### 5.1.1 租户列表

| 项目 | 说明 |
|------|------|
| 入口 | 系统管理 → 租户管理（新增一级菜单） |
| 可见性 | 仅系统管理员可见 |
| 列表字段 | 租户名称、租户编码、状态（启用/禁用/已归档）、用户数、存储用量/配额、创建时间 |
| 存储用量 | 实时统计该租户下所有知识空间文件总容量作为"已用量"，以进度条形式展示（已用量/总配额 × 100%）`[合入自多租户管理 PRD]` |
| 操作 | 搜索（按名称/编码）、筛选（按状态）、分页 |

#### 5.1.2 新建租户

| 字段 | 说明 |
|------|------|
| 租户名称 | 必填，2~50 字符，不支持特殊表情符号 |
| 租户编码 | 必填，唯一，英文+数字+下划线，2~32 字符。创建后不可修改。系统自动生成格式 `t_{随机}` 供参考，也可手动填写 |
| Logo | 可选，图片上传 |
| 联系人信息 | 可选，姓名/电话/邮箱 |
| 存储配额 | 必填，单位 GB，默认值 100 `[合入自多租户管理 PRD]` |
| 租户管理员 | 必填，至少指定一人。支持两种方式：① 从已有用户中选择（被选中的用户自动加入该租户）；② 新建管理员账号（填写人员 ID + 初始密码，系统自动创建用户并加入该租户）`[合入自多租户管理 PRD]` |

**创建后自动执行**：

1. 创建 tenant 记录
2. 创建根部门（部门名称 = 租户名称）
3. 将租户管理员加入 `user_tenant` 关联
4. 写入 OpenFGA 元组：
   - `(tenant:{id}, admin, user:{admin_id})`（每个管理员）
   - `(tenant:{id}, member, user:{admin_id})`（每个管理员）
5. 为租户管理员在 OpenFGA 中写入根部门的 `admin` 关系

#### 5.1.3 编辑租户

| 可修改 | 不可修改 |
|--------|---------|
| 租户名称、Logo、联系人信息 | 租户编码 |
| 状态（启用/禁用） | — |
| 租户管理员（增减） | — |
| 租户管理员密码（重置） `[合入自多租户管理 PRD]` | — |
| 配额设置 | — |

**禁用租户** `[合入自多租户管理 PRD 细化]`：

拨至"关闭"时，弹出二次确认框：*"停用后，该企业下所有成员将被立即强制下线，且无法再次登录系统。是否确认停用？"*

确认后执行：

1. **立即吊销**该 `tenant_id` 下所有活跃账号的 JWT Token，触发全员强制登出
2. **API 层拦截**所有携带该 `tenant_id` 的请求，返回 403 Forbidden，提示"企业已停用"
3. 该租户的 **Celery 异步任务**立即挂起暂停（知识库处理、工作流执行等）
4. 该租户的**第三方同步定时任务**暂停
5. 数据保留，不删除。重新启用后恢复访问

**归档/删除租户**：

- 归档：数据保留但不可访问，管理员可恢复
- 物理删除：需二次确认，提示"租户内所有数据（知识库、应用、对话记录等）将被永久删除"
- 删除前检查：是否有运行中的应用、是否有活跃用户

#### 5.1.4 租户配额设置

在租户编辑页面设置租户级资源配额（详见 §6）。

## 6. 配额管理体系

### 6.1 三级配额层次

```
① 租户级配额（tenant.quota_config）— 硬性上限
   该租户下所有用户的资源总量不能超过此限额
     ↓
② 角色级配额（role.quota_config）[原 PRD §2.5] — 用户级上限
   单个用户的资源创建上限
     ↓
③ 用户有效配额 = min(租户剩余配额, 角色级配额)
```

### 6.2 租户配额字段

`tenant.quota_config` JSON 结构：

```json
{
  "storage_gb": 100,              // 存储空间上限（GB），仅统计 MinIO 文件大小
  "knowledge_space": 50,          // 知识空间数量上限
  "knowledge_file": 5000,         // 知识文件总数上限
  "workflow": -1,                 // 工作流数量上限，-1=不限
  "assistant": -1,                // 助手数量上限
  "tool": -1,                     // 工具数量上限
  "channel": 20,                  // 频道数量上限
  "model_tokens_monthly": 10000000, // 每月模型调用 token 上限，按自然月重置
  "user_count": 200               // 租户内用户数上限
}
```

### 6.3 配额检查逻辑

用户创建资源时，按以下顺序检查配额：

1. **租户级检查**：统计当前租户的资源总量，与 `tenant.quota_config` 中对应配额比较。已满则拒绝
2. **角色级检查**：统计当前用户已创建的资源数量，与用户角色配额比较。已满则拒绝
3. 用户有效配额 = min(租户剩余配额, 角色级配额)

**并发说明**：配额检查不使用分布式锁，在高并发场景下允许轻微超额（如两人同时上传最后一个额度的文件）。超额部分在下一次检查时会被阻断。

> 具体实现见技术方案文档。

### 6.4 存储配额超限行为 `[合入自多租户管理 PRD]`

当某租户的物理文件上传导致存储配额使用率达到 **100%** 时：

- 该租户内**所有成员**（包括租户管理员）在调用"上传文件"、"新建知识库"等占用存储空间的接口时，后端一律**阻断拦截**
- 前端提示：*"当前企业存储配额已耗尽（XX/XX GB），请联系平台方进行配额扩容。"*
- 已有资源的查看、编辑、删除操作不受影响
- 删除文件可释放配额，释放后恢复上传能力

### 6.5 配额监控（P2）

| 功能 | 说明 |
|------|------|
| 租户用量仪表盘 | 系统管理员可见，展示各租户的配额使用率（存储、资源数量、模型调用量），进度条渲染 `[合入自多租户管理 PRD]` |
| 告警阈值 | 使用率达到 80% 时黄色预警，90% 时橙色警告，100% 时红色且阻断创建 |
| 通知方式 | 站内信通知租户管理员 + 系统管理员 |

## 9. 数据迁移方案

> 基于 `[原 PRD §3.4]` 的迁移框架，以下为多租户的增量迁移步骤。

### 9.1 迁移策略概览

多租户迁移与权限迁移在同一升级脚本中执行，顺序为：

```
Phase 0: 多租户 DDL + 默认租户初始化（新增）
  ↓
Phase 1: 权限数据迁移到 OpenFGA（原 PRD）
  ↓
Phase 2: 双写过渡期（原 PRD）
  ↓
Phase 3: 切换到 OpenFGA（原 PRD）
  ↓
Phase 4: 清理旧表（原 PRD）
```

### 9.2 Phase 0：多租户初始化

#### 9.2.1 DDL 变更

```sql
-- 1. 创建 tenant 表
CREATE TABLE tenant (...);  -- 见 §2.1

-- 2. 创建 user_tenant 表
CREATE TABLE user_tenant (...);  -- 见 §2.1

-- 3. 所有业务表添加 tenant_id 字段（见 §2.3 清单）
ALTER TABLE department ADD COLUMN tenant_id INT UNSIGNED NOT NULL DEFAULT 0;
ALTER TABLE flow ADD COLUMN tenant_id INT UNSIGNED NOT NULL DEFAULT 0;
ALTER TABLE assistant ADD COLUMN tenant_id INT UNSIGNED NOT NULL DEFAULT 0;
-- ... 其余表同理

-- 4. 添加索引
ALTER TABLE department ADD INDEX idx_tenant (tenant_id);
ALTER TABLE flow ADD INDEX idx_tenant (tenant_id);
-- ... 其余表同理
```

#### 9.2.2 默认租户初始化

```sql
-- 1. 创建默认租户
INSERT INTO tenant (id, tenant_code, tenant_name, status)
VALUES (1, 'default', '默认租户', 'active');

-- 2. 创建默认租户的根部门
INSERT INTO department (name, tenant_id, parent_id, path, source)
VALUES ('默认租户', 1, NULL, '/1/', 'system');
-- 回写 root_dept_id
UPDATE tenant SET root_dept_id = LAST_INSERT_ID() WHERE id = 1;

-- 3. 所有业务表 tenant_id 回填为 1
UPDATE department SET tenant_id = 1 WHERE tenant_id = 0;
UPDATE flow SET tenant_id = 1 WHERE tenant_id = 0;
-- ... 其余表同理

-- 4. 所有现有用户加入默认租户
INSERT INTO user_tenant (user_id, tenant_id, is_default, status)
SELECT id, 1, 1, 'active' FROM user;

-- 5. 现有管理员写入 OpenFGA 租户管理员元组
-- 所有拥有 AdminRole 的用户 → (tenant:1, admin, user:{id})
```

#### 9.2.3 外部存储兼容

| 存储 | 迁移策略 |
|------|---------|
| **MinIO** | 默认租户不改路径，保持原有文件结构不变。新租户使用 `tenant_{code}/` 前缀 |
| **Milvus** | 默认租户的 collection 保持原命名。新租户使用 `t{tenant_id}_` 前缀 |
| **ES** | 默认租户的 index 保持原命名。新租户使用 `t{tenant_id}_` 前缀 |
| **Redis** | 默认租户的 key 保持原格式。新租户使用 `t:{tenant_id}:` 前缀 |

**核心原则：默认租户零迁移，新租户用新规范。** 这确保升级过程中不需要移动任何文件或重建索引。

### 9.3 迁移幂等性

- 所有 DDL 变更使用 `IF NOT EXISTS` / `IF NOT COLUMN_EXISTS` 防护
- 默认租户创建使用 `INSERT IGNORE` 或检查已存在则跳过
- `tenant_id` 回填使用 `WHERE tenant_id = 0` 条件，重跑不影响已回填的数据
- 迁移脚本记录进度 checkpoint，支持断点续传

## 10. 部署与配置变更

### 10.1 config.yaml 新增项

```yaml
# 多租户配置
multi_tenant:
  enabled: false                    # 是否启用多租户
  default_tenant_code: "default"   # 默认租户编码
  storage_isolation:
    milvus: "collection_prefix"    # collection_prefix | partition
    minio: "path_prefix"           # path_prefix | separate_bucket
    elasticsearch: "index_prefix"  # index_prefix
    redis: "key_prefix"            # key_prefix | separate_db
```

### 10.2 环境变量

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `BS_MULTI_TENANT_ENABLED` | 是否启用多租户 | `false` |

### 10.3 Docker Compose 变更

无需额外容器。多租户通过应用层逻辑实现，OpenFGA 服务已在 `[原技术方案 §3.1]` 中定义。

### 10.4 Nginx 配置（可选）

如果需要按子域名区分租户（`tenant-a.bisheng.local`），Nginx 需配置：

```nginx
# 可选：按子域名路由（大多数场景不需要）
server {
    server_name ~^(?<tenant>.+)\.bisheng\.local$;
    location / {
        proxy_pass http://frontend;
        proxy_set_header X-Tenant-Code $tenant;
    }
}
```

大多数私有化部署场景下，租户通过登录后的 JWT 区分，**无需 Nginx 子域名配置**。

## 附录 C：关键设计决策汇总

| 决策项 | 结论 | 原因 |
|--------|------|------|
| 隔离方式 | 逻辑隔离（共享数据库 + tenant_id） | 5~50 个租户，物理隔离运维成本过高 |
| 租户与部门关系 | 租户 = 部门树的根节点 | 与 `[原 PRD]` 部门树设计天然融合，无需额外层级抽象 |
| tenant 是否加入 OpenFGA | 是，作为独立 type | 需要 tenant admin 短路判定和 tenant member 归属校验 |
| 用户与租户关系 | 多对多（user_tenant 表） | 集团 IT 人员需要跨租户操作 |
| 默认租户策略 | 升级时自动创建，存量数据归入 | 确保向后兼容，零配置升级 |
| 外部存储隔离 | 默认路径前缀 + 可选独立存储 | 路径前缀满足大多数场景，独立存储满足中粮等强隔离需求 |
| 多租户开关 | config.yaml 中 `multi_tenant.enabled` | 社区版/小型部署可关闭，减少复杂性 |
| 配额层级 | 租户级（硬上限）> 角色级（用户上限） | 租户配额保障资源公平分配，角色配额控制个人使用 |
| 数据过滤机制 | SQLAlchemy event 自动注入 | 避免逐个 DAO 手动修改，降低遗漏风险 |
| JWT 变更 | payload 新增 tenant_id 字段 | 最小侵入，切换租户通过重新签发 JWT 实现 |
| 租户管理员创建 | 选择已有用户 + 新建用户两种方式 | 兼顾"集团IT统一管理"和"租户独立管理"两种场景 |

## 附录 D：与《多租户管理 PRD》的合并记录

> 本附录记录与另一位产品经理编写的《2.5 多租户管理 PRD》的对比结果，包括合入的功能和设计分歧的决策。

### 已合入的功能

| 来源功能 | 合入位置 | 说明 |
|---------|---------|------|
| 首次登录强制创建租户 | §5.1.0 | 多租户模式下首个用户必须先创建租户 |

| 停用租户的详细行为 | §5.1.3 | JWT 吊销、API 层拦截、异步任务挂起等具体操作 |
| 存储配额进度条 | §5.1.1 | 租户列表展示已用量/总配额进度条 |

| 租户管理员密码重置 | §5.1.3 | 平台管理员可修改租户超管密码 |
| 独立存储模式（Presigned URL） | §4.4.2 | 作为高级选项，支持租户自带 Bucket/AK/SK |

### 设计分歧与决策

| 分歧点 | 对方方案 | 本文档方案 | 决策理由 |
|--------|---------|-----------|---------|
| **存储隔离策略** | 物理隔离（每租户独立 Bucket/AK/SK，前端 Presigned URL 直传） | 默认逻辑隔离（路径前缀）+ 可选物理隔离 | 两种方案合并：路径前缀满足大多数场景，独立存储作为高级选项满足强隔离需求。避免所有租户都承担 Presigned URL 改造的成本 |
| **租户切换机制** | 前端改变 tenant_id，重新拉取数据（不重签 JWT） | 重新签发含 tenant_id 的 JWT + 页面刷新 | JWT 中必须包含 tenant_id 作为安全凭证，否则后端无法在 token 层面校验租户归属，仅靠前端传递 tenant_id 存在被篡改的风险 |
| **租户管理员创建** | 创建租户时新建管理员账号（填人员 ID + 密码） | 支持选择已有用户 + 新建用户两种方式 | 合并两种方式，兼顾"集团IT统一管理"和"租户独立管理"场景 |
