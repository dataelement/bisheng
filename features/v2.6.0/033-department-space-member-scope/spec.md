# Feature: 部门知识空间成员授权范围收敛

**关联 PRD**: 需求来源 = 内部需求口径（部门知识空间成员管理范围调整，2026-06）
**优先级**: P1
**所属版本**: v2.6.0
**依赖**: 部门知识空间能力（`DepartmentKnowledgeSpace` 绑定）、F006/F007 ReBAC 资源授权

> **范围边界**
> - **本次纳入**：
>   - 部门知识空间「成员管理 → 新增用户授权」弹窗的可授权范围收敛为「绑定部门 + 其所有子部门」「这些部门下的成员」。
>   - 部门知识空间禁用「用户组」授权维度（前端隐藏 + 后端拒绝）。
>   - 一次性脚本清理部门知识空间历史遗留的用户组授权。
> - **本次明确排除**：
>   - **不改普通知识空间**的成员管理（user / department / user_group 三维度、全租户范围）——逐条回归保证不变。
>   - 不改 platform 端 `PermissionDialog`（只服务 `knowledge_library`）。
>   - 不改部门知识空间创建时「默认授予绑定部门 viewer」的既有逻辑。
>   - 不改成员列表展示 / 角色升降 / 移除逻辑。
>   - 不为部门空间用户组授权保留运行期兼容（历史态由脚本清理，原因：异常/历史态集中清理优于长期背兼容分支）。

---

## 1. 用户故事

作为 **部门知识空间的管理员 / 创建者（含 super_admin）**，
我希望 **新增成员授权时只能选择本部门及子部门、以及其中的成员**，
以便 **部门知识空间的访问授权严格限定在本部门组织范围内，不被跨部门或用户组维度扩散**。

---

## 2. 验收标准

> 错误码 / 实现归属为 How，见 design.md §2 / §3 / §6；本节只写可观测对外行为。

- **AC-01** — WHERE 当前资源是部门知识空间, THE SYSTEM SHALL 在「新增用户授权」弹窗只提供「用户」「部门」两个维度，不展示「用户组」维度（对所有身份生效，含 super_admin）。

- **AC-02** — WHERE 当前资源是部门知识空间, THE SYSTEM SHALL 在「部门」维度只返回「该空间绑定的部门及其所有子部门」，不返回租户内其它部门。

- **AC-03** — WHERE 当前资源是部门知识空间, THE SYSTEM SHALL 在「用户」维度只返回「绑定部门及其所有子部门下的成员」（用户归属其中任一部门即可见）。

- **AC-04** — IF 对部门知识空间提交「用户组」授权、或「用户/部门」对象不在绑定部门子树内, THEN THE SYSTEM SHALL 拒绝该授权并返回权限不足错误（对所有身份生效，super_admin 同样被拒）。

- **AC-05** — THE SYSTEM SHALL 保持普通知识空间的成员授权行为完全不变：仍提供用户/部门/用户组三维度，范围仍为全租户，用户组授权仍可用。

- **AC-06** — WHEN 一次性清理脚本以 `--apply` 执行, THE SYSTEM SHALL 清除所有部门知识空间上的用户组授权；`--dry-run`（默认）只输出受影响的（空间、用户组、波及用户数）清单，不改动任何授权。

---

## 3. 边界情况

- 当绑定部门已归档 / 子树为空时，部门与用户维度返回空列表，提交非空授权被拒，不报错（合理降级）。
- 当用户归属多个部门时，只要任一部门落在绑定部门子树内即视为可见 / 可授权。
- 当对部门空间执行撤销授权（含撤销历史用户组授权）时，范围约束不拦截，便于清理与移除。
- 无论前端如何标记，系统以「空间与部门的绑定关系」为唯一判定依据，列表与授权准入不被绕过。
