# 用户组管理员语义对齐说明

日期：2026-04-17

## 结论

当前共识是：

- 用户组应当支持“独立管理员”
- 用户组的管理员语义应与部门保持一致
- `create_user` 与 `admin` 是两套不同概念，不应混用
- 这一问题当前先不继续修复，先把语义和上下文固化下来，避免后续再沿错误方向继续改

## 概念定义

### 创建者

- 表示“谁创建了该对象”
- 是元数据/审计字段
- 一般只有一个
- 典型字段：`create_user`

### 管理员

- 表示“谁有管理该对象的权限”
- 是权限关系，不是元数据
- 可以有多个
- 典型表现：单独的授权关系或管理员列表

## 部门模块当前语义

部门模块已经明确区分了“创建者”和“管理员”：

- `department.create_user` 记录创建者
- 部门管理员通过独立的 `admin` 关系维护
- 新建部门时可显式设置 `admin_user_ids`
- 权限检查主要依赖“系统管理员”或“部门管理员”关系，而不是 `create_user`

这套模型符合“创建者 != 管理员”的目标语义。

## 用户组模块当前冲突

用户组模块目前存在文档与实现分叉：

### 主 PRD（当前 2.5 管理页方向）

`docs/PRD/2.5 权限管理体系改造 PRD/2.5 权限管理体系改造 PRD.md`

从主 PRD 的“3.2.2 用户组管理”看，页面强调的是：

- 创建者
- 可见范围
- 组内成员

没有把“独立管理员列表/配置项”作为前台能力写出来。

### 旧 Feature Spec（F003）

`features/v2.5.0/003-user-group/spec.md`

旧 spec 中明确包含：

- `admin_user_ids`
- `group_admins`
- `PUT /api/v1/user-groups/{group_id}/admins`
- “设置组管理员”的完整逻辑

也就是说，旧 spec 是支持独立管理员的。

### 当前代码趋势

当前分支中的部分实现已经向“创建者模型”收口，例如：

- 系统页用户组列表显示“创建者”
- 编辑页将创建者作为只读展示
- `UserGroupService.aset_admins()` 当前直接拒绝独立管理员配置

但同时仍有很多旧链路、旧结构、旧字段保留，形成混合状态。

## 暂定产品语义

本次分析后的暂定语义是：

- 用户组应支持独立管理员
- 语义与部门一致
- 创建者不等于管理员，但创建者可以默认成为管理员之一

换句话说，未来用户组应回到下面这套模型：

- `create_user`：谁创建了这个用户组
- `user_group admin`：谁有权限管理这个用户组
- 用户组可以存在多个管理员

## 当前状态说明

虽然目标语义应支持独立管理员，但这一问题当前先暂停，不继续修复。原因是：

- 主 PRD 与旧 F003 spec 冲突
- 当前系统页和后端实现已经部分按“创建者模型”收口
- 如果直接继续沿某一侧推进，容易把另一侧彻底改坏

因此本次先记录语义，不继续在这条线上追加代码修改。

## 后续恢复修复时的原则

后续若恢复这条线，应以“用户组与部门保持一致”为目标，至少满足：

1. 创建者与管理员解耦
2. 创建用户组时允许设置初始管理员
3. 编辑用户组时允许维护管理员集合
4. 所有权限判断统一依赖“管理员关系”，而不是把 `create_user` 当成唯一管理员
5. 旧链路若仍依赖 legacy admin 行，需要明确保留兼容或统一迁移

## 本说明的作用

本文件只用于固定当前共识：

- 不接受“用户组只能有创建者、不能有独立管理员”作为最终产品语义
- 当前代码如果出现与此相反的实现，应视为暂时偏移，而不是最终设计
- 该问题当前暂停，不在本轮继续展开修复
