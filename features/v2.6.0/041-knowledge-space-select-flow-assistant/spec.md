# Feature: F041-knowledge-space-select-flow-assistant（工作流 / 助手应用支持选择知识空间）

> **本文档定位 — 纯 What（需求口径，不随代码漂移）**
> spec 只回答 **做什么 / 验收标准 / 不做什么**；所有 How（数据流、字段、API、Service、前端组件、文件清单）在 [design.md](./design.md) 与 [tasks.md](./tasks.md)。

**关联 PRD**: 首钢合入需求 §6「工作流、助手应用支持选择知识空间」（6.1 选择器新增知识空间 tab / 6.2 高级检索配置正确生效 / 6.3 检索结果格式一致）
**优先级**: P0（涉及知识空间越权读取风险与 INV-7 安全不变量的口径调整，权限过滤正确性为本次「重点点名」项）
**所属版本**: v2.6.0
**模块编码**: 沿用 180（`knowledge_space`）/ 109（`knowledge`）；**不新增错误码**（权限过滤为「静默剔除」语义，无权即空命中，不抛错）
**依赖**:
- **F029**（知识空间 AI 问答 - 检索权限过滤）—— 复用其 `KnowledgeFileVisibilityService`（索引层 `can_read` 粗滤 + 结果层 `view_file` 精滤）与角标溯源 `view_file` 精滤；F029 曾**明确将工作流 KNOWLEDGE_RETRIEVER 节点排除**，理由是「涉及流程节点执行上下文中的『运行用户身份』问题，留待后续 feature」——**本 Feature 即接盘该遗留项**。本 Feature 在其角标溯源精滤上叠加 `accessScope`（`per_user` / `shared`）分级（方案 B），**扩展**而非改写 F029 直接入口语义。
- **F030**（知识资源统一对外 API）—— 复用「知识空间是 `knowledge` 表 `type=3` 行、与知识库同表全局唯一、由后端按 `row.type` 分派」的多态检索范式（`KnowledgeSpaceChatService.aretrieve_chunks` 已多态）。

> **范围边界**（写清「明确不做什么」比「做什么」更能防 scope 膨胀）
>
> - **本次纳入**：

>   - **B. 两个工作流节点重命名**（仅显示名 / i18n）：文档知识库问答节点 → **知识库问答节点**；文档知识库检索节点 → **知识库检索节点**。
>   - **C. 高级检索配置对知识空间生效**：默认元数据过滤（**仅内置字段**）与「用户知识库权限校验」开关在选中知识空间时同样成立；开关**开** = 仅**运行使用者**有 `view_file` 的空间文件进结果，开关**关** = 按**配置者** `view_file` 共享（借用配置者可见范围，见 D / AC-14）。

>   - **F. 角标溯源门禁与检索开关对齐**（方案 B 分级暴露）：给 citation 标注访问模式（`per_user` / `shared`）；开关关闭产生的 `shared` 档 citation 在溯源解析中**不整条剔除**、照常展示来源元数据（文件名 / 知识库名 / snippet / 可点角标），但**完整文件的预览 / 下载 URL 仍按运行使用者 `view_file` 判定**。涉及改动 citation 模块（属 F029 territory），须在契约同步。

>   - **QA 知识库检索节点（`qa_retriever`）** —— 需求未点名；不加知识空间 tab、不改名（QA 检索走单条问答匹配、语义不同）。
>   - **知识空间的自定义元数据字段过滤** —— F030 已确认知识空间无自定义元数据能力；本期仅支持内置字段过滤（见 AC-07）。
>   - **OpenFGA 模型变更** —— 复用现有 `can_read` / `view_file`，不新增 ReBAC 关系（沿用 F029 AD-05）。
>   - **个人库（`type=2`）对外可选** —— 选择器不列出个人库（沿用现状，个人库仅 workstation/linsight 内部使用）。
>   - **匿名 / 分享链路检索** —— 知识空间检索依赖登录身份解析 `view_file`；无登录身份场景（share link）不在本期（沿用 F029 AC-20 口径）。
>   - **对外 RPC `/api/v2/filelib/retrieve` 的改造** —— 该「代用户检索」路径已由 F030 覆盖（`user_id` 协议），本 Feature 不重复改造。
>   - **临时会话文件（`tmp`）tab 的既有行为** —— 不改动，仅新增知识空间 tab 与其检索 / 过滤行为。（注：4 个入口现状均**无 QA 知识库 tab**——助手应用为无 tab 的文档库平铺选择、工作流节点为「文档知识库 / 临时会话文件」两 tab；本期不引入 QA 知识库 tab。）

## 1. 用户故事

**故事 A（配置者 · 选空间）**
作为 **配置助手应用或工作流节点的人**，
我希望 **在知识库选择器里有一个独立的「知识空间」tab，能把我有权查看的知识空间配置为助手 / 节点的检索范围**，
以便 **让助手 / 工作流也能利用知识空间里的内容，而不必先把文件搬进普通知识库**。

**故事 B（运行使用者 · 权限符合预期）**
作为 **实际使用助手 / 触发工作流的终端用户**，
我希望 **当配置者开启「用户知识库权限校验」后，检索结果只包含我本人有『查看』权限的空间文件，无权文件的内容、文件名、来源、角标都不会出现**，
以便 **不会通过助手 / 工作流间接读到我在列表 UI 里本就看不到的内容**。

**故事 C（配置者 · 维持线上行为）**
作为 **已有大量线上助手 / 工作流的配置者**，
我希望 **新增的权限校验开关默认关闭，存量助手 / 节点行为不因本次升级而改变**，
以便 **升级平滑、不产生「原本能查到的内容突然查不到」的回归**。

**故事 D（终端用户 · 溯源无差别）**
作为 **使用助手 / 工作流并查看回答引用的人**，
我希望 **知识空间来源的引用角标、来源卡片、预览跳转与文档知识库来源完全一致**，
以便 **在同一条回答里混合引用知识库与知识空间时，体验和溯源逻辑没有割裂**。

**故事 E（研发）**
作为 **后端研发**，
我希望 **知识空间检索复用 F029 的双层 `view_file` 过滤与统一 Document / citation 产出，按运行使用者身份过滤、按 `row.type` 多态分派**，
以便 **不重复实现两套检索，且『列表 UI 可见性 = 权限校验开启时的 AI 检索可见性』语义与 F029 一致**。

## 2. 验收标准

> AC-ID 在本特性内唯一，格式 `AC-NN`。P0 项用 EARS 句型。
> 「运行使用者」= 实际触发工作流 / 使用助手的登录用户（后端 `self.user_id` / `invoke_user_id`）；「配置者」= 编辑助手 / 工作流配置的登录用户。

### 2.1 选择器新增「知识空间」tab（覆盖 4 入口）— 对应 PRD 6.1

| ID | 角色 | 操作 | 预期结果 |
|----|------|------|---------|
| AC-01 | 配置者 | 打开 4 个入口任一的知识库选择器 | 新增一个独立的「知识空间」tab，与该入口现有的**文档知识库**选择并列（工作流节点另含**临时会话文件** tab；助手应用现为无 tab 的文档库平铺选择，新增后以 tab 形式呈现）。**不含 QA 知识库 tab**（现状即无） |
| AC-02 | 配置者 | 切到「知识空间」tab | 列出的知识空间口径**与 client 日常模式一致** = 配置者「我创建的 + 我加入的 + 部门知识空间」去重并集（`/api/v1/knowledge/space/mine` + `/joined` + `/department`），**不含知识广场（`/square`）**；各端点沿用其现有 ReBAC 过滤与 admin 行为；无权空间不出现 |
| AC-03 | 配置者 | 在「知识空间」tab 选中若干空间并保存，随后重新打开配置回显 | 所选空间被持久化进该入口的检索配置，且**每条选择带资源类型标记**（文档知识库 / 知识空间），回显时据此把每项**归类到正确的 tab**（沿用 client 日常模式 `type: 'space'` 范式；存量未标记项默认按文档知识库兜底）；空间 id 与知识库 id 同表全局唯一，后端检索按 `row.type` 分派 |
| AC-04 | 配置者 | 在**助手应用**（现有跨 tab 多选，选择结果并入同一 `knowledge_list`）同时选择文档知识库与知识空间 | 二者可共存于同一检索范围、检索时都参与、结果合并（多库合并 / 截断沿用现状）。工作流三节点当前选择器为「单一来源类型」（文档库 / 临时文件互斥）——空间与文档库能否在**单个节点内**并存，属选择器交互的 design 决策，本 spec 不强制要求，仅要求各节点可将知识空间选为检索范围 |

- **AC-05** — THE SYSTEM SHALL 按 client 日常模式方式加载知识空间列表（`/space/mine` + `/joined` + `/department` 三端点去重并集、一次性加载 + **前端名称搜索**），不套用文档知识库 tab 的游标分页；列表口径见 AC-02。

### 2.2 节点重命名 — 对应 PRD 6.1.2

- **AC-06** — THE SYSTEM SHALL 将「文档知识库问答」节点显示名改为「知识库问答」，「文档知识库检索」节点显示名改为「知识库检索」，覆盖中文 / 英文 / 日文三语。
- **AC-07** — THE SYSTEM SHALL 保持两节点的内部类型标识与存量流程数据不变（重命名仅作用于显示名，已保存的工作流不受影响、无需迁移）。

### 2.3 高级检索配置对知识空间生效（本次重点）— 对应 PRD 6.2

- **AC-08** — WHERE 知识库问答节点 / 知识库检索节点选中了知识空间, THE SYSTEM SHALL 让「高级检索配置」中的「默认元数据过滤」对空间文件生效，支持内置字段（`document_id` / `document_name` / `upload_time` / `update_time` / `uploader` / `updater`）。
- **AC-09** — THE SYSTEM SHALL 不为知识空间提供自定义元数据字段过滤（知识空间无自定义元数据能力；仅内置字段可用）。
- **AC-10** — THE SYSTEM SHALL 让 4 个入口均具备「用户知识库权限校验」开关；其中助手应用配置与工作流助手节点为**新增**开关，并在开关旁提供 tips 文案说明两档语义（开 = 按使用者本人权限过滤、关 = 按配置者可见范围共享；文案见本节末尾），覆盖中 / 英 / 日。
- **AC-11** — THE SYSTEM SHALL 让**新增**开关（助手应用、助手节点）默认为**关闭**，与现有知识库问答 / 检索节点的默认（`user_auth=false`）一致，保证存量助手 / 节点行为不变（关档对知识空间的语义见 AC-14——按配置者可见范围，安全有界，**非**"完全不过滤"）。
- **AC-12** — WHERE 「用户知识库权限校验」开启 AND 检索范围含知识空间, THE SYSTEM SHALL 仅让**运行使用者**有 `view_file` 权限的、解析成功的主版本空间文件进入检索结果（复用 F029 索引层粗滤 + 结果层精滤双层过滤）。
- **AC-13** — WHERE 「用户知识库权限校验」开启, THE SYSTEM SHALL 保证运行使用者无权空间文件的 chunk、文件名、来源、预览链接、角标溯源均不出现在模型上下文与回答中（对齐 INV-7 语义）。
- **AC-14** — WHERE 「用户知识库权限校验」关闭 AND 检索范围含知识空间, THE SYSTEM SHALL 按**配置者**（该助手 / 工作流配置的创建者）的 `view_file` 过滤空间文件——即所有运行使用者共享「配置者可见范围」，检索**永不越过配置者本人的可见边界**（连配置者都无权查看的文件不会被检索）；文档知识库侧沿用既有 `check_auth=false`（整库检索，配置者本就拥有整库权限，二者语义对称）。
- **AC-15** — WHEN 管理员在两次检索之间授予 / 收回（开档下的运行使用者 / 关档下的配置者）对空间文件的 `view_file`, THE SYSTEM SHALL 在下一次检索使用最新权限（沿用 F029 权限缓存 TTL ≤ 10s 或 `invalidate_user` 失效）。
- **AC-16** — IF 检索身份（开档 = 运行使用者 / 关档 = 配置者）对所选空间无任何 `view_file` 文件, THEN THE SYSTEM SHALL 对该空间返回空命中、不报错、不中断对话（模型按现有 prompt 应答）。

> **开关 tips 文案（AC-10 要求，随本 spec 交付；中文基准，英 / 日在实现期补 i18n）**
> - **开关名**：用户知识库权限校验
> - **Tips**：「开启后，检索结果按**当前使用者本人**的知识空间文件查看权限过滤，每人只能检索到自己有权查看的文件；关闭时，按**配置者**的可见范围检索，所有使用者共享配置者能看到的文件（配置者无权查看的文件同样不会被检索）。文档知识库不受细粒度文件权限影响。」

### 2.4 检索结果格式与角标溯源一致 — 对应 PRD 6.3

- **AC-17** — THE SYSTEM SHALL 让知识空间的检索结果与文档知识库返回同构结构（相同的 chunk / document 元数据字段：`knowledge_id` / `document_id` / `document_name` / `chunk_index` / 时间 / 上传者等），使助手与工作流在使用与展示时无感知差异。
- **AC-18** — THE SYSTEM SHALL 让知识空间检索命中的文件复用统一角标溯源（citation registry / `annotate_rag_documents_with_citations`），citation 结构（`citationId` / `type=rag` / `sourcePayload`）与文档知识库一致，可复用角标溯源解析接口。
- **AC-19** — THE SYSTEM SHALL 为每条 RAG citation 标注访问模式：直接空间问答（F029）及「用户知识库权限校验」开启的工作流 / 助手来源标为 `per_user`；开关关闭的工作流 / 助手来源标为 `shared`（存量 / 未标注 citation 默认按 `per_user` 从严处理）。

### 2.5 角标溯源门禁与检索开关对齐（方案 B）— 对应 PRD 6.2 / 6.3

> 溯源期门禁必须与检索期门禁一致：`per_user` 保持 F029 严格精滤；`shared` 不整条剔除，但完整文件预览 / 下载仍按运行使用者 `view_file` 把关（分级暴露）。

- **AC-20** — WHERE citation 为 `per_user`, THE SYSTEM SHALL 在角标溯源解析（`citations/resolve` 批量与单条）时按运行使用者 `view_file` 精滤，无权则整条剔除（F029 现有行为完全不变）。
- **AC-21** — WHERE citation 为 `shared` AND 运行使用者对该文件无 `view_file`, THE SYSTEM SHALL 在溯源中保留该条、返回来源元数据（文件名 / 知识库名 / snippet / 可点角标），但**不返回**完整文件的 `previewUrl` / `downloadUrl`；单条 `resolve` 此时**不再抛 `NotFoundError`**。
- **AC-22** — WHERE citation 为 `shared` AND 运行使用者对该文件有 `view_file`（或为 admin）, THE SYSTEM SHALL 返回完整溯源（含 `previewUrl` / `downloadUrl`），与 `per_user` 有权时一致。

### 2.6 混合与边界行为

- **AC-23** — WHERE 检索范围同时含知识空间与文档知识库（助手应用，或跨节点组合）, THE SYSTEM SHALL 分别按各自路径检索后合并结果；开启权限校验时空间侧按运行使用者 `view_file` 过滤，文档库侧沿用其既有 `check_auth` 行为。
- **AC-24** — WHERE 运行使用者是管理员 / super_admin, THE SYSTEM SHALL 跳过 `view_file` 过滤（`list_accessible_ids` 对 admin 短路返回 None，沿用现状）。
- **AC-25** — IF 选中的知识空间被删除 / 空间内文件处于解析中 / 解析失败, THEN THE SYSTEM SHALL 沿用现有过滤（非主版本、解析状态过滤），不将其纳入检索、不报错。

## 3. 边界情况

- **配置者可见但运行使用者不可见的空间**：权限校验**开启**时按运行使用者过滤（其无权则空命中，AC-16）；**关闭**时按**配置者** `view_file` 检索（运行使用者共享配置者可见范围，AC-14）。两档都**不越过配置者本人的可见边界**——这是本次开关口径的核心表现，须与需求方对齐预期。
- **配置者对所选空间已无任何 `view_file`（边角：配置者事后被收回权限）**：关档检索为空命中、不报错（与 AC-16 对称，检索身份此时为配置者）。
- **角标溯源与开关的交互（方案 B）**：检索期门禁由开关控制（开 = 运行使用者 `view_file`，关 = 配置者 `view_file`，AC-12/AC-14）；溯源期按 citation 的访问模式**对齐**检索期门禁——`per_user`（直接空间问答 / 开关 ON）照 F029 按**运行使用者** `view_file` 整条剔除；`shared`（开关 OFF，进入答案的内容已收敛到**配置者可见范围**）不剔除、展示来源元数据与 snippet，但完整文件的预览 / 下载 URL 仍按**运行使用者** `view_file` 判定（AC-19~22）。即「开关 OFF 时，答案用了配置者可见的内容、溯源看得到是哪个文件与片段，但运行使用者无 `view_file` 时打不开完整原文件」——既不断裂、又不越配置者边界、也不把整份文件泄露给无权使用者。
- **空间内无任何可见文件 / 交集为空**（校验开启 + 元数据过滤 / 标签过滤后为空）：返回空命中 + 正常应答，不抛错（沿用 F029 空集语义）。
- **权限实时性**：沿用 F029 缓存（TTL ≤ 10s / `invalidate_user`），不引入本特性专属缓存。
- **多租户**：空间列表与检索均在当前租户内解析，沿用现有租户隔离，不跨租户。
- **INV-7 口径调整（须在契约登记，编排者所需的例外）**：本 Feature 使「知识空间 AI 检索可见性 ⊆ 列表 UI 可见性」在**工作流 / 助手入口**成为**开关可选**（非强制），需在 `release-contract.md` 的 INV-7 追加例外说明（形式类比 F040 对 INV-6 的例外登记）。例外须**同时覆盖检索侧与溯源侧**：开关 OFF 时按**配置者** `view_file` 检索（共享配置者可见范围、不越其边界），该来源 citation 标为 `shared`——溯源期不整条剔除且返回来源元数据，但完整文件预览 / 下载仍按**运行使用者** `view_file` 把关；开关 ON（按运行使用者 `view_file`）及直接空间 AI 问答入口（F029）维持强制语义完全不变。
