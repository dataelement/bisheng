# 用户组管理员现状说明

日期：2026-04-17

## 目的

本文件只记录当前仓库中的“文档、实现、遗留结构”在用户组管理员语义上的现状。

本文件不定义未来产品结论，也不替代 PRD。

## 基本概念

### 创建者

- 表示“谁创建了该对象”
- 通常落在元数据字段中
- 典型字段：`create_user`

### 管理员

- 表示“谁有权限管理该对象”
- 通常是一条单独的权限关系或管理员集合
- 可以与创建者重合，也可以不重合

## 部门模块当前状态

部门模块当前对“创建者”和“管理员”是分离处理的：

- `department.create_user` 记录创建者
- 部门管理员通过单独的 `admin` 关系维护
- 新建部门时可传 `admin_user_ids`
- 权限检查依赖“系统管理员”或“部门管理员”关系，而不是 `create_user`

因此，部门模块当前语义中：

- 创建者 = 谁创建了部门
- 管理员 = 谁能管理部门

两者不是同一个概念。

## 用户组模块相关文档现状

### 主 PRD（2.5 权限管理体系改造 PRD）

在主 PRD 的“3.2.2 用户组管理”部分：

- 用户组列表强调的是可见性、创建者、成员
- 新建/编辑用户组页面描述中没有单独列出“管理员配置项”
- 删除权限描述和“创建者”直接相关

从该章节文字看，用户组前台管理更偏向“创建者 + 成员”的表达。

### F003 用户组 Feature Spec

`features/v2.5.0/003-user-group/spec.md` 中仍保留了更早的一套定义：

- `admin_user_ids`
- `group_admins`
- `PUT /api/v1/user-groups/{group_id}/admins`
- Service 逻辑里创建用户组时可设置初始 admins

从 F003 spec 看，用户组是支持“独立管理员”的。

### 文档冲突点

当前仓库中的文档并不完全一致：

- 主 PRD 没有把“独立管理员”作为用户组前台能力展开描述
- F003 spec 明确保留了“独立管理员”能力

因此，单看文档，用户组管理员语义并未完全收口。

## 用户组模块当前实现状态

### 前端系统管理页

当前系统管理页中的用户组 UI 主要按“创建者模型”展示：

- 列表列名是“创建者”
- 编辑页显示只读“创建者”
- 页面上没有独立管理员编辑入口

也就是说，当前前端交互层没有暴露“单独维护多个管理员”的能力。

### 后端服务层

当前后端服务层处于混合状态：

- `create_user` 仍然存在，用于记录创建者
- `group_admins` / `admin_user_ids` 相关结构仍未完全移除
- `UserGroupService.aset_admins()` 当前直接返回“不支持独立管理员”的错误
- 新建用户组时，当前代码会为创建者写入兼容性的 legacy admin 行，并同时写入 FGA admin 关系
- 当前创建接口 schema 仍保留 `admin_user_ids`
- 当前 service 对“额外的 admin_user_ids”做显式拒绝，而不是静默忽略

### 旧链路兼容状态

仓库中仍有若干旧链路会从 legacy admin 行读取“用户组管理员”：

- `/user/info` 中的 `admin_groups`
- 若干 auth / audit / legacy service 逻辑

因此，当前仓库不是一个纯粹的“只有创建者，没有管理员”的实现。

更准确地说，当前状态是：

- 用户组前台交互正朝“创建者模型”收口
- 后端和历史链路中仍保留管理员结构与兼容逻辑

## 当前仓库可观察到的语义

按现状看，用户组语义并不是单一模型，而是混合的：

### 已明确存在的事实

- 用户组有创建者（`create_user`）
- 用户组历史上有管理员结构（`group_admins` / `admin_user_ids` / legacy admin 行）
- 当前系统管理页主要展示创建者，不暴露独立管理员编辑
- 当前后端没有继续支持“通过新 API 配置额外管理员”

### 尚未统一的问题

- 用户组是否应与部门完全一致，继续支持“创建者 != 管理员”
- 还是应整体收口成“创建者主导”的模型
- 主 PRD 与 F003 spec 哪一份应作为这一块的最终实现约束

## 本文件的边界

本文件只说明“当前状态是什么”：

- 文档有冲突
- 实现是混合状态
- 前端、后端、遗留链路尚未完全统一

本文件不表达以下结论：

- 不声明“用户组最终必须支持独立管理员”
- 不声明“用户组最终必须只有创建者”
- 不作为后续实现方向的最终裁决

如果后续要继续修复这一块，应先明确采用哪一份产品语义作为准绳，再继续代码变更。
