## 1. 背景和基础概念

### 1.1 背景

* 中粮：支持文件夹级别的精细化权限隔离，可设置为继承上级的成员默认可读或手动圈定的指定成员可读。

* 首钢：作为底层基础设施支持知识门户功能；支持文件夹级别的知识空间协作管理；支持知识空间内的文件移动。

## 3. 详细设计

### 3.1 新权限体系设计综述

```yaml
model
  schema 1.1

# ============================================
# 基础类型
# ============================================
type user

# ============================================
# 系统超级管理员短路逻辑：权限引擎第一步就是校验此逻辑，直接放行。
# system 不是一个业务资源，唯一用途就是标记谁是超级管理员。        
# ============================================
type system
  relations
    define super_admin: [user]

# ============================================
# 部门 — 严格树形组织架构
# ============================================
type department
  relations
    define parent: [department]
    define admin: [user] or admin from parent        # 部门管理员权限向下传递
    define member: [user]                             # 成员仅限直接隶属，不自动继承

# ============================================
# 用户组 — 跨部门临时项目组
# ============================================
type user_group
  relations
    define admin: [user]
    define member: [user]

# ============================================
# 知识空间
# ============================================
type knowledge_space
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 文件夹 — 支持嵌套，权限从父级继承
# ============================================
type folder
  relations
    define parent: [knowledge_space, folder]          # 父级可以是空间或文件夹
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner or can_manage from parent
    define editor: [user, department#member, user_group#member] or manager or can_edit from parent
    define viewer: [user, department#member, user_group#member] or editor or can_read from parent
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner or can_manage from parent

# ============================================
# 知识文件 — 权限继承自父文件夹/空间
# ============================================
type knowledge_file
  relations
    define parent: [folder, knowledge_space]
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner or can_manage from parent
    define editor: [user, department#member, user_group#member] or manager or can_edit from parent
    define viewer: [user, department#member, user_group#member] or editor or can_read from parent
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner or can_manage from parent

# ============================================
# 工作流
# ============================================
type workflow
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 助手
# ============================================
type assistant
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 工具
# ============================================
type tool
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 频道
# ============================================
type channel
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner

# ============================================
# 看板
# ============================================
type dashboard
  relations
    define owner: [user]
    define manager: [user, department#member, user_group#member] or owner
    define editor: [user, department#member, user_group#member] or manager
    define viewer: [user, department#member, user_group#member] or editor
    define can_manage: manager
    define can_edit: editor
    define can_read: viewer
    define can_delete: owner
```

> **Tenant 树补充说明**（2026-04-18 更新）：
>
> v2.5 多租户改版后，权限模型额外引入 `type tenant`：
>
> ```yaml
> type tenant
>   relations
>     define admin: [user]              # Child Tenant admin；Root 不写；不沿 parent 继承
>     define member: [user]             # Tenant 归属反查；不进资源授权
>     define shared_to: [tenant]        # Root → Child 显式共享分发
> ```
>
> **2026-04-20/21 收窄**：删除 `tenant#parent` 关系（FGA 冗余，依赖 MySQL `parent_tenant_id` 字段）。所有资源类型 `viewer` 仅新增 `tenant#shared_to#member`（显式共享）；`manager` / `editor` **不**引入 `tenant#` 任何关系——资源授权回归 `owner + user + department#member + user_group#member` 四源。Tenant 作为"配额/存储/归属边界"不参与资源级授权。
>
> 完整 DSL 见《多租户需求文档》附录 A。**管理员模型**为两层：全局超管（`system:global#super_admin`）+ Child Admin（`tenant:{child_id}#admin`），原独立的 "Root Admin" 已合并到全局超管；**Root Tenant 不写 `tenant:1#admin` 元组**。

**知识文件级权限对应表**

文件（`knowledge_file`）的权限从父级（文件夹或知识空间）继承，也可单独授权覆盖。取并集：

| 关系 | 查看文件内容 | 编辑文件（重命名/替换/标签） | 删除文件 |
|------|-------------|--------------------------|---------|
| **can_read**（直接授权或继承自父级） | ☑️ | 🔲 | 🔲 |
| **can_edit**（直接授权或继承自父级） | ☑️ | ☑️ | 🔲 |
| **owner**（文件上传者） | ☑️ | ☑️ | ☑️ |
| **can_manage from parent**（父级的 manager 及以上） | ☑️ | ☑️ | ☑️ |

> **继承规则**：文件夹/空间级的 can_read/can_edit/can_manage 自动传递到其下所有文件。如果文件有单独的直接授权，取直接授权与继承权限的并集。

#### 权限检查流程（五级短路）

> **2026-04-18 PRD Review 更新**：管理员模型从"三层"简化为"两层"（全局超管 + Child Admin），原独立的 "Root 管理员" 合并到全局超管。因此 ③ 阶段仅检查 Child Tenant admin，**不再沿 `parent` 继承**；Root 级资源的管理权由 ① 全局超管统一短路。

```
请求进入
  ↓
① 全局超管检查：system:global#super_admin → 全部放行
    （合并原 Root Admin 职责；作用域含所在 Root 及其所有 Child）
  ↓ （非全局超管）
② 租户归属检查（IN 列表）：资源 tenant_id 是否 ∈ {用户叶子 Tenant + Root Tenant}（2 层下至多 2 个值）
    或资源对该集合中任一 tenant 开放共享（shared_to）→ 不属于则拒绝
  ↓ （归属通过）
③ Child Tenant admin 检查：FGA check(user, admin, tenant:{resource_tenant_id})
    仅作用于 Child Tenant 内部资源；不沿 parent 继承
  ↓ （非 Child admin）
④ ReBAC 权限检查：OpenFGA Check（按资源类型查询 can_read/can_edit/can_manage 等关系）
  ↓
⑤ RBAC 菜单/配额检查：WEB_MENU 控制前端导航可见性，角色级配额控制资源创建上限
```

**关键变化**（对比 v2.5 早期版本）：

- ② 从"严格相等"改为"IN 列表"，2 层下可见集合 = {leaf, Root}（集团共享资源场景需要）
- ③ 去掉"Root admin 继承 Child admin"能力；OpenFGA DSL 中 `tenant.admin` 不再使用 `or admin from parent`
- ① 全局超管等价承担原 Root admin 的全树可见性（§3.3 管理员身份对照表）
- **2026-04-20 收窄**：本实例固定单 Root，删除所有 SaaS 多客户 / 跨 Root 运维相关场景；Root Tenant 由迁移脚本自动创建、不可删除/禁用
- **2026-04-21 收窄**：DSL 资源 manager/editor/viewer 移除 `tenant#member`；Tenant 不参与资源级授权

> **多租户关闭时**（`multi_tenant.enabled = false`），步骤 ②③ 自动跳过，流程等效为 ①→④→⑤。
> Tenant 树架构的权限检查完整描述详见《多租户需求文档 §3.1-§3.3》。

#### 隐式管理范围（不占「当前权限」名额）

以下关系**不写入 OpenFGA**、**不出现在资源「当前权限」列表**，由服务端在鉴权时按组织关系计算；随**岗位**变化而自动生效/失效（非绑定到某个自然人账号的历史元组）。

| 角色 | 资源范围 | 等效档位 | 说明 |
|------|----------|----------|------|
| **超级管理员** | 租户内应用（`workflow` / `assistant`）、知识库（`knowledge_space`） | **所有者**（`owner`） | 与既有「超管短路」一致；不在授权弹窗中展示为一条可编辑/可删除的授权。 |
| **部门管理员** | 其所辖部门及**下级部门**内成员创建的上述资源 | **可管理**（与关系模型 `manager` / FGA `can_manage` 同级） | 判定：资源创建者的**任职部门**（`user_department`）落在该部门管理员管辖子树内（含上级管下级的场景）。不在授权弹窗中展示。 |

隐式档位**不**授予 `owner`（非资源创建者）、**不**单独放开 `can_delete`（删除仍以显式 FGA/业务规则为准，避免误删他人资源）。列表查询（`list_objects`）对部门管理员与 OpenFGA 结果取**并集**，以便在构建页/知识列表中看到管辖范围内成员创建的资源。

### 3.4 版本升级数据迁移逻辑

> 2.4 → 2.5 为大版本升级，权限体系从 RBAC 迁移到 RBAC + ReBAC（OpenFGA）。迁移采用**一次性升级脚本 + 验证对账**策略，确保权限零丢失后直接切换到 OpenFGA。

#### 3.4.1 原有授权数据迁移方案

##### 组织结构迁移

| 现有数据 | 迁移目标 | 迁移规则 |
|----------|----------|----------|
| `group` 表（用户组） | 保留，语义正式改为"用户组"（跨部门临时组） | group 表数据保持不变，`department` 表从零开始建（通过第三方同步或手动创建） |
| `user_group` 表（用户-组关联） | OpenFGA `user_group#member` 元组 | `is_group_admin=true` → `user_group` 的 `admin` 关系；`is_group_admin=false` → `member` 关系 |
| 管理员用户（role_id=1） | OpenFGA `system:global` 的 `super_admin` 关系 | 所有拥有 AdminRole 的用户写入 `(system:global, super_admin, user:X)` 元组 |

##### 资源授权数据迁移

| 现有数据 | 迁移目标 | 迁移规则 |
|----------|----------|----------|
| `role_access`（type=WEB_MENU） | **不迁移，保留在 MySQL** | 菜单权限继续由 RBAC 管理 |
| `role_access`（type=KNOWLEDGE） | OpenFGA `knowledge_space` 的 `viewer` 关系 | 展开为每个拥有该角色的用户的直接元组 |
| `role_access`（type=KNOWLEDGE_WRITE） | OpenFGA `knowledge_space` 的 `editor` 关系 | 同上 |
| `role_access`（type=WORKFLOW） | OpenFGA `workflow` 的 `viewer` 关系 | 同上 |
| `role_access`（type=WORKFLOW_WRITE） | OpenFGA `workflow` 的 `editor` 关系 | 同上 |
| `role_access`（type=ASSISTANT_READ） | OpenFGA `assistant` 的 `viewer` 关系 | 同上 |
| `role_access`（type=ASSISTANT_WRITE） | OpenFGA `assistant` 的 `editor` 关系 | 同上 |
| `role_access`（type=GPTS_TOOL_READ） | OpenFGA `tool` 的 `viewer` 关系 | 同上 |
| `role_access`（type=GPTS_TOOL_WRITE） | OpenFGA `tool` 的 `editor` 关系 | 同上 |
| `role_access`（type=DASHBOARD） | OpenFGA `dashboard` 的 `viewer` 关系 | 同上 |
| `role_access`（type=DASHBOARD_WRITE） | OpenFGA `dashboard` 的 `editor` 关系 | 同上 |

> **展开逻辑说明**：现有 `role_access` 是"角色级"授权（一条记录覆盖所有拥有该角色的用户），OpenFGA 是"主体级"授权。迁移时需要将每条 `role_access` 展开为 N 条用户直接元组（N = 拥有该角色的用户数）。

##### 知识空间/频道成员迁移

| 现有数据 | 迁移目标 | 迁移规则 |
|----------|----------|----------|
| `space_channel_member`（business_type=space, user_role=creator） | OpenFGA `knowledge_space` 的 `owner` 关系 | 直接映射 |
| `space_channel_member`（business_type=space, user_role=admin） | OpenFGA `knowledge_space` 的 `manager` 关系 | 直接映射 |
| `space_channel_member`（business_type=space, user_role=member） | OpenFGA `knowledge_space` 的 `viewer` 关系 | 直接映射 |
| `space_channel_member`（business_type=channel） | OpenFGA `channel` 的对应关系 | 同上规则，`creator→owner, admin→manager, member→viewer` |
| 仅迁移 `status=ACTIVE` 的记录 | — | PENDING/REJECTED 状态的记录不迁移 |

##### 组资源关系迁移

`group_resource` 表中的资源关联关系，按权限级别映射到 OpenFGA：

| 现有数据 | 迁移目标 | 迁移规则 |
|----------|----------|----------|
| `group_resource`（读权限） | 对应资源类型的 `viewer` 关系 | 展开为 `user_group#member` 的 `viewer` 元组 |
| `group_resource`（写权限） | 对应资源类型的 `editor` 关系 | 展开为 `user_group#member` 的 `editor` 元组 |
| `group_resource`（owner 权限） | 对应资源类型的 `owner` 关系 | 直接映射为用户的 `owner` 元组 |

> **说明**：需要根据 `group_resource.type` 映射到对应的 OpenFGA 资源类型（KNOWLEDGE→knowledge_space, WORKFLOW→workflow, ASSISTANT→assistant, GPTS_TOOL→tool, DASHBOARD→dashboard 等）。

##### 资源所有权迁移

所有资源表中的 `user_id` 字段，迁移为 OpenFGA 的 `owner` 关系：

| 资源表 | OpenFGA 对象类型 | 迁移规则 |
|--------|-----------------|----------|
| `knowledge`（知识库/知识空间） | `knowledge_space` | `(knowledge_space:{id}, owner, user:{user_id})` |
| `flow`（工作流/助手） | `workflow` 或 `assistant`（按 `flow_type` 区分） | `({type}:{id}, owner, user:{user_id})` |
| `gpts_tools`（工具） | `tool` | `(tool:{id}, owner, user:{user_id})` |
| `channel`（频道） | `channel` | `(channel:{id}, owner, user:{user_id})` |

##### 文件夹层级关系迁移

知识空间内的文件夹和文件的父子关系，需要写入 OpenFGA 以支持权限继承：

| 数据 | OpenFGA 元组 |
|------|-------------|
| 顶层文件夹（直属空间） | `(folder:{folder_id}, parent, knowledge_space:{space_id})` |
| 子文件夹 | `(folder:{child_id}, parent, folder:{parent_id})` |
| 文件（在文件夹内） | `(knowledge_file:{file_id}, parent, folder:{folder_id})` |
| 文件（直属空间） | `(knowledge_file:{file_id}, parent, knowledge_space:{space_id})` |

##### 旧表处理

迁移完成后，以下表的授权相关记录标记为废弃（不立即删除，保留观察期）：

- `role_access`：保留 `type=WEB_MENU(99)` 的记录，其余标记废弃
- `group_resource`：全部标记废弃
- `space_channel_member`：全部标记废弃
- `user_group`：成员关系已迁移到 OpenFGA，可在稳定后废弃

##### 迁移策略

1. **Phase 1**（升级脚本）：执行一次性数据迁移，将所有授权数据写入 OpenFGA 元组
2. **Phase 2**（验证对账）：迁移完成后，自动对比旧表和 OpenFGA 的授权结果，确保每个用户对每个资源的权限一致，零丢失后方可继续
3. **Phase 3**（上线）：权限检查仅走 OpenFGA，旧表授权记录标记废弃。保留旧表数据作为回溯参考，不立即删除

#### 3.4.2 看板数据迁移方案

看板（Dashboard）的迁移与其他资源一致：

1. **看板所有权**：`dashboard` 表中每条记录的创建者，写入 `(dashboard:{id}, owner, user:{user_id})` 元组
2. **看板读权限**：现有 `role_access`（type=DASHBOARD, 11）记录，展开为每个用户的 `viewer` 关系
3. **看板写权限**：现有 `role_access`（type=DASHBOARD_WRITE, 12）记录，展开为每个用户的 `editor` 关系
4. **组内看板**：`group_resource`（type=DASHBOARD, 6）记录，按上述组资源关系迁移规则（读→viewer、写→editor、owner→owner）映射到 OpenFGA

> **注意**：看板的分享功能（`from_share` 参数）在新体系中由 `can_read` 权限统一管理，不再需要单独的分享标记。

#### 3.4.3 流量控制迁移逻辑

现有流量控制机制与权限体系是独立的（主要通过应用级别的配置控制）。2.5 版本将流量控制与 ReBAC 授权集成：

1. **现有流量控制数据**：保留现有流量控制配置不变
2. **新增流量控制维度**：在应用授权管理页面中，对被授权的主体（部门/用户组/用户）可以单独设置流量上限
3. **迁移规则**：
   - 现有全局流量控制配置 → 保留，作为默认值
   - 现有按用户组的流量控制 → 迁移为按 user_group 的流量配置
   - 新增按 department 的流量控制配置
4. **配额与流量的区分**：
   - **配额**（quota）：角色级，控制"能创建多少个"资源，由 `role.quota_config` 管理
   - **流量**（rate limit）：资源级，控制"能调用多少次"某个应用，由应用的授权管理页面配置

> 详细的流量控制数据模型见技术方案文档。
