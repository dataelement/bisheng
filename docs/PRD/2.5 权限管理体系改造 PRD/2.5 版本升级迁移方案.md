# BiSheng 2.5 版本升级迁移方案

> 本文档是《2.5 权限管理体系改造 PRD》的增量补充，专门描述从 v2.3/v2.4 升级到 v2.5 的数据迁移逻辑。原 PRD §3.4 定义了迁移映射规则，本文档补充升级的完整流程、用户体验、验证策略和异常处理。

---

## 1. 背景与目标

### 1.1 背景

BiSheng v2.5 进行了两项架构级改造：

1. **权限体系升级**：从 RBAC（基于角色的访问控制）迁移到 ReBAC（基于关系的访问控制），引入 OpenFGA 作为权限引擎
2. **多租户支持**：引入租户（Tenant）概念，所有业务数据增加租户隔离

这两项改造涉及数据库结构变更和大量存量数据迁移。本文档定义升级过程中数据如何安全迁移、如何验证正确性、以及出现异常时如何处理。

### 1.2 升级目标

| 目标 | 优先级 | 说明 |
|------|--------|------|
| 权限零丢失 | **P0** | 升级前用户能访问的资源，升级后必须仍能访问，不得出现权限丢失 |
| 业务连续性 | **P0** | 升级后所有功能正常可用，无需用户手动调整 |
| 向后兼容 | **P0** | 单租户部署升级后行为与 v2.4 一致，用户无感知 |
| 可验证 | **P1** | 提供升级前预览和升级后校验能力，运维可确认迁移结果 |
| 可恢复 | **P1** | 升级失败时可回退到 v2.4 状态 |

### 1.3 适用范围

| 升级路径 | 是否支持 | 说明 |
|----------|---------|------|
| v2.4 → v2.5 | 支持 | 主要升级路径，本文档重点描述 |
| v2.3 → v2.5 | 支持 | 先执行 v2.3 基线迁移，再执行 v2.5 迁移 |
| 全新安装 v2.5 | 支持 | 无存量数据，仅初始化默认数据，跳过权限迁移 |

## 2. 升级全景

### 2.1 升级前后对比

#### 用户和组织

| 维度 | 升级前 (v2.4) | 升级后 (v2.5) |
|------|---------------|---------------|
| 用户归属 | 用户直接属于系统 | 用户归入「默认租户」，通过 user_tenant 关联 |
| 组织架构 | 无部门概念 | 自动创建根部门「Default Organization」，后续可手动或同步建子部门 |
| 用户组 | group 表，全局唯一命名 | 保留，改为租户内唯一命名，增加公开/私密属性（默认公开） |
| 管理员 | role_id=1 的用户 | 同时写入 OpenFGA `super_admin` 关系 |

#### 权限体系

| 维度 | 升级前 (v2.4) | 升级后 (v2.5) |
|------|---------------|---------------|
| 资源授权 | role_access 表（角色级间接授权） | OpenFGA 元组（主体级直接授权） |
| 知识空间/频道成员 | space_channel_member 表 | OpenFGA 元组 |
| 菜单权限 | role_access (type=WEB_MENU) | **不变**，继续由 role_access 管理 |
| 资源所有权 | 各资源表的 user_id 字段 | 同时写入 OpenFGA owner 元组 |
| 文件夹权限继承 | 无（文件夹无独立权限） | OpenFGA parent 关系，支持权限继承 |
| 权限层级 | 读/写 两级 | owner/manager/editor/viewer 四级 |

#### 角色

| 维度 | 升级前 (v2.4) | 升级后 (v2.5) |
|------|---------------|---------------|
| 角色作用域 | 全局 | 增加 role_type（全局/租户级），可绑定部门作用域 |
| 内置角色 | 系统管理员(1)、普通用户(2) | 标记为 `role_type=global`（全局角色） |
| 配额管理 | knowledge_space_file_limit 字段 | 统一迁入 quota_config (JSON)，支持多种配额 |

### 2.2 升级整体流程

从运维视角，升级分为三个阶段：

```
阶段一：停服 + 数据库结构升级
├── 停止所有 BiSheng 服务
├── 备份数据库
└── 执行数据库结构迁移（自动）
    ├── 创建新表（租户、用户租户关联、部门、权限补偿队列、组织同步配置等）
    ├── 48 张业务表添加 tenant_id 字段
    ├── 角色表增加作用域和配额字段
    ├── 用户组表增加可见性字段和租户唯一约束
    └── 种子数据写入（默认租户、用户-租户关联回填）

阶段二：启动服务 + 数据迁移
├── 启动 API 服务（自动触发以下步骤）
│   ├── 初始化默认租户和根部门（幂等）
│   ├── 初始化默认用户组属性（幂等）
│   └── 一次性权限迁移：旧表数据 → OpenFGA 元组
└── 启动后台 Worker

阶段三：验证 + 上线
├── 运行权限校验（自动或手动）
├── 确认功能正常
└── 正式上线
```

### 2.3 用户体验影响

#### 升级过程中

- **服务不可用**：升级期间需停服，用户无法访问系统
- **预计停服时间**：取决于数据量，典型场景（万级用户 + 万级资源）约 5-15 分钟

#### 升级完成后

| 用户角色 | 体验变化 |
|----------|---------|
| 普通用户 | **无感知**。能访问的资源与升级前完全一致，操作流程不变 |
| 系统管理员 | 后台新增「组织与成员」「用户组管理」功能模块，原有功能不变 |
| API 调用方 | API 接口保持兼容，无需修改调用方式 |

## 3. 多租户初始化

### 3.1 默认租户创建

升级时自动创建默认租户，确保存量数据平滑过渡：

| 项目 | 值 |
|------|---|
| 租户 ID | 1（固定值） |
| 租户编码 | default |
| 租户名称 | Default Tenant |
| 状态 | active |

**用户归属处理**：所有现有用户自动关联到默认租户，标记为该用户的默认租户。升级后用户登录时无需选择租户，直接进入默认租户，体验与 v2.4 一致。

### 3.2 根部门创建

为默认租户自动创建根部门：

| 项目 | 值 |
|------|---|
| 部门 ID | BS@root |
| 部门名称 | Default Organization |
| 上级部门 | 无（根节点） |
| 所属租户 | 默认租户 (tenant_id=1) |

> **说明**：根部门创建后，组织架构树为空（仅有根节点）。管理员可在升级后通过「组织与成员」功能手动创建子部门，或通过第三方同步导入组织架构。**现有用户不会自动挂到根部门下**，需管理员后续手动调整或通过组织同步完成。

### 3.3 业务表 tenant_id 回填

所有业务表（共 48 张）新增 `tenant_id` 字段，存量记录的 tenant_id 统一设为 `1`（默认租户）。

这意味着：
- 升级后所有存量数据归属默认租户
- 单租户模式下，系统行为与 v2.4 完全一致
- 后续开启多租户后，可将资源迁移到其他租户

### 3.4 角色与用户组调整

**角色变更**：
- 系统管理员（id=1）和普通用户（id=2）标记为「全局角色」（所有租户可见）
- 现有配额字段 `knowledge_space_file_limit` 自动迁入 `quota_config` JSON 字段
- 角色唯一约束从全局改为租户内唯一（不影响现有数据，因所有角色都在默认租户下）

**用户组变更**：
- 所有现有用户组默认设为「公开」可见性
- 用户组命名唯一约束从全局改为租户内唯一

## 4. 权限数据迁移

### 4.1 核心概念：从「角色级授权」到「主体级授权」

理解迁移逻辑前，需先理解新旧体系的本质区别：

**v2.4 旧体系（RBAC）— 间接授权**：
```
用户 → 角色 → 资源
```
授权记录存储在 role_access 表，一条记录表示「某角色对某资源有某权限」。拥有该角色的所有用户间接获得此权限。

**v2.5 新体系（ReBAC）— 直接授权**：
```
用户 → 资源
```
授权记录存储在 OpenFGA 中，每条元组（tuple）表示「某用户/某部门/某用户组 对 某资源 有 某关系」。

**迁移的核心工作**：将旧体系中分散在多张表里的间接授权关系，展开并转换为 OpenFGA 中的直接授权元组。

### 4.2 OpenFGA 元组格式说明

OpenFGA 中每条权限记录称为「元组」，格式为三元组：

```
(主体, 关系, 资源对象)
```

例如：
- `(user:7, viewer, knowledge_space:100)` — 用户 7 对知识空间 100 有查看权限
- `(user:3, owner, workflow:abc-123)` — 用户 3 是工作流 abc-123 的所有者
- `(user:5, super_admin, system:global)` — 用户 5 是系统超级管理员

v2.5 支持四种关系层级（从高到低）：

| 关系 | 含义 | 包含的操作权限 |
|------|------|--------------|
| owner（所有者） | 资源创建者，最高权限 | 全部操作 + 删除 |
| manager（管理者） | 协作管理者 | 全部操作（不含删除） |
| editor（编辑者） | 内容编辑者 | 查看 + 编辑（不含权限管理） |
| viewer（查看者） | 只读用户 | 仅查看/使用 |

高级别关系自动包含低级别的操作权限（owner 自动拥有 manager/editor/viewer 的全部权限）。

### 4.3 迁移范围

#### 要迁移的数据

| 数据来源 | 迁移内容 | 目标 |
|----------|---------|------|
| userrole 表 (role_id=1) | 系统管理员身份 | OpenFGA super_admin 关系 |
| usergroup 表 | 用户组成员/管理员关系 | OpenFGA user_group member/admin 关系 |
| role_access 表 (非菜单权限) | 资源读/写授权 | OpenFGA viewer/editor 关系 |
| space_channel_member 表 | 知识空间/频道成员 | OpenFGA owner/manager/viewer 关系 |
| 各资源表 user_id 字段 | 资源创建者 | OpenFGA owner 关系 |
| knowledgefile 层级路径 | 文件夹-文件父子关系 | OpenFGA parent 关系 |

#### 不迁移的数据

| 数据 | 原因 |
|------|------|
| role_access 中 type=WEB_MENU(99) 的记录 | 菜单权限继续由 RBAC 管理，不进入 OpenFGA |
| 系统管理员角色(role_id=1)的 role_access 记录 | 系统管理员在新体系中通过 super_admin 短路，无需逐资源授权 |
| flow_type=15(工作台)/20(灵思)/25(频道文章)/30(知识空间) 的资源 | 这些资源类型不受 ReBAC 控制 |
| space_channel_member 中 status 非 ACTIVE 的记录 | 待审批/已拒绝的成员关系不迁移 |

### 4.4 迁移步骤详解

迁移按以下 6 个步骤顺序执行，每一步完成后自动记录进度检查点。

---

#### 步骤 1：系统管理员

将所有系统管理员用户写入 OpenFGA，使其在新权限体系中获得最高权限。

**读取**：userrole 表中 role_id=1（系统管理员角色）的所有 user_id

**转换规则**：

| userrole 记录 | → OpenFGA 元组 |
|---|---|
| user_id=5, role_id=1 | `(user:5, super_admin, system:global)` |
| user_id=12, role_id=1 | `(user:12, super_admin, system:global)` |

**效果**：这些用户在新权限检查链的第一级即被放行，拥有对所有资源的完整操作权限。

---

#### 步骤 2：用户组成员关系

将用户组的成员和管理员关系写入 OpenFGA。

**读取**：usergroup 表的 user_id、group_id、is_group_admin 三个字段

**转换规则**：

| usergroup 记录 | → OpenFGA 元组 |
|---|---|
| user_id=7, group_id=3, is_group_admin=1 | `(user:7, admin, user_group:3)` |
| user_id=8, group_id=3, is_group_admin=0 | `(user:8, member, user_group:3)` |

**效果**：后续管理员对某资源授权时选择「按用户组授权」，组内成员自动获得对应权限。例如对知识空间 A 授予 `user_group:3#member` 的 viewer 关系后，用户 7 和用户 8 都能查看知识空间 A。

---

#### 步骤 3：角色授权展开（最复杂）

这是迁移的核心步骤。v2.4 的授权模型是「用户 → 角色 → 资源」的间接授权，需要展开为「用户 → 资源」的直接授权。

**读取两张表**：
- role_access 表：获取「角色对资源的权限」（排除 type=99 菜单权限，排除 role_id=1 管理员角色）
- userrole 表：获取「用户拥有哪些角色」（排除 role_id=1）

**权限类型映射表**：

v2.4 的 role_access.type 字段是数字编码，需要映射为 v2.5 的资源类型和关系：

| role_access.type | v2.4 含义 | → v2.5 资源类型 | → v2.5 关系 |
|---|---|---|---|
| 1 | 知识库-读 | knowledge_space | viewer |
| 3 | 知识库-写 | knowledge_space | editor |
| 5 | 助手-读 | assistant | viewer |
| 6 | 助手-写 | assistant | editor |
| 7 | 工具-读 | tool | viewer |
| 8 | 工具-写 | tool | editor |
| 9 | 工作流-读 | workflow | viewer |
| 10 | 工作流-写 | workflow | editor |
| 11 | 看板-读 | dashboard | viewer |
| 12 | 看板-写 | dashboard | editor |
| 99 | 菜单权限 | **不迁移** | — |

> **关系映射说明**：v2.4 只有「读」和「写」两级权限。迁移时「读」对应 viewer（可查看/使用），「写」对应 editor（可查看 + 编辑）。更高级别的 owner 和 manager 不会从 role_access 产生，它们来自步骤 4（成员关系）和步骤 5（资源创建者）。

**展开逻辑**：

旧体系中一条 role_access 记录覆盖所有拥有该角色的用户，迁移时需要「展开」为每个用户的独立元组：

```
对于每条 role_access 记录 (role_id, third_id, type):
    1. 查映射表，得到 (资源类型, 关系)
    2. 找到所有拥有该 role_id 的用户列表
    3. 对每个用户，生成一条 OpenFGA 元组
```

**完整示例**：

假设 v2.4 中有以下数据：

| role_access 表 | | |
|---|---|---|
| role_id | third_id | type |
| 3 (编辑角色) | 100 (知识库A) | 3 (知识库-写) |
| 3 (编辑角色) | 200 (工作流B) | 9 (工作流-读) |

| userrole 表 | |
|---|---|
| user_id | role_id |
| 7 | 3 (编辑角色) |
| 8 | 3 (编辑角色) |
| 9 | 3 (编辑角色) |

迁移后生成 6 条 OpenFGA 元组：

| 来源 | → OpenFGA 元组 | 解释 |
|---|---|---|
| role_id=3 对知识库100有写权限 + user:7拥有角色3 | `(user:7, editor, knowledge_space:100)` | 用户7对知识库A有编辑权限 |
| 同上 + user:8拥有角色3 | `(user:8, editor, knowledge_space:100)` | 用户8对知识库A有编辑权限 |
| 同上 + user:9拥有角色3 | `(user:9, editor, knowledge_space:100)` | 用户9对知识库A有编辑权限 |
| role_id=3 对工作流200有读权限 + user:7拥有角色3 | `(user:7, viewer, workflow:200)` | 用户7对工作流B有查看权限 |
| 同上 + user:8拥有角色3 | `(user:8, viewer, workflow:200)` | 用户8对工作流B有查看权限 |
| 同上 + user:9拥有角色3 | `(user:9, viewer, workflow:200)` | 用户9对工作流B有查看权限 |

> **数据量说明**：展开后的元组数 = role_access 记录数 × 平均每角色用户数。如果有 500 条 role_access 记录、平均每角色 20 个用户，则产生约 10,000 条元组。这是迁移中数据量最大的一步。

---

#### 步骤 4：知识空间/频道成员

将 space_channel_member 表中的存量成员关系同步到 OpenFGA，使已有的成员权限在新体系中生效。

**读取**：space_channel_member 表中 status='ACTIVE' 的记录（PENDING/REJECTED 状态的不迁移，因为这些用户尚未获得权限）

**资源类型映射**：

| space_channel_member.business_type | → v2.5 资源类型 |
|---|---|
| space | knowledge_space |
| channel | channel |

**角色映射**：

| space_channel_member.user_role | → v2.5 关系 | 含义 |
|---|---|---|
| creator | owner | 空间/频道创建者 → 所有者 |
| admin | manager | 空间/频道管理员 → 管理者 |
| member | viewer | 普通成员 → 查看者 |

**转换示例**：

| space_channel_member 记录 | → OpenFGA 元组 |
|---|---|
| business_id=100, business_type=space, user_id=5, user_role=creator | `(user:5, owner, knowledge_space:100)` |
| business_id=100, business_type=space, user_id=7, user_role=admin | `(user:7, manager, knowledge_space:100)` |
| business_id=100, business_type=space, user_id=8, user_role=member | `(user:8, viewer, knowledge_space:100)` |
| business_id=50, business_type=channel, user_id=3, user_role=creator | `(user:3, owner, channel:50)` |

> **与步骤 3 的关系**：同一用户对同一知识空间，可能在步骤 3 中通过 role_access 获得了 viewer 权限，在步骤 4 中又通过 space_channel_member 获得了 manager 权限。去重规则（见 §4.5）会保留更高的 manager，丢弃 viewer。

> **升级后的运行时行为**：space_channel_member 表在升级后**继续使用**，与 OpenFGA 保持双写同步（详见 §4.7）。

---

#### 步骤 5：资源所有权

确保每个资源都有明确的所有者。从各资源表的创建者字段（user_id）读取，写入 OpenFGA owner 关系。

**各资源表的读取方式**：

| 资源表 | 读取字段 | → v2.5 资源类型 | 备注 |
|--------|---------|----------------|------|
| knowledge | id, user_id | knowledge_space | 知识库/知识空间的创建者 |
| flow (flow_type=5) | id, user_id | assistant | 助手的创建者 |
| flow (flow_type=10) | id, user_id | workflow | 工作流的创建者 |
| t_gpts_tools (is_delete=0) | id, user_id | tool | 未删除的工具的创建者 |
| channel | id, user_id | channel | 频道的创建者 |
| dashboard | id, user_id | dashboard | 看板的创建者（表不存在时跳过） |

**转换示例**：

| 资源表记录 | → OpenFGA 元组 |
|---|---|
| knowledge: id=100, user_id=5 | `(user:5, owner, knowledge_space:100)` |
| flow: id=abc-123, user_id=7, flow_type=10 | `(user:7, owner, workflow:abc-123)` |
| flow: id=def-456, user_id=7, flow_type=5 | `(user:7, owner, assistant:def-456)` |
| t_gpts_tools: id=30, user_id=8 | `(user:8, owner, tool:30)` |

> **flow_type 映射**：flow 表通过 flow_type 字段区分不同类型的应用。只迁移 flow_type=5（助手）和 flow_type=10（工作流），其他类型（15=工作台, 20=灵思, 25=频道文章, 30=知识空间）不受 ReBAC 控制，不迁移。

> **与步骤 4 的去重**：用户 5 可能在步骤 4 中通过 space_channel_member(creator) 已获得了 knowledge_space:100 的 owner 关系，步骤 5 会再次产生同样的 owner 关系。去重时两者等级相同，结果不变（仍是 owner）。

---

#### 步骤 6：文件夹层级关系

建立知识空间内的文件夹和文件的父子层级关系，使 OpenFGA 能通过 parent 关系链实现权限自动继承。

**读取**：knowledgefile 表的 id、knowledge_id、file_type、file_level_path 四个字段

**file_type 含义**：
- file_type=0 → 文件夹（在 OpenFGA 中对应 `folder` 类型）
- file_type≠0 → 文件（在 OpenFGA 中对应 `knowledge_file` 类型）

**parent 确定规则**（基于 file_level_path 物化路径字段）：

| 场景 | file_level_path | parent 确定方式 | 示例 |
|------|----------------|----------------|------|
| 文件/文件夹直属知识空间 | 空 或 无路径 | parent = knowledge_space:{knowledge_id} | 文件直接挂在空间根目录下 |
| 文件/文件夹在子文件夹内 | 如 `/5/12/` | parent = folder:{路径最后一段} | 路径最后一段 12 即为父文件夹 ID |

**转换示例**：

假设知识空间 100 下有以下文件结构：
```
知识空间 100
├── 文件夹 A (id=10, file_type=0, file_level_path="")
│   ├── 文件夹 B (id=20, file_type=0, file_level_path="/10/")
│   │   └── 文件 X (id=30, file_type=1, file_level_path="/10/20/")
│   └── 文件 Y (id=31, file_type=1, file_level_path="/10/")
└── 文件 Z (id=32, file_type=1, file_level_path="")
```

生成的 OpenFGA 元组：

| 文件/文件夹 | → OpenFGA 元组 | 解释 |
|---|---|---|
| 文件夹 A (id=10, 无路径) | `(knowledge_space:100, parent, folder:10)` | 文件夹A的parent是知识空间100 |
| 文件夹 B (id=20, 路径/10/) | `(folder:10, parent, folder:20)` | 文件夹B的parent是文件夹A |
| 文件 X (id=30, 路径/10/20/) | `(folder:20, parent, knowledge_file:30)` | 文件X的parent是文件夹B |
| 文件 Y (id=31, 路径/10/) | `(folder:10, parent, knowledge_file:31)` | 文件Y的parent是文件夹A |
| 文件 Z (id=32, 无路径) | `(knowledge_space:100, parent, knowledge_file:32)` | 文件Z的parent是知识空间100 |

**权限继承效果**：建立 parent 关系后，OpenFGA 的权限模型会自动传递权限。例如：
- 用户对知识空间 100 有 viewer 权限 → 自动可查看空间下所有文件夹和文件
- 用户对文件夹 A 有 editor 权限 → 自动可编辑文件夹 A 下的文件夹 B、文件 X、文件 Y

### 4.5 跨步骤去重规则

同一个用户对同一个资源可能从多个步骤获得不同级别的权限。迁移过程中维护一个全局去重表，按「就高不就低」原则只保留最高级别。

**权限优先级**：`owner(4) > manager(3) > editor(2) > viewer(1)`

**去重逻辑**：对于同一个 (用户, 资源) 组合，如果新的权限级别 ≥ 已有的权限级别，则覆盖；否则丢弃。

**典型去重场景**：

| 步骤 | 产生的权限 | 去重结果 |
|------|-----------|---------|
| 步骤 3：role_access 给了 user:7 对 knowledge_space:100 的 **viewer** | 记录 viewer | |
| 步骤 4：space_channel_member 给了 user:7 对 knowledge_space:100 的 **manager** | manager > viewer，覆盖为 **manager** | |
| 步骤 5：knowledge 表 user_id=7 对 knowledge_space:100 的 **owner** | owner > manager，覆盖为 **owner** | |
| **最终写入 OpenFGA** | | `(user:7, owner, knowledge_space:100)` — 仅此一条 |

这确保：
- 不产生冗余元组（同一用户对同一资源只有一条记录）
- 不丢失任何权限（取最高权限 ≥ 所有来源的权限）
- 减少 OpenFGA 的存储和查询负担

### 4.6 旧表处理

迁移完成后，旧表数据处理策略：

| 表 | 处理方式 | 说明 |
|----|---------|------|
| role_access (type=WEB_MENU) | **保留，继续使用** | 菜单权限仍由 RBAC 管理 |
| role_access (其他 type) | **保留但不再使用** | 权限判定已迁至 OpenFGA，旧记录作为回溯参考保留 |
| group_resource | **保留但不再使用** | 同上 |
| space_channel_member | **保留，继续使用（双写）** | 权限判定已迁至 OpenFGA，但该表继续承担成员生命周期管理职责（详见 §4.7） |
| usergroup | **保留，继续使用** | 成员关系已同步到 OpenFGA，但表本身仍被业务使用 |

> **设计原则**：权限判定统一由 OpenFGA 负责，旧表中的授权相关记录不再参与权限检查。对于不再使用的表，保留数据进入「观察期」，在确认 v2.5 运行稳定后可在后续版本中清理。

### 4.7 知识空间/频道成员管理的双写机制

space_channel_member 表在升级后并未废弃，而是与 OpenFGA 保持**双写同步**。这是因为该表除了「权限」之外，还承担了 OpenFGA 无法表达的「成员身份生命周期管理」职责。

#### 职责分工

| 职责 | 由谁负责 | 说明 |
|------|---------|------|
| **权限判定**（用户能否查看/编辑/管理空间） | **OpenFGA** | 所有 can_read/can_edit/can_manage 检查走 PermissionService → OpenFGA |
| **成员身份状态**（申请中/已加入/已拒绝） | **space_channel_member 表** | PENDING/ACTIVE/REJECTED 等中间态只在 DB 中管理 |
| **成员列表展示**（空间里有哪些人） | **space_channel_member 表** | 前端展示成员列表仍查 DB |
| **业务规则判断**（管理员能否移除其他管理员） | **space_channel_member 表** | 需要查询成员的角色（creator/admin/member） |
| **审批工作流**（订阅→待审批→批准/拒绝） | **space_channel_member 表** | 审批流程的状态机在 DB 中流转 |

#### 各操作的双写行为

| 用户操作 | space_channel_member 表 | OpenFGA |
|----------|------------------------|---------|
| 创建知识空间/频道 | 插入一条 creator + ACTIVE 记录 | 写入 owner 元组 |
| 用户订阅公开空间 | 插入 member + ACTIVE 记录 | 写入 viewer 元组 |
| 用户申请加入需审批的空间 | 插入 member + **PENDING** 记录 | **不写入**（未批准，无权限） |
| 管理员批准申请 | PENDING → **ACTIVE** | 写入 viewer 元组 |
| 管理员拒绝申请 | PENDING → **REJECTED** | **不写入**（无权限） |
| 修改成员角色（如 member→admin） | 更新 user_role 字段 | 撤销旧关系（viewer）+ 写入新关系（manager） |
| 移除成员 | 删除记录 | 撤销对应关系 |
| 删除知识空间/频道 | 删除所有非创建者成员 | 清理该资源的全部元组 |

#### 为什么不完全替换为 OpenFGA

OpenFGA 是纯粹的权限引擎，只回答「谁对什么有什么权限」，不支持：

1. **中间态**：审批工作流需要 PENDING（待审批）和 REJECTED（已拒绝）状态，OpenFGA 只有「有权限」和「无权限」两种状态
2. **成员枚举**：OpenFGA 的 ListObjects 接口是「给定用户，列出有权限的资源」，而非「给定资源，列出有权限的用户」。展示"空间有哪些成员"需要查 DB
3. **角色区分**：业务逻辑需要区分 creator/admin/member 来判断操作权限层级（如"管理员不能移除其他管理员"），这超出了 OpenFGA 的关系模型

因此采用**分离职责**的设计：space_channel_member 管理「谁是成员、什么状态」，OpenFGA 管理「谁有什么权限」。两者通过双写保持一致。

## 5. 迁移可靠性保障

### 5.1 幂等性

所有迁移步骤均支持安全重跑：

| 层面 | 幂等机制 |
|------|---------|
| 数据库结构迁移 | 每个变更前检查是否已存在（字段/索引/表），存在则跳过 |
| 默认租户/部门初始化 | 检查是否已存在对应记录，存在则跳过 |
| 权限迁移 | 通过 Redis 标记记录是否已完成，完成则跳过 |
| OpenFGA 元组写入 | 重复写入已存在的元组，视为成功（幂等） |

### 5.2 断点续传

权限迁移的 6 个步骤支持断点续传：

- 每完成一个步骤，自动保存进度检查点
- 如果中途失败或中断，重新启动时会从上次完成的步骤之后继续
- 不会重复执行已完成的步骤

### 5.3 失败补偿

对于 OpenFGA 写入失败的个别元组：

1. **即时降级**：批量写入失败时，自动降级为逐条写入，最大限度减少失败范围
2. **补偿队列**：仍然失败的元组记入补偿队列表（failed_tuple），包含完整的写入信息和失败原因
3. **自动重试**：后台定时任务每 30 秒检查补偿队列，自动重试待处理的元组
4. **重试上限**：每条元组最多重试 3 次，超过后标记为「死信」，需人工介入

### 5.4 并发控制

- 数据库结构迁移：在服务启动前单独执行，无并发问题
- 应用初始化和权限迁移：通过分布式锁确保只有一个进程执行，其他进程等待
- 补偿队列处理：通过分布式锁确保不会多个 Worker 同时处理

## 6. 验证策略

### 6.1 升级前预览

运维人员可在升级前运行预览模式，了解迁移规模：

**预览内容**：
- 将要迁移的元组总数
- 按步骤分别统计的元组数
- 按资源类型统计的分布
- 按权限级别统计的分布

预览模式**不会**写入任何数据到 OpenFGA，仅输出统计报告。运维人员可据此评估迁移规模和预计耗时。

### 6.2 升级后校验

迁移完成后，可运行校验模式验证权限一致性：

**校验逻辑**：
1. 从旧表中抽样用户（最多 100 个非管理员用户）和资源（每种类型最多 100 个）
2. 对每个 (用户, 资源) 组合，分别用旧体系和新体系判断是否有读取权限
3. 比较结果，生成校验报告

**校验结果分类**：

| 结果 | 含义 | 是否可接受 |
|------|------|-----------|
| Match（一致） | 旧有权限、新也有权限，或旧无、新也无 | 正常 |
| Expansion（扩展） | 旧无权限、新有权限 | **可接受** — 新体系中 owner 继承的 can_read 等导致 |
| Regression（回退） | 旧有权限、新无权限 | **不可接受** — 说明有权限丢失 |

**通过标准**：Regression = 0（零回退）

> 校验模式为抽样检查，如需全量验证可调整采样参数。

### 6.3 人工验证清单

升级完成后，建议运维和业务人员按以下清单验证：

| # | 验证项 | 验证方法 |
|---|--------|---------|
| 1 | 系统管理员登录正常 | 管理员登录后可看到完整后台菜单 |
| 2 | 普通用户登录正常 | 普通用户登录后可看到工作台 |
| 3 | 知识空间可见性 | 用户能看到升级前能看到的所有知识空间 |
| 4 | 应用可见性 | 用户能看到升级前能看到的所有应用 |
| 5 | 工具可见性 | 用户能看到升级前能看到的所有工具 |
| 6 | 知识空间编辑权限 | 有编辑权限的用户仍能上传文件 |
| 7 | 应用编辑权限 | 有编辑权限的用户仍能修改应用 |
| 8 | 菜单权限 | 角色的菜单可见性与升级前一致 |
| 9 | 对话功能 | 用户可正常使用助手/工作流对话 |
| 10 | 后台新功能 | 管理员可进入「组织与成员」「用户组管理」页面 |

## 7. 异常处理

### 7.1 数据库结构迁移失败

**现象**：升级过程中提示数据库迁移错误

**处理方式**：
1. 检查错误日志，确定失败原因（常见：字段已存在、索引冲突、磁盘空间不足）
2. 修复问题后重新执行数据库迁移（支持幂等重跑）
3. 如无法修复，使用备份还原数据库，回退到 v2.4

### 7.2 权限迁移失败

**现象**：启动日志显示 F006 migration failed

**可能原因及处理**：

| 原因 | 处理方式 |
|------|---------|
| OpenFGA 服务未启动 | 启动 OpenFGA 服务后重启 API，迁移会自动重试 |
| OpenFGA 连接超时 | 检查网络连通性和 OpenFGA 配置，重启 API |
| 中途进程崩溃 | 重启 API，迁移从断点续传 |
| 个别元组写入失败 | 已自动记入补偿队列，后台任务会自动重试 |

### 7.3 升级后权限不符

**现象**：用户反馈升级后无法访问原本可以访问的资源

**排查步骤**：
1. 运行校验模式确认是否存在系统性的权限丢失
2. 检查补偿队列是否有未完成的元组
3. 检查 OpenFGA 服务是否正常运行
4. 如为个别资源问题，可通过管理后台手动调整该资源的权限

### 7.4 回退方案

如升级后出现严重问题需要回退：

1. **停止 v2.5 服务**
2. **还原数据库**至升级前备份
3. **部署 v2.4 版本**重新启动

> **注意**：数据库可以完整回退，但写入 OpenFGA 的元组无法自动清理（不影响回退后的系统运行，因为 v2.4 不使用 OpenFGA）。如需清理 OpenFGA 数据，可重建 OpenFGA Store。

## 9. 场景说明

### 9.1 单租户部署升级

绝大多数现有用户的场景。升级后：

- 多租户功能默认**关闭** (`multi_tenant.enabled = false`)
- 所有数据归入默认租户 (tenant_id=1)
- 用户登录无需选择租户
- 权限检查链路自动跳过租户相关步骤
- 系统行为与 v2.4 保持一致

### 9.2 升级后开启多租户

部分客户（如中粮、首钢）在升级后需要开启多租户。开启流程：

1. 在配置中设置 `multi_tenant.enabled = true`
2. 重启服务
3. 系统管理员在后台创建新租户
4. 为新租户创建组织架构和管理员
5. 将用户分配到对应租户
6. 按需迁移资源到新租户

> **注意**：开启多租户后，存量数据仍在默认租户中。资源向新租户的迁移需要管理员手动操作（或通过后续版本提供的批量迁移工具）。

### 9.3 全新安装

全新安装 v2.5 时：
- 数据库从空库创建，所有表结构一步到位
- 自动创建默认租户和根部门
- 权限迁移步骤自动跳过（无存量数据）
- 首个注册用户成为系统管理员

## 10. 统计看板（Dashboard）升级方案

### 10.1 背景

v2.5 引入部门概念后，统计看板的数据采集链路新增了 `user_department_infos` 维度（与已有的 `user_group_infos` 对等），使管理员能够按部门统计用户行为、应用使用、知识库规模、用户反馈等指标。

本章节面向**开源社区用户**和**商业版用户**提供统一的升级指南。两类用户的升级动作不同，但升级机制保持一致。

| mid_doc_parse_dtl | 文件解析事件 | 商业 | bisheng-telemetry 独立服务 |
| mid_model_call_dtl | 模型调用事件 | 商业 | bisheng-telemetry 独立服务 |

> **`is_commercial_only` 字段**：数据集记录通过此字段区分。开源版本启动时商业数据集在 UI 中因 license 校验不通过而隐藏，不影响开源用户。

### 10.3 涉及的仓库与产物

本次改造涉及三个仓库，最终产出两个 Docker 镜像：

| 仓库 | 改动内容 | 编译/打包到 |
|------|---------|------------|
| **bisheng**（主仓库） | 事件采集、Schema 定义、4 个开源 Celery 任务、前端 i18n | `bisheng-backend` 镜像 |
| **bisheng-telemetry-search** | 11 个数据集定义、`dashboard_dataset` 表自动升级逻辑 | 编译为 .pyc 后打包进 `bisheng-backend` 镜像 |
| **bisheng-telemetry** | 7 个商业中间表同步 Job | `bisheng-telemetry` 独立镜像 |

### 10.4 开源社区用户升级

开源用户只涉及 4 个开源数据集，**只需升级 `bisheng-backend` 一个镜像**。

#### 升级步骤

```bash
# 1. 停止后端服务
docker-compose stop backend

# 2. 拉取新版镜像
docker pull dataelement/bisheng-backend:<new-version>

# 3. 启动后端服务
docker-compose up -d backend

# 4. 查看启动日志，确认 init_dashboard_datasets 成功执行
docker logs -f <backend-container> | grep -i dashboard
```

#### 自动完成的动作

后端启动时 `init_default_data()` → `init_dashboard_datasets()` 会按以下逻辑处理：

| 场景 | 行为 |
|------|------|
| 全新安装（`dashboard_dataset` 表为空） | 直接写入 11 个数据集（含部门维度） |
| 从 v2.4 升级（表已有数据） | 遍历现有数据集，检测 `schema_config` 中是否已有 `user_department_infos` 维度，缺失则在 `user_group_infos` 之后追加；已存在则跳过（幂等） |

升级完成后，4 个开源数据集的 `dimensions` 列表会自动新增：
- 「部门ID」(`user_department_infos.department_id`)
- 「部门名称」(`user_department_infos.department_name`)

#### 数据写入时机

- **新事件**：升级后产生的所有遥测事件都会自动携带部门信息（通过 `_init_user_context` 加载 `user.departments`）
- **中间表**：每日 00:30 Celery Beat 触发 `sync_mid_*_increment` 任务时，从 `user.departments` 写入部门字段
- **历史数据**：v2.4 无部门概念，历史中间表记录该字段为空，**这是预期行为**

### 10.5 商业版用户升级

商业版用户除了开源版 4 个数据集，还需要 7 个商业数据集的同步服务正常运行，**需升级两个镜像**。

#### 升级步骤

```bash
# 1. 停止所有相关服务
docker-compose stop backend
docker stop <bisheng-telemetry-container>

# 2. 拉取新版镜像
docker pull dataelement/bisheng-backend:<new-version>
docker pull cr.dataelem.com/dataelement/bisheng-telemetry:<new-tag>

# 3. 启动后端（数据集定义升级由此完成）
docker-compose up -d backend

# 4. 启动商业遥测服务（中间表数据同步）
docker run -itd \
    -v /data/bisheng-telemetry/config.yaml:/app/config.yaml \
    cr.dataelem.com/dataelement/bisheng-telemetry:<new-tag>
```

#### 自动完成的动作

**bisheng-backend 启动时**（同开源版）：
- `dashboard_dataset` 表升级 — 11 个数据集全部追加部门维度

**bisheng-telemetry 启动后**（每日 Celery Beat 00:05 执行）：
- 7 个商业 mid-table Job 从 `base_telemetry_events` 提取 `user_department_infos` 并写入各自 ES 索引
- `mid_knowledge_file_increment` 从 MySQL 直接 JOIN `user_department` + `department` 表

### 10.6 无需迁移 / 无需人工操作的内容

| 内容 | 原因 |
|------|------|
| ES `base_telemetry_events` 索引映射 | object 类型动态兼容新字段，新文档自动含 `user_department_infos`，旧文档该字段自然缺失 |
| 各 mid-table ES 索引映射 | 同上，新文档自动携带，旧文档为空 |
| `dashboard_dataset` 表结构 | 无表结构变更，仅 JSON 字段内容升级，由 backend 启动时自动完成 |
| 预置看板（preset_oss / preset_commercial） | 预置看板不引用部门维度，原有图表照常工作 |
| 历史遥测数据回填 | v2.4 无部门，历史数据无部门是语义正确的。如需回填可手动触发中间表全量重跑，但**不建议**（耗时且意义有限） |
| 用户已创建的自定义看板 | 不受影响。用户可自行编辑图表，选择新增的部门维度 |

### 10.7 升级后验证

| # | 验证项 | 验证方法 | 预期结果 |
|---|--------|---------|---------|
| 1 | 后端启动无异常 | `docker logs <backend-container>` | 无 ERROR 级别日志，看到 `init_dashboard_datasets` 执行记录 |
| 2 | 数据集维度完整 | 登录系统后调用 `GET /api/v1/telemetry/dashboard/dataset/list` | 每个数据集 `schema_config.dimensions` 包含 `user_department_infos.department_id` 和 `user_department_infos.department_name` |
| 3 | 看板编辑器显示部门 | 进入任一数据集的图表编辑器 | 维度下拉列表出现「部门ID」「部门名称」 |
| 4 | 新数据包含部门字段 | 触发遥测事件（如登录、创建应用），查询 ES `base_telemetry_events` 最新文档 | `user_context.user_department_infos` 数组包含该用户所属部门 |
| 5 | 中间表数据包含部门字段 | 等待次日 Celery Beat 执行完后，查询 mid-table ES 索引最新文档 | 文档包含 `user_department_infos` 数组 |
| 6 | 按部门维度作图 | 新建图表，选择部门维度作为 X 轴，选择任一指标 | 图表能正确按部门分组显示数据 |

### 10.8 异常处理

| 现象 | 可能原因 | 处理方式 |
|------|---------|---------|
| 启动日志提示 `init_dashboard_datasets` 失败 | 数据库连接异常 / JSON 字段格式异常 | 检查 MySQL 连接；检查 `dashboard_dataset` 表 `schema_config` 字段手动编辑痕迹；修复后重启（幂等） |
| 看板编辑器维度列表仍无部门项 | 浏览器缓存 / 后端未重启 | 清除浏览器缓存；确认后端已拉取新镜像并重启 |
| 新事件缺少部门字段 | 用户未关联任何部门 | 检查 `user_department` 表，确认该用户有记录；无部门的用户该字段为空数组属预期行为 |
| 中间表数据无部门字段 | bisheng-telemetry 服务未升级 | 检查 bisheng-telemetry 容器镜像 tag；升级到新版本 |
| 按部门作图数据缺失 | 历史数据无部门信息 | 预期行为 — 部门维度只对升级后新数据生效 |

### 10.9 回退方案

若升级后发现问题需回退：

1. **回退镜像**：将 `bisheng-backend` 和 `bisheng-telemetry` 镜像回滚到升级前版本
2. **`dashboard_dataset` 表处理**：新版自动升级逻辑已为 `schema_config` 追加了部门维度字段，但**这不影响旧版运行**。旧版 bisheng-backend 不识别新维度时会忽略它们，图表照常工作。**无需手动清理数据库**。
3. **ES 数据处理**：升级后产生的新 mid-table 文档含部门字段，旧版不使用该字段时也不会报错，可保留。

> 回退是**完全安全**的 — 没有破坏性的 schema 变更，也没有不可逆的数据迁移。
