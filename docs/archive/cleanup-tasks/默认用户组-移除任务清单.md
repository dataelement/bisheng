## 任务 9 — 删组副作用去重收敛（反馈④，小范围）

任务 1+2 完成后，两处删组副作用退化为近乎相同的残留：**删组管理员行 + 推送 `delete_group` Redis 通知**（旧版另含审计日志）。
- **方案**：在 `user_group` 模块（或 `GroupChangeHandler` 附近）提供单一函数承载"删组残留副作用（admin 行清理 + redis 通知）"，新版 `_sync_user_group_delete_side_effects` 与旧版 `delete_group_hook` 都调用它，消除重复。
- 注意新旧异步差异：新版走 async（`GroupChangeHandler.execute_async`），旧版走 sync（`_run_async_safe`）；共享部分限定在纯同步的 admin 清理 + redis 通知，OpenFGA 删组元组各自保留现有调用。

## 任务 10 — 「新旧两套用户组服务统一」评估（⏸ 已确认：留待后续版本优化，本次不做）

**现状**（已核实）：
- 新版 F003 `/user-groups`（[user_group/](src/backend/bisheng/user_group/)）：CRUD + 成员 + 管理员，前端 [userGroups.ts](src/frontend/platform/src/controllers/API/userGroups.ts) 调用，服务于新用户组管理页。
- 旧版 `/group/*`（[api/v1/usergroup.py](src/backend/bisheng/api/v1/usergroup.py) + [RoleGroupService](src/backend/bisheng/api/services/role_group_service.py)）：list/create/update/delete/set_user_group/get_group_user/set_group_admin/set_group_members + **独有的** `get_group_resources` / `roles` / `manage/resources`，前端 [user.ts](src/frontend/platform/src/controllers/API/user.ts)、[log.ts](src/frontend/platform/src/controllers/API/log.ts) 调用。
- **两套均在线、被不同前端页面使用**，并非纯死代码。

**结论**：完整统一为一套是**独立的较大迁移**，超出"移除默认组特殊逻辑"范围：
1. 旧版独有的 resources/roles/manage 端点需先在新模块补齐或确认废弃；
2. 前端 `user.ts`/`log.ts` 的 `/group/*` 调用需迁移到 `/user-groups/*`；
3. 需回归两套权限/可见性差异（旧版 RoleGroupService vs 新版 `_can_view_all_groups`）。

**决策（已确认）**：本次只做**任务 9 的删组逻辑去重**；完整服务统一**留待后续版本优化**，另立 SDD 任务跟踪，本次不动。
