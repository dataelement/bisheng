# 1. 需求背景与目标

## 1.1 背景与问题

灵思（Linsight）是 BiSheng 的自主智能体（autonomous agent）模块，当前为**纯自研架构**：

- 执行内核自研中文 ReAct（`bisheng_langchain/linsight/{agent,task,react_task,manage}.py`），不基于任何成熟 agent 框架；
- 任务规划、子任务拆解、工具调用循环全部靠自定义中文 JSON prompt 驱动（`react_prompt.py`）；
- 流程知识以 **SOP（标准操作流程）** 形式由 LLM 动态生成、向量化沉淀、检索复用。

存在的核心问题：

1. **效果不稳定**：自研 ReAct 的规划质量、工具选择准确率、多步推理稳定性不及业界成熟框架；中文 JSON 解析脆弱，长流程易跑偏。
2. **维护成本高**：planning / sub-agent / 虚拟文件系统 / todo 等机制全部自研，跟不上社区演进。
3. **能力扩展靠改源码**：新增一类流程能力需要改 prompt、重启服务，无法热插拔。

业界开源框架 [`deepagents`](https://github.com/langchain-ai/deepagents)（LangChain 出品，基于 LangGraph 1.0）已内置成熟的 planning（`write_todos`）、生成式 UI、子智能体委派（`task`）、**虚拟文件系统**、Skill 体系（progressive disclosure）等能力。本地已跑通前后端 demo（`/Users/lilu/Projects/deepagents`）。本次目标是**用 deepagents 替换自研执行内核**，并把自研 SOP 概念**迁移为 deepagents 的 Skill 概念**。

## 1.2 产品目标

| 编号 | 目标 |
|-|-|
| G1 | 用 `deepagents.create_deep_agent` 替换自研 ReAct 执行内核，提升任务规划与执行的稳定性、工具调用准确率。 |
| G2 | 把流程知识从"运行时 LLM 动态生成的 SOP"迁移为"可热插拔的静态 Skill 包（`SKILL.md`）"，对齐 deepagents / Anthropic Agent Skills 规范。 |
| G3 | 提供 Skill 全生命周期管理（列表 / 上传 / 新建 / 编辑 / 删除）与 per-run 白名单激活的前端体验。 |
| G4 | 完整保留 BiSheng 企业级能力：多租户隔离、OpenFGA 权限、工具/知识库/文件/代码沙箱集成、DM8 双库兼容。 |
| G5 | 存量 SOP 资产一次性转换为 Skill；自研 ReAct 内核下线。 |

## 1.4 术语解释

| 术语 | 定义 |
|-|-|
| Linsight / 灵思 | BiSheng 自主智能体模块 |
| SOP | 现状：每次提问 LLM 动态生成的标准操作流程，向量化沉淀于 `linsight_sop` 表 |

| deepagents | LangChain 开源自主智能体框架，基于 LangGraph 1.0 |

| HITL | Human-in-the-loop，执行中暂停等待用户输入/审批 |
| 适配层 | 保留 BiSheng 横切基础设施、仅替换 agent 推理内核的迁移路径 |

## 3.3 核心范式变更：SOP → Skill

| 维度 | bisheng SOP（现状，下线） | deepagents Skill（目标） |
|-|-|-|
| 来源 | 每次提问 LLM **动态生成** | 预先**静态编写**的 SKILL.md 文件 |
| 用户编辑 | 执行前改当前会话 SOP（/workbench/sop-modify） | Skill 管理页编辑 .md |
| 复用/学习 | 执行后向量化沉淀，下次相似问题 EnsembleRetriever 检索注入 | 无沉淀、无检索 |
| 激活 | 自动（检索命中即注入） | **per-run 白名单手动勾选** |
| 粒度 | 一次任务一个完整流程 | 一个 skill 一类可复用能力 |
| 存储 | MySQL + Milvus + ES | 磁盘 SKILL.md + frontmatter |

### 关键取舍（已决策：纯手动白名单）

本期采用 **纯手动白名单** 激活，主动放弃 SOP 的三项能力：

1. **放弃自动生成**：用户从"系统帮我生成流程"变为"从 Skill 库勾选预编写的流程"。
2. **放弃向量自动匹配**：相似历史经验不再自动复用，由用户手动选择激活哪些 Skill。
3. **放弃执行沉淀 learning loop**：sync_sop_record / 评分反馈再生成闭环下线。

### 存量 SOP 处置（已决策：一次性转换 + 旧引擎下线）

- 现有 linsight_sop 表记录（name / description / content 三字段齐备）一次性批量导出为 SKILL.md（frontmatter: name≤64 / description≤1024，body=content）。
- 转换后自研 ReAct 内核与 SOP 动态生成 / 沉淀链路下线。
- linsight_sop_record、SOP 向量库（col_linsight_sop）随沉淀链路停用（数据保留归档，不再写入）。

## 4.1 发起任务·统一输入区与上下文组装

灵思的一切都从「统一输入区」开始。本节聚焦：用户如何进入灵思（任务模式）、如何通过「+」菜单组装本轮上下文（技能 / 知识库 / 附件）与输入框工具栏选用工具、以及提交前如何一目了然地一览与逐项调整。附件的解析状态与按需处理见 §4.2，本节只在「+」菜单中引出其入口。

### 4.1.1 统一输入区总览

灵思取消了原首页的「对话 / 任务」tab 切换。所有交互收敛到一个统一输入区——是否处于任务模式，由输入框顶部一枚可点除的「任务模式」chip 决定。

| 界面元素 | 位置 | 说明 |
|-|-|-|
| 任务模式 chip | 输入框顶部 | 进入灵思的唯一标识；带「×」可移除以退出 |
| 占位文案 | 输入框内 | 「发送消息，或用『+』添加 技能 / 知识空间 / 组织知识库 / 附件」 |
| 「+」按钮 | 底部左侧 | 展开上下文菜单：技能 / 知识空间 / 组织知识库 / 附件 |
| 上下文 chip 区 | 输入框内/上方 | 已挂的 技能 / 知识空间 / 组织知识库 / 附件 以 chip 呈现，逐项可移除（工具不以 chip 呈现，见 §4.1.5） |
| 发送按钮 | 底部右侧 | 提交任务；可提交性判定见边界表 |

### 4.1.2 任务模式的进入与退出（退出后上下文记忆 + 显式开关）

| 动作 | 触发方式 | 进入/退出后变化 |
|-|-|-|
| 进入任务模式 | 在首页选择灵思入口 / 默认即处于任务模式 | 输入框顶部出现「任务模式」chip；「+」菜单中可添加 技能 / 知识空间 / 组织知识库 / 附件；占位文案变为灵思引导语 |
| 退出任务模式 | 点击「任务模式」chip 上的「×」（**仅显式退出，不再"收到消息就自动退出"**） | chip 消失，回到普通对话模式。**技能上下文随之取消**——技能为任务模式专属能力，**日常模式暂不支持使用技能**；**文件 / 知识库 / 工具的选择被会话「记住」**：退出后在日常对话期间**不接管、不强求生效**，但**再次进入任务模式时自动回填并生效**（即"会话级记忆"，机制见下方策略表） |
| 再次进入 | 在同一会话重新挂上「任务模式」chip | 文件 / 知识库 / 工具的选择自动回填、继续沿用（除已被用户逐项移除）；**技能需重新勾选**（退出时已清除） |

> 关键设计：**模式是「会话内由用户显式开关的能力」**（取消"消息到达自动退出"）。退出时按上下文类型分流：**技能随退出一并取消**（日常模式暂不支持技能）；**文件 / 知识库 / 工具是「会话级记忆」**——退出后**仅被会话记住、不在日常对话链路接管**，待用户**再次进入任务模式时回填生效**；只有用户在「上下文一览」（4.1.6）里逐项点「×」移除才是真正放弃。

> 为何是"仅记忆、不接管"：日常对话模式与灵思任务模式是两套独立链路，在文件 / 知识库 / 工具的接入方式上能力不对等（日常对话不具备 offload-first 文件空间）。本期不改造日常对话链路去承接这些上下文，只在会话内记住选择、供再次进入任务模式复用。工程落地见技术方案 §9.3.8。

**退出后上下文去留与记忆策略：**

| 维度 | 策略 |
|-|-|
| 退出后的去留 | **技能：退出任务模式即取消**（任务模式专属，日常模式暂不支持技能，再次进入需重新勾选）；**文件 / 知识库 / 工具：会话级记忆**——选择被记住、日常对话期间不接管，再次进入任务模式时回填生效 |
| 上下文归属 | 文件 / 知识库 / 工具的选择归属「当前会话」，记忆生命周期 = 会话；技能仅在任务模式内有效 |
| 退出后文件如何处理 | 退出后文件**不在日常对话中生效**，仅被会话记住；再次进入任务模式时复用同一文件（offload-first 的"按需翻阅"仍发生在任务模式执行期内） |
| 移除即放弃 | 用户在「上下文一览」移除某 chip → 从会话记忆中摘除，再次进入任务模式不再回填 |
| 跨会话 | 新建会话不继承任何记忆；需长期 / 跨会话复用的资料应放入知识库 |
| 文件记忆有效期 | 文件记忆复用同一上传产物，受后端临时产物留存期约束（约 24h）；超期后该附件记忆失效，再次进入任务模式时提示用户重新上传（不静默丢弃，见技术方案 §9.3.8） |

> 产品约束：退出任务模式仅由用户显式触发；文件 / 知识库 / 工具的选择在会话内被记住、再次进入任务模式时自动沿用并生效；日常对话期间不接管这些上下文；用户主动移除才是放弃；跨会话不继承。工程落地（显式开关、前端会话级记忆、文件跨 version 复用的幂等与失效兜底）见技术方案 §9.3.8。

### 4.1.3 「+」菜单框架

「+」是组装上下文的统一入口，展开后分四类（其中「知识空间」与「组织知识库」为两个相互独立的入口——沿用现有产品逻辑，「+」菜单内**没有合并的「知识库」选项**；本文档中出现的「知识库」为这两者的统称）：

| 分类 | 作用 | 已选呈现 |
|-|-|-|
| 技能 | 选用流程知识（SOP 化的做事方法） | 已选项生成技能 chip；详见 4.1.4 |
| 知识空间 | 指定灵思检索的**个人私有**知识范围 | 已选项生成知识空间 chip；详见 4.1.5 |
| 组织知识库 | 指定灵思检索的**组织共享**资料范围 | 已选项生成组织知识库 chip；详见 4.1.5 |
| 上传文件 | 上传本期一次性参考的本地文件 | 已上传项生成附件 chip；解析态详见 §4.2 |

- 重复添加同一项自动去重，不产生重复 chip。
- 某类无可选内容时（如本租户无可用技能、无可见的知识空间 / 组织知识库），该分类项展示空态引导而非禁用消失（见边界表）。

### 4.1.4 技能（Skill）勾选激活

点开「+ → 技能」进入技能选择器：

| 选择器能力 | 说明 |
|-|-|
| 搜索 | 按名称 / 摘要关键词过滤 |
| 列表 | 单一扁平列表，仅展示本租户管理员在后台（§4.5）新建 / 导入并启用的自定义技能；用户仅选用、不可自建技能（内核 built-in skill 不在此列、对用户不可见） |
| 条目信息 | 每条显示 名称 + 一句话摘要，便于判断用途 |
| 勾选 | 多选；勾选项进入「已选」并在输入区生成技能 chip |
| 清空 | 一键清空本轮所有勾选 |

技能激活规则（二元，简单直白）：

| 用户操作 | 灵思行为 |
|-|-|
| 勾选 N 个技能 | 本轮只启用选中的这 N 个技能 |
| 不勾选（空） | 本轮不使用任何技能 |

> 规则极简：**勾了才用、不勾不用**。不存在「不勾选 = 灵思自动用全部技能」的隐式行为，避免用户对「这一轮到底带没带技能」产生歧义。

> 本端记忆：勾选状态在任务模式内持续保留；**一旦退出任务模式，技能选择即清除**（日常模式暂不支持技能，再次进入任务模式需重新勾选，见 §4.1.2）。可发现性上，扁平列表 + 搜索让用户一眼看到本租户「有哪些技能可用」。

### 4.1.5 知识空间 / 组织知识库与工具选择

灵思的资料检索范围与能力扩展，分别由「知识空间 / 组织知识库」和「工具」决定。

知识空间 / 组织知识库（「+」菜单中两个相互独立的入口，无合并的「知识库」选项）：

| 能力 | 说明 |
|-|-|
| 范围选择 | 「知识空间」与「组织知识库」为两个独立选项，分别决定灵思检索资料的范围 |
| 多选 | 两类均支持多选；已选分别生成对应 chip（知识空间 chip / 组织知识库 chip） |
| 可见范围 | 用户能选哪些，由其归属与授权决定（详见 §4.7） |

工具（工作台预设工具，**入口为输入框工具栏，沿用现有设计，不进「+」菜单**）：

| 能力 | 说明 |
|-|-|
| 入口 | 输入框内的工具栏（与「+」菜单并列的独立入口），沿用现有灵思工具选择器形态 |
| 来源 | 由管理员在工作台预先配置，用户在此选用，不能自行新增 |
| 选用与呈现 | **与日常对话模式交互完全一致**：在工具栏内勾选启用，选中态由工具栏自身体现，**不在上下文一览区生成工具 chip**（区别于技能/知识空间/组织知识库/附件） |
| 可见范围 | 能否选用由管理员配置决定（详见 §4.7） |

技能 / 知识（知识空间·组织知识库）/ 工具的区别（防混淆小表）：

| 要素 | 是什么 | 解决什么 | 来源 |
|-|-|-|-|
| 技能 Skill | 流程知识（怎么做一类事） | 让灵思按既定方法办事 | 本租户管理员新建 / 导入的自定义技能（内核 built-in skill 不在前端暴露） |
| 知识空间 / 组织知识库 | 资料库（检索什么内容） | 给灵思可检索的参考资料 | 知识空间（个人私有）+ 组织知识库（租户共享） |
| 工具 | 能力扩展（能调用什么） | 给灵思额外的执行能力 | 管理员在工作台预设 |

### 4.1.6 提交前上下文一览（可见可改）

这是「可见可改」的命门：发送前，用户在输入区清楚看到「这一轮带了什么」，并可逐项移除。

- 已挂的 任务模式 / 技能 / 知识空间 / 组织知识库 / 附件 以 chip 形式集中呈现（**工具不在此列**：工具选用沿用日常模式交互，由输入框工具栏自身呈现选中态，不生成 chip）。
- 每个 chip 显示类别图标 + 名称（技能/知识空间/组织知识库显示名称，附件显示文件名 + 解析状态）。
- 每个 chip 可单独点「×」移除，移除即从本轮上下文中剔除，不影响其他项。
- 技能 chip 移除到一个不剩，即等于「本轮不使用任何技能」，无需额外确认（勾了才用、不勾不用）。

| 移动端 / 窄屏「+」菜单 | 「+」菜单以底部抽屉呈现；上下文 chip 超出时折叠为「已添加 N 项」可展开；拖拽上传降级为点击（详见 §4.2 与 §4.0 响应式约定） |

### 4.1.10 模型选择与运行时切换

> 目标：让灵思任务的执行模型可由用户在**发起任务时**选择并切换，而无需重建代理；管理端不再单独维护「灵思任务执行模型」，而是在「工作台对话模型」列表中用单选指定一个**灵思默认模型**。

<whiteboard token="EtnIwquf3hiDLCbWTzFcGIZ4nud"></whiteboard>

**为什么改**：迁移前，灵思执行模型由模型配置页的「灵思任务执行模型」区单独配置（一个模型 + 一个「执行模式」Function Call / ReAct），用户侧无从选择、也无法切换；同时该模型与日常对话模型是两套互不相通的配置。deepagents 迁移后，内核模型可热替换、执行模式由框架接管，因此把灵思执行模型**收敛到「工作台对话模型」同一份模型池**，对用户和管理员都更直观。

- **管理端（租户管理员）**：在「模型 → 系统模型配置 → 工作台模型 → 工作台对话模型」列表中，每个模型行新增一个「灵思默认模型」单选；整列**只能选中一个**，选中者即为灵思任务的默认执行模型。原「灵思任务执行模型」配置区（模型 + 执行模式）**整体移除**。
- **终端用户**：发起任务的统一输入区提供模型选择器，候选即「工作台对话模型」列表，**默认选中管理员标记的「灵思默认模型」**；用户可在发起前切换为列表中任一模型，按本轮任务生效。

### 4.2.3 offload-first：为什么「不把整份文件塞进对话」

本期产品原则：**上传的文件正文绝不直接塞进对话**，而是被灵思「按需翻阅」。

| 维度 | 整份塞入对话（不采用） | 按需翻阅 / offload-first（本期采用） |
|-|-|-|
| 上传后等待 | 需等正文全部进入对话才能提交 | 上传+解析完成即可提交，体感更快 |
| 大文件表现 | 容易卡顿、撑爆上下文、对话变慢 | 大文件也不卡、不爆上下文 |
| 灵思如何用 | 一次性「读完」，抓不住重点 | 先看文件目录/结构，用到哪段翻哪段 |
| 含图文档 | 图文混杂、理解不可控 | 复用 BiSheng 解析转为图文一体的 markdown，图内文字一并抽取 |

> 实现落点（结合 deepagents 框架）：附件的**解析结果**落入 agent 工作区（deepagents 文件空间）的 `/uploads/<原文件名>/index.md`，含图文档的图片同目录存 `images/`、由 `index.md` 相对引用；原件不进工作区、正文零进对话，模型按需 `read_file` 分页取用。详见技术方案 §9.3.2。

| 同一文件重复添加 | 去重提示「该文件已添加」，避免重复占用与重复解析 |

## 4.3 任务执行过程可视化（含产物交付）

任务提交后，灵思进入「执行视图」。本节聚焦：用户提交任务后**实时看到的执行过程**，以及任务结束后**如何拿到并使用产物**。核心产品目标是把「黑盒转圈」变成「可观察、可信任、可溯源」的过程——用户能看见灵思在规划、在委派、在翻阅附件、在检索知识库、在产出文件，而不是只看到一个加载动画。

> **能力归属说明（结合 deepagents 框架实情，避免把适配层自建当框架原生）**：本节交互按数据来源分四层——

- **① deepagents 库原生**：todo 三态清单（`TodoListMiddleware`）、工具调用 name/args/output（`AIMessage.tool_calls` + `ToolMessage`）、子智能体委派信标（`task` 工具）。
- **② model-dependent（langchain provider 集成层）**：独立的「思考过程」流仅在 Anthropic（thinking 模式开启）或 DeepSeek-R1（需 `langchain-deepseek` 专属包）下成立；通用 LLM 经 ChatOpenAI/DashScope 接入时 `reasoning_content` 被静默丢弃，思考混在正文里。
- **③ langgraph 平台/SDK 层**：Generative UI 的 `push_ui_message`、子图事件 `subgraphs`——deepagents demo 靠 LangGraph SDK（`useStream`/`LoadExternalComponent`）直连消费，**BiSheng 不接 LangGraph SDK、走自建 WebSocket，这一层须适配层等价自建**。
- **④ 适配层自建**：`call_reason` 步骤标题、todo 失败态推断、知识库检索可视化、genUI 卡片注册表、子步骤流 `namespace` 归并。

> 凡 demo 里靠 LangGraph SDK 直连呈现的效果，BiSheng 自建 WS 下都要在适配层重做——这是本节工作量的主要来源。下文各区块以 ·*能力归属* 标注其层级。所有「调用中/失败/检索中」等中间/推断态在框架层多无原生信号，统一约定为「适配层推导态」，经 `task_execute_step.data` 的 `step_type`/`status`/`is_thinking` 标记承载，前端按标记渲染，**不新增 `MessageEventType` 枚举值**（保持既有 10 类事件协议与前端 LPOP 消费侧零改动）。

### 4.3.2 执行流区块规格

执行过程以**自上而下、流式追加**的步骤流呈现。每类区块有统一的卡片形态（标题 + 状态点 + 展开/收起），用户可逐项展开查看细节，默认折叠以保持过程清爽。

| 区块类型 | 用户看到什么 | 状态呈现 | 默认展开态 |
|-|-|-|-|
| 思考过程（流式文字）·*model-dependent* | 灵思的实时思考逐字流出，表达「我打算怎么做、为什么这么拆」（仅 Anthropic thinking 模式 / DeepSeek-R1 专属包下可独立于正文流出，通用 LLM 下与正文合并，见 §4.3.1 注①）；后端经 `task_execute_step(step_type=thinking)` 承载，非独立事件类型 | 流式打字 / 已定格 | 展开（执行中），完成后可收起 |
| 规划任务清单（todo）·*框架原生* | 一份可勾选的任务清单，逐项随执行点亮 | 每项：待办 / 进行中 / 已完成（deepagents 原生三态）；失败为适配层推断态（对应工具步骤持续 error 时联合标红，P2） | 展开 |
| 工具调用步骤卡·*框架原生* | 步骤名 + 输入摘要 + 输出摘要 + 状态 | 调用中（适配层推导态：start 帧已到、end 帧未到）/ 成功 / 失败（框架原生 `ToolMessage.status` 仅 success/error 两终态） | 折叠（点击展开看输入输出详情） |
| 子任务（委派）·*适配层自建* | 「灵思委派子智能体处理 XX」的子任务块，内含其自身的子步骤流（子 agent 库层为黑盒、只回传汇总结果；子步骤流须开启 LangGraph `subgraphs=True`，由适配层按 `namespace` 归并到父任务后透出，详见 §4.3.6） | 进行中 / 已完成 / 失败 | 折叠（展开看子过程） |
| 知识库检索步骤·*方案 B 前提* | 「正在检索知识库：XX」 | 检索中 / 已检索 | 折叠 |
| 产物文件 | 生成的文件以文件卡出现（文件名 + 类型图标 + 预览/下载） | 生成中 / 已就绪 / 生成失败 | 文件卡 |
| Generative UI 卡片·*适配层自建（P1）* | 灵思动态渲染的预定义卡片（本期先支持研究摘要卡、澄清问卡 1–2 种；deepagents 库零内置 genUI，须适配层后端转译 `state.ui` + 前端自建卡片注册表，详见 §4.3.6） | 渲染中 / 已渲染 / 不支持降级 | 展开 |

补充规则：

- **当前步骤高亮**：执行中的步骤始终有明显的「进行中」视觉（动效点/呼吸态），让用户一眼定位灵思此刻在哪一步。
- **长输出折叠**：单个步骤输出过长时自动折叠为摘要，提供「展开全部」；超长流式文字提供「折叠思考过程」开关。
- **中间产物 vs 最终产物**：执行过程中产生的临时文件标注「中间结果」，最终交付物在产物区独立聚合，避免用户混淆。

**工具步骤文案映射（让步骤可读、可溯源）**

步骤标题优先取自工具的「调用原因」（`call_reason`，提示词约束尽量控制在一行，超行 `...` 截断），并按工具类别套用统一文案模板，使「黑盒工具调用」变为用户可读的动作描述。**注**：`call_reason` 非 deepagents/LangGraph 原生字段（2.0 由 `manage.py` 向工具 schema 注入必填字段实现），迁移后由适配层经提示词工程 + `wrap_model_call` 动态注入 schema 实现；模型未填时降级为「工具展示名」兜底，POC 须验证 deepseek 等中文模型的填写遵从率。

| 工具类别 | 步骤文案模板 | 可点击溯源 |
|-|-|-|
| 联网检索 | `正在联网搜索：{query}` | 默认 —；`ToolMessage` 文本已含标题/URL/摘要，适配层解析后可加强为可点击结果列表（P2） |
| 知识库 / 文件内容检索 | `正在检索知识库：{query}` | — |
| 文件读取（read） | `正在阅读文件：{文件名}`（仅末级文件名） | 蓝字下划线，点击打开文件预览抽屉 |
| 文件写入 / 编辑（write/replace） | `正在写入 / 编辑文件：{文件名}` | 蓝字下划线，点击打开文件预览抽屉 |
| 网页抓取（web→markdown） | `正在抓取网页：{标题/URL}` | 蓝字下划线，点击新标签页打开原网页 |
| 主动求助（追问） | 见 §4.4（独立追问卡，不走普通步骤卡） | — |
| 其他所有工具 | `正在使用「{工具展示名}」` | — |

> 工具展示名而非真实名（真实名可能是一串 id）：步骤卡一律渲染「工具展示名称」，与技术方案 §3 的 10 类 `MessageEventType` 事件映射一一对齐（事件 `data` 携带工具类别与参数，前端据此套模板）。

### 4.3.6 边界与异常

| 触发场景 | 用户看到什么 / 产品如何兜底 |
|-|-|
| 并发超限进入排队 | 全局状态为「排队中」，文案「当前使用人数较多，正在排队中…」；提供「取消排队」；离开页面后台保留位次，轮到后自动转「规划中」。（等待时长的精确展示为后续增强，本期至少给可感知的排队态，不黑屏；估时口径见技术方案 §3） |

| 单个步骤失败但任务可继续 | 该步骤卡标红为「失败」并附简短原因，灵思自动尝试绕行或换路，任务整体继续推进，不打断用户。 |

| 断连后重连 | 重新连接后自动续看：已完成步骤、当前进行中步骤与产物原样恢复，期间产生的步骤补齐显示，用户无需刷新或重提交。 |
| 用户离开页面再返回 | 长任务在后台继续，返回该任务即恢复完整执行流与最新状态；任务已完成的，直接展示产物区。 |
| 产物生成失败 / 产物缺失 | 对应产物卡标「生成失败」并说明，提供「重试生成该产物」；其余成功产物正常可下载，不因单个失败而阻塞整体交付。 |
| 打包下载时部分产物不可用 | 打包前提示「N 个产物可打包，M 个不可用」，用户可选择仅打包可用项或逐个下载。 |
| Generative UI 卡片无法渲染 | 自动降级为文本/表格摘要，并标注「当前卡片以简版展示」，保证信息不丢失；**HITL 中断期间（`interrupt()` halt）的卡片不得依赖 `push_ui_message` 实时推送**（langgraph 1.2.1 已知缺陷：halt 期 pending writes 不持久化、刷新即丢），须改从 `AIMessage.tool_calls.args` 持久化数据渲染（对齐 demo `ClarificationCard` 方案）。 |

## 4.4 人机协同（执行中追问）

灵思在执行长任务时，遇到关键信息缺失、方向存在多种合理解读、或需要用户授权某项操作时，会**主动暂停当前任务并向用户发起追问**。追问不是失败，而是灵思「把不确定的地方摊开问清楚」的协同动作——即便用户已经上传了附件，遇到含糊问题灵思仍会追问，而不是擅自假设。本节规定追问在用户侧的呈现、回答方式、任务续跑逻辑，以及多任务并行时的追问隔离与各类等待异常的兜底。

### 4.4.1 用户交互流程图

    B -->|否,信息充分| C[继续执行直至产出]
    B -->|是| D[当前任务暂停<br/>挂起到追问态]
    D --> E[执行区出现追问卡<br/>顶部高亮 等待你的输入]

### 4.4.2 追问卡界面规格

追问卡出现在该任务的执行区内（紧贴当前执行步骤之下），与普通对话气泡有明显视觉区分，让用户一眼分辨「这是灵思在等我回答」。

| 区域 | 内容 | 说明 |
|-|-|-|
| 顶部状态条 | 醒目标识「⏳ 等待你的输入」 | 高亮色块 + 图标，是追问卡最强的视觉信号；任务列表/标签处同步打「待回复」角标 |
| 问题正文 | 灵思要问的具体问题 | 用自然语言清晰提出，一次聚焦一个决策点 |
| 为什么需要 | 简短说明「为何卡在这里 / 这个回答会影响什么」 | 帮助用户理解追问动机，降低「为什么又问我」的困惑 |
| 输入控件 | 根据问题类型自适应：自由文本框 / 单选项 / 多选项 / 确认（是否授权） | 选项型问题直接给可点选项，减少用户输入成本；自由问题给文本框 |
| 操作区 | 「提交」主按钮；次级「终止此任务」入口 | 提交后追问卡收起为「已回答」摘要，可展开回看 |
| 上下文锚点 | 标注该追问属于哪个任务（多任务并行时） | 显示任务标题/序号，避免答错任务 |

**视觉与状态**

- 等待回答时，该任务的进度区停留在「已暂停 · 等待你的输入」，不再滚动新步骤。
- 提交回答后，追问卡折叠成一行「✓ 已回答：<回答摘要>」，任务恢复为执行中。
- 同一任务可能多次追问，每次都是一张新的追问卡，历史追问按时间顺序留痕、可回看。

### 4.4.3 单点定向追问（多任务并行隔离）

同一会话中可同时存在多个并行执行的任务，每个任务的追问彼此**完全独立、互不串扰**：

| 规则 | 用户可见表现 |
|-|-|
| 定向归属 | 每张追问卡明确锚定到发起它的那个任务，回答只作用于该任务 |
| 互不阻塞 | 任务 A 在等追问时，任务 B 仍照常执行、照常产出，不被 A 的等待拖住 |
| 各自留痕 | 多个任务各自的追问历史分别归档在各自的执行区，不混排 |
| 多卡并存提示 | 若多个任务同时在等回答，统一在顶部/任务列表给出「N 项待回复」聚合提示，逐项点入回答 |
| 答案不串台 | 用户在某张追问卡填写的内容只提交给对应任务，绝不影响其他任务 |

| FR-4.4.1 | 灵思在执行中遇到信息缺失/歧义/需授权时，应主动暂停该任务并发起追问，而非擅自假设继续 | P0 |
| FR-4.4.2 | 含糊问题即使已附带附件/知识库，仍应触发追问；附件不作为「跳过澄清」的理由 | P0 |
| FR-4.4.3 | 追问卡须包含：问题正文、为什么需要、自适应输入控件、提交按钮、醒目「等待你的输入」标识 | P0 |
| FR-4.4.4 | 用户提交回答后，任务应从中断点继续执行，无需用户重述前文或重新发起 | P0 |
| FR-4.4.5 | 追问期间该任务进入暂停态，进度区停留在「等待你的输入」，不再推进新步骤 | P0 |
| FR-4.4.6 | 同会话多并行任务的追问相互隔离：A 等待不阻塞 B 执行，回答只作用于对应任务 | P0 |
| FR-4.4.7 | 每张追问卡须标注其归属任务（任务标题/序号），避免用户答错对象 | P0 |
| FR-4.4.8 | 多个任务同时待回复时，须在顶部/任务列表给出聚合「待回复」提示与逐项入口 | P1 |
| FR-4.4.9 | 用户提交前可修改回答内容；提交为空或不合法应被拦截并提示，不打断追问态 | P1 |
| FR-4.4.10 | 追问等待期间，用户可主动终止该任务；终止须二次确认，并保留已产出的中间结果 | P1 |
| FR-4.4.11 | 追问中断现场经持久化 checkpointer 落盘，用户可隔**任意时长**回来回答并从中断点续跑；本期不设置等待超时终止（与 §1 稳定性目标一致，见技术方案 §4） | P1 |
| FR-4.4.12 | 提交回答后，追问卡折叠为「已回答」摘要并留痕可回看；同任务多次追问按时序归档 | P1 |
| FR-4.4.13 | 刷新、重新进入会话、乃至 Worker 进程重启后，未回答的追问卡须仍然存在并可继续回答、可真正续跑（依赖持久化 checkpointer，不丢失追问态） | P1 |
| FR-4.4.14 | 追问与普通对话在视觉上须明确区分，让用户分辨「这是需要我回答的待办」 | P2 |

### 4.4.5 边界与异常

| 触发场景 | 用户看到什么 / 产品如何兜底 |
|-|-|
| 长时间未回复追问（数小时/隔天） | 追问态经 checkpointer 持久保存、不自动超时终止；用户回来仍可在原追问卡回答并从中断点续跑（本期取"可随时回来"而非"超时清理"，见 §1 决策与技术方案 §4）；若用户确不再处理，可主动终止该任务 |
| 用户重复提交同一追问（连点/多端） | 仅首次提交生效，追问卡即时切换为「已回答」并锁定，重复提交被忽略且不报错；任务不会因重复回答而紊乱 |
| 提交回答后又想修改 | 追问卡提交后即锁定为「已回答」，不可再改；如需修正，用户可在后续对话中补充说明，灵思据此调整（本期不支持「撤回已提交的追问回答」） |
| 等待期间用户主动终止任务 | 弹出二次确认「终止后将停止该任务，已完成的部分会保留」；确认后任务结束、追问卡消失，中间产物可查 |
| 提交了空内容 / 必选项未选 | 「提交」拦截并就地提示「请先填写/选择后再提交」，追问态保持不变，不会误把任务推进 |
| 回答内容明显答非所问 | 灵思可能基于回答继续，也可能就同一决策点再追问一次（生成新追问卡）；不会静默卡死 |
| 刷新页面 / 关闭后重新进入 | 未回答的追问卡原样恢复，仍标注「等待你的输入」，可直接继续回答；已回答的折叠摘要也保持 |
| 多任务同时待回复，用户只回答其中一个 | 已回答任务恢复执行，其余仍保持「待回复」并在顶部计数；用户可逐个处理，未处理的不影响已处理的 |
| 等待追问期间用户离开页面、长任务仍在后台 | 仅「等待用户回答」的任务保持暂停；不依赖追问的其他任务继续后台执行，回来后状态如实呈现（与 §4.3 长任务后台运行联动） |
| 网络中断导致回答提交失败 | 提示「提交失败，请重试」，追问卡保持可编辑、内容不丢失，恢复后可再次提交 |

## 4.5 Skill 管理页

> 面向**租户管理员**（管理本租户自定义 Skill）；**系统管理员**本身不在此页直接管理任何技能，如需维护某租户的技能，须先经 admin-scope 切换进入该租户、按该租户管理员权限操作。本节只讲"在管理页里怎么操作"；完整可见性矩阵收口在 §4.7。终端用户不进入本页，只在输入区选用（参见 §4.1）。**内核自带的 built-in skill 不在本页出现、不纳入 UI 管理。**

### 4.5.1 入口与角色差异

Skill 管理页是技能的"后台仓库"，与终端用户在输入区的"选用"是两件事。本页只承载**租户自定义技能**（从 0 新建或导入），不同角色进入后可操作范围不同：

| 角色 | 能否进入本页 | 进入后看到 | 能操作的对象 |
|-|-|-|-|
| 租户管理员 | 是 | 本租户自定义技能（单一列表） | 增删改启停本租户自定义技能 |
| 系统管理员 | 须先 admin-scope 切入目标租户 | 切入后等同该租户视角：该租户自定义技能 | 切入后按租户管理员权限增删改启停 |
| 终端用户 | 否（无此页入口） | —（仅在输入区"+ 技能"中选用） | 不可管理，仅选用 |

> 说明：技能均为**租户自定义**、按租户隔离——本租户技能只有本租户可见、可管理，不会串到别的租户（跨租户隔离详见 §4.7）。内核 built-in skill 是任务模式自带能力，**不在管理页出现，任何角色都不通过 UI 增删改启停它**。

### 4.5.2 信息架构（页面布局）

页面分三个区域：**左侧列表区**（单一列表 + 搜索）、**中部/右侧详情区**（Preview 渲染 ⇄ Source 原文切换）、**顶部操作区**（上传 / 新建 / 编辑 / 删除 / 启停）。

```Plaintext
flowchart LR
  subgraph 操作区[顶部 操作区]
    A1[上传 Skill]
    A2[新建 Skill]
    A3[编辑]
    A4[删除]
    A5[启用/停用 开关]
  end
  subgraph 列表区[左侧 列表区]
    L0[搜索框]
    L1[本租户自定义技能列表<br/>名称·描述·启停状态<br/>无分组]
    L0 --> L1
  end
  subgraph 详情区[右侧 详情区]
    D1[Preview 渲染视图]
    D2[Source 原文视图]
    D1 -. 切换 .-> D2
  end
  列表区 -- 选中某 Skill --> 详情区
  操作区 -- 作用于选中项 --> 列表区
```

> 📎 **draw.io 源文件**：4.5.2-Skill管理页信息架构.drawio

| 区域 | 内容与状态 |
|-|-|
| 列表区 | 单一列表（**无分组**），仅本租户自定义技能。每项显示名称、简要描述、启用/停用状态点；迁移而来的项带「由 SOP 迁移」来源标识（见 §4.6）。顶部搜索框按名称/描述实时过滤；可附加按状态（启用/停用）筛选 |
| 详情区 | 选中某项后展示其内容，提供 **Preview / Source** 两个视图切换：Preview 为渲染后的可读视图（结构化排版、图片以引用形式展示）；Source 为原始文本（技能定义原文） |
| 操作区 | 「上传 Skill」「新建 Skill」常驻；选中某项后「编辑」「删除」「启用/停用开关」可用 |

### 4.5.3 上传 / 新建 Skill 交互流程

支持两种创建方式，最终都进入"即时校验 → 预览确认 → 保存生效"：

- **上传**：点击选择文件，或将技能文件拖拽到上传热区。
- **新建**：在表单中填写名称、描述、技能正文等字段。

```Plaintext
flowchart TD
  Start([点 上传 或 新建]) --> Choose{选择方式}
  Choose -- 上传 --> Drop[选文件 / 拖拽到热区]
  Choose -- 新建 --> Form[填写 名称/描述/正文 表单]
  Drop --> Validate[即时校验]
  Form --> Validate
  Validate -- 校验失败 --> ErrHint[就地标红 + 提示如何修正<br/>保存按钮置灰]
  ErrHint --> FixLoop[用户修改] --> Validate
  Validate -- 校验通过 --> Preview[预览确认<br/>Preview/Source 双视图]
  Preview -- 返回修改 --> FixLoop
  Preview -- 确认保存 --> Save[保存]
  Save -- 成功 --> Done[列表出现新项<br/>默认启用<br/>成功提示]
  Save -- 失败 --> SaveErr[保存失败提示 + 可重试<br/>不丢失已填内容]
  SaveErr --> Preview
```

> 📎 **draw.io 源文件**：4.5.3-上传新建Skill流程.drawio

要点：

- 校验是**即时**的（边填边校 / 选完文件即校），不必等到点保存才报错。
- 保存前必须能在 **Preview** 看到渲染效果、在 **Source** 看到原文，确认无误再保存。
- 新建/上传成功后默认**启用**，并即时反映到终端用户输入区的可见技能列表（见 §4.5.6）。

### 4.5.4 编辑 / 删除 / 启停 交互

| 操作 | 适用对象 | 交互 |
|-|-|-|
| 编辑 | 本租户自定义技能 | 进入与新建一致的表单/正文编辑 → 即时校验 → Preview/Source 预览 → 保存生效；保存后立即对终端用户选用可见 |
| 删除 | 本租户自定义技能 | **二次确认弹窗**，提示"删除后终端用户将无法再选用该技能，且不可恢复"，确认后从列表移除 |
| 启用 / 停用 | 本租户自定义技能 | 列表项上的开关，单击即切换，无需进详情；停用后该技能从终端用户输入区可见列表中即时消失，启用后即时出现 |

> 系统管理员如需对某租户的自定义技能执行上述操作，须先 admin-scope 切入该租户，切入后权限与该租户管理员一致（详见 §4.7）。

### 4.5.5 界面状态

| 状态 | 用户看到 |
|-|-|
| 空态（本租户无自定义技能） | 列表显示引导："还没有技能，点『上传 / 新建』创建第一个" |
| 加载中 | 列表/详情骨架占位，操作按钮暂不可点 |
| 校验中 | 上传/保存时显示"校验中…"，保存按钮置灰，避免重复提交 |
| 校验失败 | 出错字段就地标红 + 具体原因 + 如何修正（文案见 §4.5.7），保存按钮保持置灰直至改正 |
| 保存失败 | 顶部提示"保存失败，请重试"，已填内容保留 |

### 4.5.6 启停 / 增删对"输入区可见技能"的即时影响

管理页的改动需让租户管理员清楚预期：它会**实时**改变终端用户在输入区「+ 技能」里能选到什么。

```Plaintext
flowchart LR
  M1[管理页 新建/启用 自定义技能] --> V1[终端用户输入区<br/>该技能 即时出现 可选]
  M2[管理页 停用 自定义技能] --> V2[终端用户输入区<br/>该技能 即时消失 不可选]
  M3[管理页 删除 自定义技能] --> V3[终端用户输入区<br/>该技能 移除<br/>已勾选者本轮失效并提示]
```

> 边界提醒：若某自定义技能在终端用户已组装但尚未提交的本轮上下文里被勾选，恰被管理员停用/删除，用户侧应给出"该技能已不可用，已自动移除"的轻提示（与 §4.1 的可见可改一览呼应），不阻塞其提交其余上下文。

### 4.5.7 校验反馈文案表

校验在上传/新建/编辑时即时触发，每条都给出"哪里错 + 怎么改"：

| 校验项 | 触发场景 | 用户看到的提示（示意） | 如何修正 |
|-|-|-|-|
| 名称含非法字符 | 名称包含不允许的符号/空白等 | "技能名称含有不支持的字符，请使用中英文、数字、连字符" | 去掉非法字符后重试 |
| 名称超长 | 名称超过长度上限 | "技能名称过长，请控制在 N 个字符以内" | 精简名称 |
| 名称重名 | 与本租户已有技能同名 | "已存在同名技能，请换一个名称" | 改名，或先编辑已有同名项 |
| 描述超长 | 描述超过长度上限 | "描述过长，请控制在 N 个字符以内" | 精简描述 |
| 必填缺失 | 名称/正文等必填项为空 | "请填写技能名称 / 技能内容" | 补全必填项 |
| 文件超大 | 上传文件超过 10MB | "文件超过 10MB 上限，请精简后重传" | 缩减文件后重传 |
| 格式不合规 | 上传文件格式/结构不符合技能规范 | "文件格式不符合技能规范，请检查后重传" | 按规范修正文件结构 |
| 正文结构非法 | 表单/文件正文不满足技能结构要求 | "技能内容结构有误，请检查必填段落是否完整" | 按提示补全/修正结构 |

> 校验失败时保存按钮始终置灰，已填内容不丢失；同一次提交存在多处错误时，逐项就地标红，不只报第一条。

| 删除二次确认 | 需二次确认 |
| 启停作用域提示 | 仅本租户生效，无全局提示 |

## 4.6 存量 SOP 迁移的用户体验

灵思从自研执行内核升级为新框架时，老的动态 SOP 会**一次性**转换为可热插拔的静态技能（Skill）。本节描述这次升级对各角色「可感知」的影响——升级在后台完成，用户无需任何手动操作，但升级后各角色在界面上会看到不同的变化与兜底。本节只谈用户体验，转换脚本与数据处理细节见配套技术方案文档。

### 4.6.1 角色感知影响总览

| 角色 | 升级后能感知到什么 | 在哪里看到 | 是否需要操作 |
|-|-|-|-|
| 租户管理员 | 管理页新增一批由旧 SOP 转换来的技能，带「由 SOP 迁移」来源标识；首次进入有一份迁移对账报告，列出成功/失败/跳过项 | 技能管理页 + 迁移对账报告入口 | 仅对失败/跳过项需人工处理，其余无需操作 |
| 终端用户 | 历史的 SOP 会话仍可打开查看（只读）；新发起的任务里不再有「SOP」入口，改用「+技能」组装上下文 | 历史任务列表 + 统一输入区 | 无需操作；新任务改用技能选择 |
| 系统管理员 | 本次迁移只针对各租户私有 SOP，与系统管理员职责无关；内核 built-in skill 不参与迁移 | —（无需进入管理页） | 无需操作 |

### 4.6.2 租户管理员：首次进入管理页的迁移结果

升级完成后，租户管理员**首次**进入技能管理页时会看到一条迁移结果提示条，引导其查看对账报告。迁移来的技能在列表中带有明确的「由 SOP 迁移」来源标识，与管理员手动新建的技能在视觉上可区分。

```Plaintext
flowchart TD
    A[升级完成后 租户管理员<br/>首次进入技能管理页] --> B[页面顶部出现<br/>迁移结果提示条]
    B --> C[点击 查看迁移对账报告]
    C --> D{对账报告<br/>按状态分组}
    D --> E[成功项<br/>已转为技能 带 由SOP迁移 标识]
    D --> F[失败项<br/>转换未成功 附原因]
    D --> G[跳过项<br/>超大SOP等 未处理 附原因]
    E --> H[可直接在管理页<br/>查看/编辑/启停]
    F --> I[人工处理路径:<br/>查看原因→修正后重建为技能]
    G --> J[人工处理路径:<br/>超大SOP拆分后<br/>重新上传为技能]
    I --> K[处理完成项<br/>从待办中消除]
    J --> K
    K --> L[对账报告可重复进入<br/>直到待办清零]
```

迁移对账报告是管理员核对「老的有没有都搬过来」的唯一可信入口，三类状态各自交代清楚：

| 状态 | 含义（用户视角） | 管理员可做的事 |
|-|-|-|
| 成功 | 旧 SOP 已转为同名/对应技能，带「由 SOP 迁移」标识，立即可用 | 查看、编辑、启停，与普通自定义技能一致 |
| 失败 | 转换过程未成功（如内容结构无法识别） | 查看失败原因；按提示修正后，手动重建为技能 |
| 跳过 | 主动未处理（如单个 SOP 体量过大、超出技能容量上限） | 查看跳过原因；将超大 SOP 拆分为多个较小技能后重新上传 |

### 4.6.3 终端用户：历史会话兼容与新会话入口

终端用户不参与迁移操作，但需要对「老会话还能不能看、新任务怎么发」有清晰预期：

- **历史 SOP 会话只读兼容**：升级前用 SOP 跑过的历史任务会话，仍保留在历史任务列表中，可打开**查看**已产生的对话与产物，作为留存记录；但历史会话内不可再「续跑」基于旧 SOP 的逻辑，也不再出现旧的 SOP 操作入口。
- **新会话改用技能**：升级后新发起的任务，统一输入区里**不再有「SOP」入口**；用户改用「+」菜单里的「技能」来组装本轮上下文（激活规则见 §4.1：勾了才用、不勾不用）。迁移来的技能若已由租户管理员保留启用，会正常出现在用户的可见技能列表中。
- **认知引导**：在历史会话只读态与首次进入新输入区时，给一句轻量说明——「原来的 SOP 已升级为『技能』，可在输入框点『+ 技能』选用」，帮助用户把旧习惯平滑映射到新入口。

### 4.6.4 FR 需求表

| 编号 | 需求描述（产品语言） |
|-|-|
| FR-6.1 | 升级时，各租户的私有 SOP 一次性转换为本租户自定义技能；内核 built-in skill 不参与转换。整个过程在后台完成，无需任何角色手动触发。 |
| FR-6.2 | 迁移来的技能在管理页须带「由 SOP 迁移」来源标识，与管理员手动新建的技能视觉可区分。 |
| FR-6.3 | 租户管理员首次进入技能管理页时，须看到迁移结果提示条，并能进入一份迁移对账报告。 |
| FR-6.4 | 对账报告须按「成功 / 失败 / 跳过」三类分组列出每个 SOP 的转换结果，并对失败、跳过项给出可读原因。 |
| FR-6.5 | 成功迁移的技能须可被租户管理员像普通自定义技能一样查看、编辑、启停。 |
| FR-6.6 | 对失败、跳过项，须提供人工处理路径：失败项可修正后重建为技能；超大 SOP（跳过项）可拆分后重新上传为技能。 |
| FR-6.7 | 对账报告须可重复进入，已处理项从待办中消除，便于管理员核对到全部清零。 |
| FR-6.8 | 终端用户的历史 SOP 会话须保留为只读可查（对话与产物可见），但不再提供旧 SOP 操作入口、不可续跑旧逻辑。 |
| FR-6.9 | 升级后新发起的任务，统一输入区不再出现「SOP」入口；用户改用「+技能」组装上下文。 |
| FR-6.10 | 在历史只读会话及新输入区，须给出轻量认知引导，说明「SOP 已升级为技能、改从『+技能』选用」。 |
| FR-6.11 | 本次为一次性切换、不与旧引擎并存；须在迁移结果提示中明确告知用户「旧 SOP 入口已下线」，避免用户继续寻找旧入口。 |

### 4.6.5 边界与异常

| 触发场景 | 用户看到什么 / 产品如何兜底 |
|-|-|
| 部分 SOP 迁移失败 | 不静默丢弃。管理员在对账报告「失败」分组中看到具体哪些 SOP 未成功及原因，并有「修正后重建为技能」的处理路径；待办项保留直到处理。 |
| 超大 SOP 被跳过 | 在「跳过」分组中明确提示「该 SOP 体量过大，已跳过」，并引导「拆分为多个较小技能后重新上传」，而非让用户误以为已迁移成功。 |
| 迁移后技能重名 | 当转换来的技能与管理员已有的自定义技能同名时，给出可识别的区分（如保留来源标识 + 标注「迁移」后缀或冲突提示），不直接覆盖管理员原有技能；管理员可在管理页自行改名或合并。 |
| 升级前正在进行中的任务会话 | 迁移仅作用于 SOP 定义本身，不打断正在跑的任务；进行中的会话按原逻辑继续直至结束，结束后归入历史会话（只读兼容）。若升级窗口内任务被中断，按 §4.3 的失败兜底处理，用户可在新框架下用技能重新发起。 |
| 终端用户打开历史 SOP 会话后想「再跑一次」 | 历史会话为只读，不提供续跑入口；给出引导「请在输入框用『+技能』重新发起新任务」，避免用户在只读界面反复尝试。 |
| 用户升级后仍在找旧的「SOP」入口 | 因一次性切换、不并存，新输入区与历史会话均给出明确说明「SOP 已升级为技能」，并指向「+技能」入口，避免用户误判为功能丢失。 |
| 租户私有 SOP 为空（无可迁移项） | 管理员首次进管理页不出现迁移提示条，或对账报告显示「无可迁移的 SOP」，不打扰，自定义技能列表按空态引导（去创建，见 §4.5）。 |

## 4.7 权限与可见性规则

本节聚焦"谁能看到、谁能操作什么"的产品规则，回答终端用户、租户管理员、系统管理员三类角色面对不同来源的技能/知识库/工具时，分别能看到什么、能改什么。本节只谈可见与可操作边界；具体怎么管理（增删改启停的操作流程与页面交互）见 §4.5，不在此重复。

### 4.7.1 资源来源与角色定义

灵思可组装的上下文资源按"归属来源"分三类，与三类角色交叉，构成全章的权限底座。

| 资源来源 | 含义 | 谁来维护 |
|-|-|-|
| 本租户自定义技能 | 当前租户从 0 新建 / 导入、仅本租户可用的技能 | 租户管理员（系统管理员经 admin-scope 切入后亦可） |
| 他租户自定义技能 | 其他租户创建的技能 | 该租户的租户管理员 |
| 知识库（组织知识库 / 知识空间） | 组织知识库（租户共享）与知识空间（用户私有） | 组织知识库由管理员授权，知识空间归用户本人 |
| 预设工具 | 工作台预设、供用户在输入区选用的工具 | 租户管理员在工作台配置 |

> 内核自带的 **built-in skill** 始终在背后生效，但对所有角色都**不在前端暴露**（不出现在选用面板，也不进管理页），**不作为可见 / 可管理资源纳入本矩阵**。下表"技能"均指**租户自定义技能**。

| 角色 | 在灵思中的定位 |
|-|-|
| 终端用户 | 在输入区选用（激活）技能/知识库/工具发起任务，不维护资源 |
| 租户管理员 | 维护本租户自定义技能、配置预设工具、授权组织知识库 |
| 系统管理员 | 本身不直接维护任何租户技能；如需维护某租户自定义技能，须经 admin-scope 切换进入该租户 |

### 4.7.2 用户视角可见性流程图

```Plaintext
flowchart TD
    A[用户打开灵思 + 菜单 选用技能] --> B{技能来源?}
    B -->|本租户自定义| D{当前角色?}
    B -->|他租户自定义| E[根本不展示<br/>列表里查不到 搜不出]
    D -->|终端用户| F[可见 可激活选用<br/>不可编辑/删除]
    D -->|租户管理员| G[可见 可激活<br/>可创建/编辑/删除/启停]
    G --> H{该技能被停用?}
    H -->|启用| I[出现在输入区可选列表]
    H -->|停用| J[输入区不再出现<br/>已选中的本轮自动失效]
    A2[系统管理员要管某租户技能] --> K[先做 admin-scope 切换<br/>进入目标租户身份]
    K --> L[在该租户视角下<br/>按租户管理员权限维护]
```

### 4.7.3 角色 × 操作权限矩阵（核心）

下表为本节收口规则。"查看激活"指能否在输入区看到并选用该资源；其余列为资源管理操作。"—（不展示）"表示该资源在此角色界面中根本不出现，而非置灰禁用。

| 角色 \ 资源 | 查看&激活 | 创建 | 编辑 | 删除 | 启停 |
|-|-|-|-|-|-|
| **终端用户** · 本租户自定义 | ✅ | ❌ | ❌ | ❌ | ❌ |
| **终端用户** · 他租户自定义 | —（不展示） | — | — | — | — |
| **租户管理员** · 本租户自定义 | ✅ | ✅ | ✅ | ✅ | ✅ |
| **租户管理员** · 他租户自定义 | —（不展示） | — | — | — | — |
| **系统管理员** · 任一租户自定义 | 须 admin-scope 切入该租户后按租户管理员权限 | 同左 | 同左 | 同左 | 同左 |

规则要点：

1. **built-in skill 不在前端暴露**：内核自带技能始终在背后生效，但不出现在任何选用面板与管理页，不作为本矩阵的资源，也不存在"激活/只读"等界面态。
2. **本租户自定义技能仅本租户可见可改**：对本租户终端用户"可见可选不可改"，对本租户管理员"可见可改"；对其他租户完全不可见。
3. **跨租户完全不可见，不是置灰而是根本不展示**：他租户的自定义技能不会出现在任何列表、搜索结果、选用面板中。用户无法感知其存在。
4. **同名不冲突**：不同租户可以有同名自定义技能；因彼此隔离展示，同名不会互相覆盖或产生歧义。用户在自己视角内只看到对自己可见的那一份。
5. **系统管理员维护某租户技能须经 admin-scope 切换**：系统管理员不能在自己的默认视角下直接编辑某个租户的自定义技能；必须先做 admin-scope 切换、进入目标租户身份，之后等同该租户的租户管理员权限。切换前后的"当前操作租户"在界面有明确标识，避免误改错租户资源。

### 4.7.4 知识库与预设工具的可见性

| 资源 | 终端用户能选用的范围由什么决定 |
|-|-|
| 组织知识库 | 由用户在本租户内的授权决定：被授权可读的组织知识库才出现在选择列表；未授权的不展示。 |
| 知识空间 | 仅本人创建/拥有的知识空间可见可选，不跨用户共享。 |
| 预设工具 | 由租户管理员在工作台是否配置并开放决定：管理员未配置或未开放的工具，用户在输入区看不到该选项。 |

跨租户隔离同样适用：他租户的组织知识库、他人的知识空间、他租户的预设工具，对当前用户都不展示。

### 4.7.5 单租户 vs 多租户模式 · 用户感知差异

| 维度 | 单租户模式 | 多租户模式 |
|-|-|-|
| 自定义技能归属 | 全平台一份私有空间，等同"本租户" | 每租户独立私有空间，互不可见 |
| 内核 built-in skill | 始终在背后生效、前端不暴露（一致） | 始终在背后生效、前端不暴露（一致） |
| "他租户"概念 | 用户无感知，不存在跨租户场景 | 存在他租户，但对用户完全不可见 |
| admin-scope 切换 | 通常不需要（无多租户切换） | 系统管理员维护某租户自定义技能时必须切换 |
| 知识库范围 | 组织知识库=全平台范围；知识空间归用户 | 组织知识库限本租户；知识空间归用户 |
| 用户整体感受 | 看不到"租户"概念，资源即"我的" | 仍看不到别租户，但能感知"本租户"边界 |

对终端用户而言，两种模式的核心体验一致：永远只看到"自己有权使用的资源"，看不到任何不属于自己范围的东西。

### 4.7.6 边界与异常

| 触发场景 | 用户看到什么 / 产品如何兜底 |
|-|-|
| 用户在选用面板搜索他租户的自定义技能名 | 返回"无结果"，而非"无权限"或"该技能存在但不可用"——不暴露其存在性，避免侧信道泄露他租户资源 |
| 终端用户试图进入技能管理/编辑入口 | 不展示管理入口；若从历史链接误入，提示"无管理权限"并引导回输入区，不展示任何可编辑表单 |
| 用户激活的某自定义技能被管理员实时停用 | 输入区该技能即时从可选列表消失；若本轮已勾选，给出"该技能已停用、本轮将不使用"的轻提示，不阻塞提交 |
| 用户激活的某知识库授权被实时收回 | 该知识库从选择列表移除；若已选中，提示"已失去该知识库访问权限"，本轮检索范围相应剔除 |
| 系统管理员未切换 admin-scope 即想改某租户自定义技能 | 自身视角下查不到该租户自定义技能（不展示）；需先完成 admin-scope 切换，界面明确提示"请先切换到目标租户" |
| admin-scope 切换中误以为仍在原租户而修改 | 界面持续显示"当前操作租户：XXX"标识，编辑前以该标识强提示，降低改错租户资源风险 |
| 多租户下用户被迁移/移出某租户 | 原租户自定义资源对其不再可见；正在组装的相关上下文自动失效并提示，已提交任务的产物可见性按任务归属处理 |
| 不同租户存在同名自定义技能 | 用户仅在本视角看到属于自己范围的那份，不会看到他租户同名项，无歧义、无冲突提示 |

### 4.7.7 任务模式菜单权限（首页 · 任务模式）

> 目标：让「灵思任务模式」成为可独立授权的工作台菜单项——在「系统 → 角色与权限 → 角色管理 → 编辑角色 → 工作台菜单」的「首页」下新增「任务模式」开关，控制某角色是否可进入灵思任务模式。

**为什么改**：迁移前，灵思任务模式入口（`/linsight`）与日常聊天（`/c`）共用同一「首页」菜单权限，二者无法分别授权——只要给了首页就同时给了任务模式。补充需求要求任务模式可单独开关，以便「只放开日常对话、不放开任务模式」这类授权诉求。

**菜单层级**：任务模式是 「首页」之下的子能力。在角色编辑器的「工作台菜单」分区，「首页」为父项，新增「任务模式」子开关：

| 菜单项 | 归属 | 语义 |
|-|-|-|
| 首页 | 工作台 | 是否可进入工作台首页（日常对话） |
| **任务模式（新增）** | 首页 | 是否可进入灵思任务模式（`/linsight`）；关闭则任务模式入口隐藏、直达链接被拦截 |
| 应用 / 订阅 / 知识空间 | 工作台 | 既有工作台子菜单（不变） |

**生效规则**：

1. **父子联动**：「任务模式」从属于工作台。若角色未开启「工作台」入口，「任务模式」即便存库也不对用户生效（与既有工作台子菜单的孤儿清理规则一致）。
2. **入口与路由双重收口**：无「任务模式」权限时，首页内的任务模式入口/切换不展示；若用户从历史直达链接进入 `/linsight`，按既有菜单拦截逻辑兜底（不手写 403 分支）。
3. **存量角色兼容**：升级时为既有「可进入工作台」的角色**默认补齐**「任务模式」权限，保证升级后既有用户的任务模式不被关闭（回填迁移见技术方案 §6.7）；新建角色默认开启（与订阅/知识空间默认一致）。
4. **单 / 多租户一致**：菜单权限按角色生效，单租户与多租户行为一致；多租户下仍按角色所属租户隔离。

| 编号 | 需求 | 验收要点（产品语言） |
|-|-|-|
| FR-7.8 | 角色编辑器「工作台菜单 → 首页」下新增「任务模式」开关，可独立授权灵思任务模式 | 开关可见可存；保存后该角色任务模式权限随之变化 |

#### 边界与异常表

| 场景 | 处置 |
|-|-|
| 角色开了「任务模式」但关了「工作台」入口 | 任务模式不对用户生效（父级缺失，孤儿清理剔除） |
| 用户无任务模式权限但持有 `/linsight` 旧链接 | 按既有菜单拦截兜底，不进入任务模式；不手写 403 分支 |
| 用户正在任务模式中、权限被实时收回 | 下次进入/刷新按最新权限判定；进行中任务的归属与产物按任务本身处理 |
| 多租户下角色被移出租户 | 该角色在原租户的菜单权限随之失效，行为与既有菜单权限一致 |
| 升级前的角色无 `linsight` 菜单记录 | 回填迁移补齐（见技术方案 §6.7）；不依赖前端默认值兜底存量数据 |
